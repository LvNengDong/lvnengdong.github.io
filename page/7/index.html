Vue 概述

Vue 是一套用于构建用户界面的渐进式框架。与其它大型框架不同的是，Vue 被设计为可以自底向上逐层应用。Vue 的核心库只关注视图层，不仅易于上手，还便于与第三方库或既有项目整合。另一方面，当与现代化的工具链以及各种支持类库结合使用时，Vue 也完全能够为复杂的单页应用提供驱动。

  • 前端框架三巨头：Vue.js、React.js、AngularJS
  • 官网：https://cn.vuejs.org/
  • 官方文档：https://cn.vuejs.org/v2/guide/
  • Git地址：https://github.com/vuejs

全局组件

  • 全局组件是通过 Vue 的静态方法定义的，定义完成后就相当于在 Vue 的构造器中新增了一个组件。这时我们再创建任何一个 Vue 实例时其内部都有这个组件，也就是说只要创建了 Vue 实例并通过 el 绑定了该 Vue 实例的作用范围后，就可以在此范围内使用我们提前定义好的全局组件。

组件诞生的原因就是为了复用，因为组件可能在多个地方被使用，如果用 data 保存数据的话，一个地方更改了组件中data[]的值，其它地方data[]的值也会跟着发生变化。而定义成方法的返回值有个好处就是每个方法都维护一个唯一的返回值。某个组件中的data发生变化只会影响自己，不会影响全局。

局部组件

局部组件就是一个普通的 JS对象，如果不在 Vue 中使用的话，它仅仅是一个 JS对象，和 Vue 是没有任何关系的。

只有当我们在 Vue 实例内部通过 components 字段关联了这个对象后，这个对象才成为了一个局部组件。

局部组件和全局组件的内容部分没有任何差别，不同的是全局组件是通过Vue的静态方法创建的，创建后在Vue实例的作用范围内可以直接使用，

而局部组件则会将内容部分赋值给一个普通的JS对象，我们在使用时，首先要在页面中引入这个 JS对象，并且在 Vue 对象中通过 components 字段引入这个局部组件，然后才能在 Vue 实例的 el 作用范围内使用。

  • 但是为了节省数据库性能，前端传递给后端的是一个省市区拼接起来的字符串记为 str1，后端也会把数据库中的省市区三个字段拼接起来记为 str2；
  • 然后用 str1 和 str2 进行模糊查询，得到查询结果！

  • 在仔细分析了数据库中的数据后，我们发现数据库中的数据存在一些问题。其中有一部分数据的省、市、区字段是不全的。即部分数据只有省的字段，而没有市和区的字段，这时SQL在执行省市区字段的拼接操作 CONCAT(province, city, district) 时因为市和区的字段为 null，所以拼接得到的结果还是 null。

MapReduce概述

MapReduce定义

MapReduce 是一个分布式运算程序的编程框架，是用户开发“基于Hadoop的数据分析应用”的核心框架。

MapReduce核心功能是将用户编写的业务逻辑代码和自带默认组件整合成一个完整的分布式运算程序，并发运行在一个Hadoop集群上。

MapReduce优缺点

优点——简单

缺点——慢

如何对数据进行切片？

Shuffle

shuffle：洗牌

从 Mapper 出去到 Reducer 接收之前全部都是 Shuffle 的范围。

Shuffle 阶段数据发生了关键的变化：

  1. 从 Mapper 出去的 <K，V> 值是无序的，而到了 Reducer 中数据则是有分组的。 也就是说，在 Shuffle 阶段会将数据整理、分组。
  2. Mapper 出来的数据是无序的，在 Shuffle 阶段首先对数据进行排序，排完序之后，相等的数据就挨在一起了，再进行分组就相对容易了。
  3. 排序方法：快排 + 归并
    • 对于快速排序来说，需要将所有的数据全部加载进内存中然后再进行比较；但是对于大数据而言，显然是不合理的。所以会先将大数据的切分，对每一块数据进行快排。
    • 然后再将多个有序块再进行归并排序。
  4. 数据会被分片（逻辑上），每一个片对应一个 MapTask 。每个 MapTask 都会进行排序，最终形成一个归并排序后的有序数据块，这些数据块最终又会汇总到一个 Reducer 中。Reducer 在拿到多个有序文件后，再进行一次归并让多个有序文件变成一个有序文件，最终输入到 Reducer 中。
  5. 这就是 Shuffle 阶段如何把数据进行排序、分组的过程。
  • 三次排序：
    1. 快排
    2. 归并
    3. 再次归并

Hadoop

Hadoop 设计的初衷是采用大量廉价的机器，组成一个集群！完成大数据的存储和计算！[通过数量的堆叠来提高算力]

Hadoop 是什么

  • Hadoop 是一个由 Apache 基金会所开发的分布式系统基础架构。
  • 主要解决，海量数据的存储和海量数据的分析计算问题。
  • 狭义上来说，单指 Hadoop 技术。
  • 广义上来说，Hadoop 通常是指一个更广泛的概念——Hadoop 生态圈

Hadoop 的优点（4高）

  1. 高可靠性

Hadoop 底层维护多个数据副本，所以即使 Hadoop 某个计算元素或存储出现故障（即使 Hadoop 集群中的某个数据副本所在的节点出现故障），也不会导致数据的丢失。

而且 Hadoop 会实时动态的监测集群中的副本，如果集群中的某个节点发生故障导致副本数小于集群预设的副本数，那么 Hadoop 就会自动创建一个新的副本，维护副本数量等于预设的副本数量。

  1. 高扩展性

在集群间分配任务数据，可方便的扩展数以千计的节点。

在处理数据时如果发现集群中的节点不够使用了，可以很方便的增加一些节点扩展整个 Hadoop 集群。

  1. 高效性

在 MapReduce 的思想下，Hadoop 是并行工作的，以加快任务处理速度。

  1. 高容错性

能够自动将失败的任务重新分配。

Hadoop 在 Map 阶段会将大的任务分成若干个小的任务进行运算，如果在运算过程中某个小的任务因某些不可抗力因素发生失败，那么 Hadoop 就会重新分配一个线程来运行未成功的任务，直至成功。

Hadoop 中的组件

  • 在 Hadoop1.x 时代，Hadoop 中的 MapReduce 同时处理业务逻辑运算和资源的调度，耦合性较大；
  • 在Hadoop2.x时代，增加了 Yarn 。Yarn 只负责资源的调度，MapReduce 只负责运算。

  • Common：HDFS 和 MR 共有的常用的工具包模块。

在 1.x 版本中，MR 既负责运行 MR 程序还负责为 MR 程序申请资源！Hadoop 只能为自身的 MR 程序提供服务！

HDFS 概述

产生背景

随着数据量越来越大，在一个机器上存不下所有的数据，就需要把数据分配到更多的机器中，但是这样又不方便管理和维护，所以我们迫切需要一种可以通过一台机器来管理多台机器上的文件的系统，这就是分布式文件管理系统。

HDFS 定义

  • HDFS（Hadoop Distribute FileSystem）：Hadoop 的分布式文件系统，本质上还是一个文件系统。
  • 文件系统即用来存储管理计算机中文件的系统。
  • HDFS 通过目录树来定位文件；其次，它是分布式的，由很多服务器联合起来实现其功能，集群中的服务器有各自的角色。
  • HDFS 只是分布式文件管理系统中的一种。

HDFS 的优缺点

HDFS 的使用场景

适合一次写入，多次读出的场景，且不支持文件的修改。适合用来做数据分析。

优点

  1. 高容错性

    1）数据自动保存多个副本。它通过增加副本的形式，提高容错性。【副本数量由客户端指定】

    2）某一个副本丢失以后 ，它可以自动恢复。

  2. 适合处理大数据

    1）数据规模：能够处理数据规模达到GB、TB、甚至 PB 级别的数据；

    2）文件规模：能够处理百万规模以上的文件数量，数量相当之大。

  3. 可构建在廉价机器上，通过多副本机制，提高可靠性。

缺点

  1. HDFS 不支持对文件的随机写

也就是说：仅支持数据 append（追加），不支持文件的随机修改。

原因：文件在 HDFS 上存储时，以 block 为基本单位存储！也就是说：

（1）没有提供对文件的在线寻址/在线打开功能！

（2）文件以块形式存储，如果修改了一个块中的内容，就会影响当前块之后的所有块，效率很低。

  1. 不适合低延时数据的访问。

意思就是：如果我们要求能尽可能快的读取数据（即数据访问时延较低），使用 HDFS 是不合适的。比如毫秒级的存储数据，是做不到的。一般处理很小量的数据时，要求速度要快，HDFS就不适合处理这些数据。

  1. 不适合存储大量的小文件。

原因：（1）如果 HDFS 存储了大量的小文件，就会降低 NN 的服务能力！

​ （2）小文件存储的寻址时间会超过读取时间，它违反了 HDFS 的设计目标。

  1. 不支持并发写入、文件随机修改

同一文件在同一时刻只能由一个线程写入，不允许多个线程同时写；。

假设 线程A 正在上传ss.txt文件，那么在线程A 上传开始到结束之间，其它所有的线程不能再上传ss.txt文件

HDFS 组成架构

  1. NameNode（nn）：就是Master，它是一个主管、管理者。

    1）管理HDFS的名称空间；

    2）配置副本策略；

    3）管理数据块（Block）映射信息；

    4）处理客户端读写请求

  2. DataNode（dn）：就是Slave。NameNode下达命令，DataNode执行实际的操作

    1）存储实际的数据块；

    2）执行数据块的读/写操作。

  3. Client：就是客户端。

    1）文件切分。文件上传HDFS的时候，Client将文件切分成一个一个的Block，然后进行上传；

    2）与NameNode交互，获取文件的位置信息；

    3）与DataNode交互，读取或者写入数据；

    4）Client提供一些命令来管理HDFS，比如NameNode格式化；

    5）Client可以通过一些命令来访问HDFS，比如对HDFS增删查改操作；

  4. Secondary NameNode（2nn）：并非NameNode的热备。当 NameNode 挂掉的时候，它并不能马上替换NameNode并提供服务。1）辅助NameNode，分担其工作量，比如定期合并 Fsimage 和 Edits ，并推送给 NameNode；

    2）在紧急情况下，可辅助恢复NameNode。

HDFS 块大小

  • HDFS 中的文件在物理上是分块存储（Block），块的大小由hdfs-default.xml配置文件中的参数 dfs.blocksize 来决定。

  • 在 Hadoop 2.x 中，默认值是134217728（128M）；

    在 Hadoop 1.x 中，默认值是 64M 。

  1. Hadoop 2.x 块大小默认为 128M 的原因：基于最佳传输损耗理论！

最佳传输损耗理论：在一次传输中，寻址时间占总传输时间1%时，本次传输的损耗最小，为最佳性价比传输！

  1. 为什么块的大小不能设置太小，也不能设置太大？

块大小需要适当的调节，既不能太大，也不能太小！

HDFS 的 API 操作

FileSystem

org.apache.hadoop.fs.FileSystem;

  • FileSystem：文件系统的抽象基类。它有两种实现（FileSystem 的实现取决于配置文件中 fs.defaultFS 的值）
    1. LocalFileSystem 本地文件系统 fs.defaultFS=file:///（默认值）
  1. DistributeFileSystem 分布式文件系统 fs.defaultFS=hdfs://IP地址:端口号

  • FileSystem是一个抽象类，不能直接 new ，但是可以通过静态方法 FileSystem.get() 方法来创建对象。get 方法有3个重载方法，分别是：

    方法 说明
    get(Configuration conf) 创建一个FileSystem对象，
    不指定使用的文件协议，以当前计算机的用户身份登录
    get(URI uri, Configuration conf) 创建一个FileSystem对象，
    指定使用的文件协议，以当前计算机的用户身份登录
    get(final URI uri, final Configuration conf, final String user) 创建一个FileSystem对象，
    指定使用的文件协议，并以指定身份登录

Configuration

org.apache.hadoop.conf.Configuration

  • 功能：读取配置文件中的参数

  • 源码解读：

    Configuration 类的源码中有一段静态代码块。静态代码块的作用就是随着类加载到 JVM 虚拟机， static 静态代码块中的内容就被执行。所以说静态代码块中的内容一定是在程序运行前提前执行的。

  • 配置文件

    但是如果想要得到读取自定义配置文件的 configuration 对象，就需要在当前的类路径下创建一个 core-site.xml 覆盖默认的配置文件。

URI

java.net.URI

  • URI 对象就是指 HDFS 使用的协议。
  • 可以是本地文件系统协议 file:///
  • 也可以是分布式文件系统协议 hdfs://IP地址:端口号（我的是 hdfs://hadoop102:8020）

Tip:

如果使用 FileSystem.get() 的重载方法中包含 URI 这一参数，且这时客户端的 core-site.xml 配置文件中只包含 fs.defaultFS这个属性，那么就可以省略配置文件。

因为 URI 这个参数对应的value 实际就是core-site.xml配置文件中fs.defaultFS的值，包含了使用的协议和 NN 的地址（如果是分布式文件系统）。

所以程序在加载时 URI 参数又会覆盖 core-site.xml 配置文件中关于 fs.defaultFS 的值，这时若配置文件中只包含 fs.defaultFS 这一配置信息则会被覆盖，所以可以省略配置文件 ！

配置文件

  • 我们在搭建集群时会配置很多参数，
  • 这些参数有些是由客户端来使用，有些是由服务端来使用，有些是客户端和服务端都来要使用。
  • 对于仅客户端使用的参数，在服务端可以无需配置，同理对于仅服务端使用的参数，在客户端也可以不配置。而对于客户端和服务器端都需要使用的参数，则需要在客户端和服务器端都要配置。
