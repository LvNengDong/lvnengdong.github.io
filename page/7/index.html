函数式接口

包含且仅包含一个抽象方法的接口，称为函数式接口。（不是规定接口中只能有一个方法）

@FunctionalInterface注解：我们可以通过在任意接口上使用@FunctionalInterface注解来检查本接口是否为函数式接口。如果某个接口是一个函数式接口，在 JavaDoc也会包含一条声明加以说明。

Lambda表达式

所谓的 Lambda 表达式，就是对接口中抽象方法的实现。

Lambda是一个匿名函数，我们可以把Lambda表达式理解为一段可以传递的代码（将代码像数据一样进行传递）。可以写出更简洁、更灵活的代码。

Lambda 表达式整体可以视为一个看起来很长的变量，可以作为方法中的参数传递。

Lambda表达式，也可以称为闭包，是Java8的新特性。

Lambda 允许把函数作为一个方法的参数（函数作为参数传递进方法中）。

原来的匿名内部类–>>Lambda表达式

Lambda 表达式要求接口是函数式接口，即接口中只有一个抽象方法，在重写方法时不会产生歧义，所以方法名可以省略。

拷贝小括号，写死右箭头，落地大括号。

拷贝小括号：接口中唯一的抽象方法参数所在的小括号全部拷贝过来；

写死右箭头；

落地大括号：具体的落地实现放在大括号中

Lambda 表达式就是解决了匿名内部类代码语法冗余的现象

函数式编程

函数：f(x) = kx + b，f(x) 是要计算得到的值， kx+b 是计算过程。

函数式编程：就是让一个指定类型的变量表示 f(x)，用 Lambda 表达式表示计算过程的编程。

Lambda 表达式语法

格式

Lambda 表达式用于实现接口中的抽象方法，那么显然它应该包含 方法参数列表、方法实现、返回值 三个部分。

Java8 中引入了一个新的操作符->，它将 Lambda 分为了两个部分：

  • 左侧：Lambda 表达式的参数列表；对应接口中抽象方法的参数列表；
  • 右侧：Lambda 表达式的方法体；对应接口中抽象方法的实现类的方法体。

Lambda 表达式对接口的要求

  • 需要函数式接口的支持。即@FunctionalInterface接口

  • 因为函数式接口中有且仅有一个抽象方法，所以使用 Lambda 表达式实现的抽象方法是确定的。

Lambda表达式的特征：

  • 可选的类型声明：不需要声明参数类型，因为JVM编译器可通过上下文推断出数据类型。
  • 可选的参数圆括号：一个参数无需定义圆括号，但多个参数需要定义圆括号
  • 可选的大括号：如果主体只包含了一条语句，就不需要使用大括号。
  • 可选的返回关键字：如果主体只有一条语句且为返回值语句，则可以省略return关键字和大括号（return关键字和大括号要么全都省略，要么全都不省略）。

Lambda 语法优化

  1. 简化参数类型：可以不写参数类型，但是必须所有参数都不写参数类型
  2. 简化参数小括号，如果只有一个参数则可以省略参数小括号。
    • 有多个参数不能省略小括号；
    • 没有参数也不能省略小括号；
  3. 简化方法体大括号，如果方法体中只有一条语句，则可以省略方法体大括号
    • 如果方法体中唯一的一条语句是 return 语句，则进行省略时需要把方法体大括号和return关键字一起省略。

Lambda 表达式只是实现了方法，但是并没有调用方法。

在前面的学习中，Lambda 方法体中的方法实现都是我们自己写的。

有时候 Lambda 表达式方法体中的具体实现不是必须要我们自己重写，我们可以引用已被实现的方法作为 Lambda 表达式的方法体，这就叫做方法引用！

可以理解为方法引用是 Lambda 表达式的另外一种表现形式。

Stream API

概念

Stream：流

Stream 是 Java8 中处理集合的关键抽象概念，它可以指定你希望对集合进行的操作，可以执行非常复杂的查找、过滤和映射数据等操作。使用 Stream API 对集合数据进行操作，就类似于使用 SQL 执行数据库查询。 也可以使用 Stream API 来并行执行操作。 简而言之，Stream API 提供了一种高效且易于 使用的处理数据的方式。

Stream 用于操作数据源（集合、数组等）所生成的元素序列。

包：java.util.stream.*

Tip：

  • Stream 自己不会存储元素。
  • Stream 不会改变源对象。相反，它们会返回一个持有结果的新的 Stream 对象。
  • Stream 操作是延迟 执行的。这意味着它们会等到需要结果的时候才执行。

Stream操作的三个步骤

  1. 创建Stream对象
    • 一个数据源（如：集合、数组）
    • 获取一个流对象
  2. 中间操作
    • 一个中间操作链，对包含数据源数据的流对象进行处理
  3. 终止操作（终端操作）
    • 一个终止操作 ，执行中间操作链，并产生结果。

创建Stream对象

创建Stream对象的4种方法

  1. Java8 中的 Collection 接口被扩展，提供了两个获取流的方法。
  2. 通过 Arrays 中的的静态方法 stream() 可以获取数组流。
  3. 通过 Stream 类中的静态方法 of() 获取流对象
  4. 通过 Stream 类中的静态方法iterate或generate创建无限流。
    • 无限流就是如果不加限制 ，将会一直生成数据的流。

并行流与串行流

并行流就是把一个内容分成多个数据块，并用不同的线程分别处理每个数据块的流。

Java8 中将并行进行了优化，我们可以很容易的对数据进行并行操作。Stream API 可以声明性的通过parallel()与sequential()在并行流和串行流之间进行切换。

接口中的默认方法与静态方法

  • Java8 之前，接口中只允许定义抽象方法（只有方法的定义，而无方法的实现）；、
  • 从 Java8 开始，接口中可以存在方法的实现，但是这个方法必须被 default 关键字修饰。
  • 从 Java8 开始，接口中也可以定义静态方法。
  • 在接口中，静态方法（被 static 修饰的方法）和默认方法（被 default 修饰的方法）可以定义任意多个，不受数量的限制。

Docket select() Docket对象中有一个select()方法，返回值是一个ApiSelectorBuilder对象，
用于配置Swagger要扫描的接口
ApiSelectorBuilder build() ApiSelectorBuilder对象中有一个build()方法,返回值是一个Docket对象

我们可以在select()和build()方法之间添加扫描条件/过滤条件等，自定义配置要扫描的接口。

对象 方法 说明
ApiSelectorBuilder .apis(RequestHandlerSelectors.basePackage()) 指定要扫描的包
同上 .apis(RequestHandlerSelectors.any()) 扫描全部
同上 .apis(RequestHandlerSelectors.none()) 全都不扫描
同上 .apis(RequestHandlerSelectors.withClassAnnotation(RestController.class)) 扫描类上的注解，参数是一个注解的反射对象。
效果：只有加了特定注解的类才能被扫描到。
同上 .apis(RequestHandlerSelectors.withMethodAnnotation(GetMapping.class)) 扫描方法上的注解，参数是一个注解的反射对象
同上 .paths(PathSelectors.ant("com.nengdong.swaggerdemo")) 过滤，被配置到过滤中的路径不能被扫描到（过滤的优先级高于扫描）

.select() ...... .build()包含扫描/过滤条件，是一个整体，两个方法之间只能使用.apis()和.paths()方法，其它方法需要在这个整体之外

MyBatis的XML映射器

1、自动映射

MyBatis的自动映射resultType可以自动映射同名的查询结果。但是如果遇到复杂的场景，还需要构建一个结果映射resultMap。实际上，这两种映射策略可以混合使用。

当自动映射resultType生效时，MyBatis会获取SQL语句执行结果中返回的列名并在Domain类中查找相同名字的属性（忽略大小写）。也就是说，如果发现了ID列和id属性，MyBatis会将列ID的值赋给id属性。

通常数据库列使用大写字母组成的单词命名，单词间用下划线分隔；而Java属性一般遵循驼峰命名法规定，为了在这两种命名方式之间启用自动映射，需要将==mapUnderscoreToCamelCase==设置为true。这种情况下，如果发现了USER_NAME列和userName属性，MyBatis会将列USER_NAME的值赋给userName属性。

甚至在提供了结果映射后，自动映射也能工作。处理的优先级是：手动映射 > 自动映射。

在下面的例子中，user_password列将根据resultMap进行映射，而user_id和user_name将根据resultType自动映射。

有3种自动映射等级：

  • ==NONE==[禁用]：禁用自动映射。仅对手动映射（ResultMap）的属性进行映射
  • ==PARTIAL==[局部]：如果有ResultMap则使用ResultMap，如果ResultMap只映射了部分字段，则剩余字段会使用ResultType进行映射。**默认值是PARTIAL**。

1、日志框架

日志的作用：

记录程序运行中产生的关键数据。

日志的发展：

  1. System.out.println("")；，将程序运行时的关键数据打印在控制台上。
  2. 日志框架，既可以将程序运行时的关键数据打印在控制台上，也可以将日志信息持久化到指定的文件夹中去。
  3. 高级日志框架：在日志框架的基础上，增加了一些人性化的功能，如：
    • 异步模式
    • 自动归档：将每天的日志单独放在一个文件中，用来做分类
    • ……

存在的问题：

显而易见，日志框架是需要不断迭代的，如果我们想要使用更高级的日志框架代替原本项目中低级的日志框架，该如何实现呢？

  1. 将以前的框架卸载下俩，换上新的框架，重新修改项目中日志相关的API。显然这是不合适的，使用这种方式的话每次更换日志框架都需要修改项目中的API。
  2. 仿照JDBC数据库驱动的实现，规定一个统一的接口层，要求所有的日志框架都必须实现该接口类，在程序中调用时直接使用接口API即可，这样即使更换日志框架也不用修改源代码。

2、SLF4j使用

1、如何在系统中使用SLF4j

我们在开发的时候调用日志框架的记录方法，不是直接调用日志框架的实现类，而是调用日志抽象层里面的方法，抽象层再去调用实现层方法。

官方Demo

给系统里面导入：

  1. slf4j的jar
  2. logback的实现jar

slf4j的四层结构：

  1. 应用程序层
  2. 抽象层
  3. 适配层：如果使用了slf4j的抽象层，又要使用没有实现slf4j规定接口的实现类，这时候就需要使用适配层来完成一个转换的功能。适配层向上实现了slf4j的接口，向下规定了实现类，只需要在适配器中完成对同一类功能之间的相互赋值，即可实现既定的功能。
  4. 实现层

case1：

通过应用程序层直接调用Slf4j的抽象层，如果没有导入抽象层的实现类，显然只能得到null。

case2：

通过应用程序层直接调用Slf4j的抽象层，并且导入抽象层的实现类，则可以正确输出日志信息。

case3：

通过应用程序层直接调用Slf4j的抽象层，并且导入抽象层的实现类，但是实现类与抽象层并不适配，这时候就需要多加一层适配层，启到承上启下的作用，加上适配器后也以正确输出日志信息。

如下图所示，log4j的是一种日志框架的实现类，slf4j是日志框架的一种抽象类，但是二者之间并不兼容，这时候我们引入了slf4j-log412这一适配器，就可将这二者关联起来。

每一个日志的实现框架都有自己 的配置文件。无论使用哪一种日志门面，配置文件还是得与具体的日志实现框架的匹配。

2、如何让系统中所有的日志都统一到slf4j

对于一个应用程序来说，往往要使用多种框架才能实现。比如有一个A程序，实现该程序使用了Spring、Hibernate、MyBatis等技术，对于Spring框架来说，其底层默认采用了commons-logging日志门面，而Hibernate则采用了jboss-logging作为日志门面。针对同一个项目使用不同的日志输出API显然非常不方便，那么如何才能让一个项目中所有框架都能使用slf4j作为日志框架的底层实现类就能完成日志的输出呢？

case1

如下图所示，假设应用程序application采用了4个框架，而这4个框架的采用的日志框架又各不相同，分别是：SLF4J、Commons logging、log4j、java.util.logging。

我们想要达到的效果：只保留SLF4J框架的实现类，对于非SLF4J的框架则替换其默认日志框架为SLF4J。做到可以调用多个是指框架的抽象层API，但是只配置SLF4J的配置文件，就能完成对日志的相关操作。

做法：

将不需要使用的日志框架排除出去，但是排除过后会出现一个问题就是原来框架的运行需要的jar包不存在了程序直接无法启动，这时候我们可以引入中间包来替换掉被排除的jar包，中间包中的所有类与原jar包中的所有类名字完全一样，这就保证了程序在运行过程中可以加载到需要的全限定类名，但是中间包中的类的方法的实现又去调用SLF4J的抽象类，SLF4J的抽象类则继续调用SLF4J的实现类。

  1. ==将系统中其他日志框架先排除出去；==
  2. ==用中间包来替换原有的日志框架；==
  3. ==我们导入slf4j的实现类==

3、SpringBoot日志关系

SpringBoot项目默认依赖spirng-boot-starter依赖，进而依赖spirng-boot-starter-logging依赖，改以来用来管理日志框架。

底层依赖关系

总结：

  1. SpringBoot底层默认使用slf4j（抽象层）+ logback（实现类）的方式进行日志记录；
  2. SpringBoot也把其他的日志框架的抽象层使用中间包替换，中间包再去依赖slf4j的抽象类；
  3. slf4j的抽象类再去调用 logback（实现类）实现类完成日志相关功能。

如图所示，我们可以看到默认日志框架的包结构与中间包的包结构完全相同，这就保证了即使用中间包替换默认包，项目的加载也不会出现问题。

但是我们进一步进入中间包的实现类中可以发现，在中间包的类中又调用了SLF4J的接口中的方法。完成了对SLF4J抽象层的依赖。

  1. 如果我们要引入其他框架，一定要把这个框架的默认日志依赖移除掉。

    如果不移除的话，就会出现如上图的情况，出现2个同名的jar包，产生依赖冲突。

==SpringBoot能自动适配所有的日志，而且底层使用slf4j+logback的方式记录日志，引入其他框架的时候，只需要把这个框架依赖的日志框架排除掉即可；==

    1. 如果username为空，那么查询语句为：select * from user where gender=#{gender} ，OK。

    3. 如果gender和username都为空，那么查询语句为select * from user where，也是错误的，Error！

    该如何解决呢？可以再加上where语句。

2、动态SQL：if+where

where标签的作用就是如果该标签的子标签中包含返回值的话，它就插入一个"where"，此外，如果标签返回的内容是以AND或OR开头的，它会自动进行剔除。

这样写我们可以看到：

  1. 如果username为空，那么查询语句为：select * from user where gender=#{gender} ，OK。
  2. 如果gender为空，那么查询语句为：select * from user where username=#{username} ，OK。
  3. 如果gender和username都为空，那么查询语句为select * from user，OK。
  4. 如果gender和username都不为空，那么查询语句为select * from user where username=#{username} ??? gender=#{gender}，OK。

IK分词器简介

IKAnalyzer是一个开源的，基于java语言开发的轻量级的中文分词工具包。从2006年12月推出1.0版开始，IKAnalyzer已经推出 了3个大版本。最初，它是以开源项目Lucene为应用主体的，结合词典分词和文法分析算法的中文分词组件。新版本的IKAnalyzer3.0则发展为 面向Java的公用分词组件，独立于Lucene项目，同时提供了对Lucene的默认优化实现。

IK分词器3.0的特性如下：

1）采用了特有的“正向迭代最细粒度切分算法“，具有60万字/秒的高速处理能力。
2）采用了多子处理器分析模式，支持：英文字母（IP地址、Email、URL）、数字（日期，常用中文数量词，罗马数字，科学计数法），中文词汇（姓名、地名处理）等分词处理。
3）对中英联合支持不是很好,在这方面的处理比较麻烦.需再做一次查询,同时是支持个人词条的优化的词典存储，更小的内存占用。
4）支持用户词典扩展定义。
5）针对Lucene全文检索优化的查询分析器IKQueryParser；采用歧义分析算法优化查询关键字的搜索排列组合，能极大的提高Lucene检索的命中率。
