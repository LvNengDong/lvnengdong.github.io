1 什么是JDBC

  1. Java Database Connectivity，Java数据库连接[规范]。
  2. JDBC规范就是Java语言内置一套接口，包含了CRUD等方法，用来访问数据库应用程序。

  1. JDBC是一个接口，这个接口的作用是：
    • 定义了一套使用Java语言操作关系型数据的的规范；
    • 但是这些规范的实现类不是存在于JDK中，而是由各大数据库厂商提供的；
    • 所以使用JDBC操作数据库时，必须要导入各大数据库厂商提供的实现类。

1.1.2 JDBC的优势

  1. 开发人员可以通过一套固定的Java代码操作所有关系型数据库。

5 数据库连接池

5.1 数据库连接池概述

  1. 概念：本质上就是一个容器（集合），存放数据库连接对象的容器。

    • 当系统初始化好之后，数据库连接池就会被创建，这个容器会申请一些连接对象，当用户访问数据库时，可以从数据库连接池中获取连接对象，当用户访问完数据库之后，会把连接对象归还给数据库连接池。

  2. 优点：

    • 节约资源

    • 访问高效。

  3. 不使用数据库连接池的情况：

    • 每次访问数据库都需要创建一个新的数据库连接对象，这个对象在访问结束后就被释放了。

  4. 使用数据库连接池的情况：

    • 程序初始化完成后就会创建一个数据库连接池，客户端访问数据库时无需创建Connection，直接从数据库连接池中拿出一个Connection对象使用即可。使用完毕后也不用释放资源，只需要把Connection对象归还即可。
    • 实现了数据库连接对象的复用。

  5. 实现：

    • 如果连接对象Connection是从数据库连接池中获取的，那么调用Connection.close()方法就不是释放资源了，而是归还数据连接对象。

    • 一般情况下，我们不去实现这个接口，而是交给数据库厂商来实现，我们只需要使用它们提供的jar包即可：

      1. C3p0：数据库连接池实现技术

1、概念

  • Servlet 是 JavaEE 的规范之一，规范就是接口。
  • Servlet 是 javax.servlet 包下的一个顶级接口，接口中定义了所有 Servlet 实现类都必须实现的方法。
  • 该接口中包含一个service()方法，用于编写处理请求的业务逻辑。
  • Servlet 是运行在服务器上的一个小型 Java 程序，通过HTTP协议接收客户端发送过来的请求，处理后响应结果给客户端。
  • Servlet 是 JavaWeb 三大组件之一。三大组件分别是：
    1. Servlet 程序
    2. Fliter 过滤器
    3. Listener 监听器

1.1、Servlet中路径的写法

相对路径

  1. 特点：不以/开头，以.开头。如：./index.heml
  2. 通过相对路径不能确定唯一的资源
  3. 规则：
    • ./ ：当前目录。【./可以省略，同样表示当前目录】
    • ../ ：上一级目录

绝对路径

  1. 特点：以/开头的路径

  2. 通过绝对路径可以确定唯一的资源

  3. 如：http://localhost:3306/day15/responseDemo2

绝对路径中斜杠/的使用规则

在 Web 中，/斜杠是一种绝对路径

  1. /如果被客户端解析，得到的地址是：http:ip:port/

  2. /如果被服务器端解析，得到的地址是：http://ip:port/工程路径

  3. 特殊情况：重定向

使用：

  1. 给客户端浏览器使用：需要手动加工程路径

    • 示例：/day15/responseDemo2

    • 建议动态获取工程路径，避免程序硬解码带来的的耦合度

    • 访问一个资源需要资源的绝对路径，由于此时/代表http:ip:port/，所有还需要加上day15/responseDemo2可以构成一个URL了。

    • 使用场景：重定向、超链接、表单提交等

  2. 给服务器使用：不需要加工程路径

    • 示例：/responseDemo2
    • 同理，此时/代表http:ip:port/day15/，只需要需要加上responseDemo2可以构成一个URL了。
    • 使用场景：请求转发

2.3、原理分析

  1. 服务器是本质上是一台安装了服务器软件的高性能的计算机。

    • 计算机：有唯一的IP地址
    • 安装了服务器软件：如TomCat，TomCat默认占用的端口号是8080

  2. 一个服务器中可以部署多个工程，每个工程都有唯一的工程路径。

  3. 每个工程下可以有多个资源文件（Servlet的实现类 & WEB目录下的资源文件），每个资源文件又有唯一的访问路径。

    • WEB目录下的静态资源文件直接通过http://localhost:port/工程路径/资源文件名+后缀访问。

2.3.1、服务器端做的事：

  1. 启动 Tomcat。
  2. 启动了一段 Servlet 程序。（创建了一个 Servlet 接口的实现类对象HelloServlet，并重写了 service() 方法作为程序处理业务逻辑的方法）
  3. 在配置文件 web/WEB-INF/web.xml 中配置 HelloServlet 程序对外暴露的访问路径。
  4. 现在HelloServlet 就作为一个程序被挂载在服务器上了，通过对外暴露的访问路径，所有客户端都可以访问这个程序。

2.3.2、客户端做的事：

  1. 客户端（浏览器）通过 HelloServlet 程序对外暴露的访问路径就可以访问服务器上的 HelloServlet 程序了。
  2. 如果客户端成功访问到了 HelloServlet程序，该程序中的 service() 方法就会被执行。

2.3.3 执行原理：

  1. 客户端发送请求；

  2. 服务器端监听到请求消息后：在 service() 方法中处理请求，并作出响应。TomCat监听到请求消息后自动进行：

    1. Tomcat 将 全限定类名.java（Servlet 程序类） 对应的 .class 字节码文件加载进内存。【Class.forName()】
    2. 通过反射机制在内存中创建实例对象[自动完成]；【class.newInstance()】
    3. [自动]调用对象中的 service() 方法，执行方法中定义的业务逻辑。

  3. 结论：

    • 通过客户端可以访问服务器端的 Servlet 程序。

3、Servlet体系结构

  1. GenericServlet【抽象类】
  • GenericServlet 是一个抽闲类，实现了 Servlet 接口中的部分抽象方法。
  • 将 Servlet 接口中的其它方法做了默认空实现，只保留了service() 方法等抽象方法。
  • 好处：Servlet 接口中的其它不常用方法比如：init()、destroy() 等可以不用在自定义的 Servlet 实现类中重写了，只需要在自定义的 Servlet 实现类中重写最常用的 service() 方法即可。非常用方法只在有需要时重写即可，简化了代码书写。
  1. HttpServlet【抽象类】：
  • HttpServlet 类继承了GenericServlet 类。对 HTTP 协议中的 GET/POST 提交方式默认做了分发处理，简化了代码的书写。
  • 好处：
    1. 自定义的 Servlet 类（程序）只需继承 HttpServlet 抽象类即可。
    2. 只需要重写对应的 doGet() 或 doPost() 方法即可。【doGet()/doPost()方法等价于service() 方法】
    3. 对应客户端浏览器两种最常见的提交方式 GET/POST。

3.1 Servlet请求的分发处理

  • Servlet 程序的主要功能就是接收客户端发送的请求参数，处理请求并响应处理结果给客户端。
  • 在 Java 中使用最多的数据传输协议就是 HTTP 协议，HTTP协议中常用的数据提交方式有2种：GET/POST。

4、如何解决客户端输出到浏览器上的中文数据乱码问题？

  1. 乱码产生的原因

乱码问题产生的原因：编码&解码使用的标准不一致。

  • Response缓冲区的默认编码是ISO-8859-1。
  • 如果服务器的响应消息中不告诉用户浏览器使用什么方式解码，那么用户浏览器就会使用默认的解码方式解码【默认解码方式与用户系统一致，比如安装的是中文系统的Windows，那么默认使用GBK编码】
  • 那么拿着ISO-8859-1编码的数据，使用GBK解码，显然会出现乱码的现象。

  1. 解决方案（3个方法）：

    • 1)、在Response获取输出流对象之前，设置该流使用的编码方式与浏览器默认的解码方式一致

    • 2)、服务器在Response消息的响应头Headers中告诉客户端浏览器我使用的编码方式是什么，让浏览器选择对应的解码方式。[我们已知了计算机系统的默认编码方式为GBK，但是如果把这个程序放到默认以UTF-8编码的计算机上以方法1运行仍然会报错，所以我们需要把 设置输出流编码方式 & 告诉浏览器输出流的编码方式 结合起来使用。]

    • 3)、resoponse对象中的setContentType()方法不仅设置了服务器使用utf-8字符集，还设置了响应头中的ContentType的值为"text/html;utf-8"。简化了方法2的书写

      Tip：此方法一定要在获取流对象之前调用才有效

6.2.2 重定向 vs. 转发

  • 重定向：redirect

    1. 重定向是一种资源跳转的方式。

      • 用户希望访问一些数据，先通过URL找到了A资源，A告诉用户我满足不了你的需求，但是我知道B能满足你的需求，所以A的Response中包含了302的响应状态码以及B的URL。告诉用户的浏览器去重新访问B资源。其中用户的浏览器对A和B都发送了Request请求，这是2次请求。

    2. 重定向的特点：

      1. 地址栏发生变化。
      2. 重定向可以访问其他站点（服务器）的资源。
      3. 重定向是两次请求。
      4. 不能共享Request域中的数据。
      5. 不能访问WEB-INF目录下的资源

    3. 请求重定向的2种方法：

      • 方法1：resp.setStatus(302) + resp.setHeader("location","/D4/demo02response")
      • 方法2(推荐使用)： resp.sendRedirect("http://www.baidu.com")

  • 转发：forward

    1. 转发时地址栏路径不变
    2. 转发只能访问当前服务器下的资源
    3. 转发是一次请求，可以共享数据。
  • 访问缓存（304）
    1. 用户浏览器会把用户机暂时访问过的部分资源缓存到客户机中，未清空缓存的情况下，用户再次访问该资源时，客户端会自动检测到这些缓存的存在，就会给用户浏览器返回304响应状态码，让用户使用缓存的资源。从而节省了传输时间，减轻了服务器的压力。

7.3.2 Session原理

服务器如何确保在一次会话中，多次获取到的Session是同一个？

  • Session的实现是依赖于Cookie的。

  1. 客户端首次访问服务器时：

    1. 客户端首次访问服务器时，请求消息头中是没有cookie信息的；

    2. 服务器在发现请求消息头中没有cookie时，就会在内存中创建一个新的Session对象，并给这个Session对象设置唯一的地址ID。

    3. 服务器向客户端发送响应消息时，响应头中除了请求资源的应有的缓存信息外，还会在额外添加一组set-cookie键值对，保存Session对象的地址ID。

    4. 客户端收到响应消息后，就会把{JSESSION=Session对象的ID值}也存储到浏览器cookie信息中。

  2. 在一次会话中，当客户端再次访问服务器时：

    1. 请求头中就会携带cookie:JSESSION=Session对象的ID信息；
    2. 服务器获取到该cookie信息后，根据该信息自动查找内存中是否存在对应ID的Session对象；
    3. 所以，Demo2中通过req.getSession()找到的Session对象与Demo1中获取的Session对象是同一个。

7.3.3 Session的细节

1、当客户端关闭后，服务器端不关闭，两次获取的session是否为同一个？

  1. 默认情况下：两次访问获得的session不是同一个。

    • 默认情况下，会话结束后，cookie资源会被释放掉，客户端浏览器关闭后重新访问，服务器会重新分配一个session对象保存到set-cookie响应头中返回给客户端浏览器。显然两次获取的session不是同一个。

  2. 如果需要两次访问获取的session对象是同一个，该如何做？

    1. 可以设置cookie的最大存活时间，让cookie持久化存储，
    2. 在最大存活时间内，cookie中的JSESSIONID不会改变，所以每次访问的session对象也就是同一个。

2、客户端不关闭，服务器关闭后再打开，两次获取的session是否为同一个？

  1. 不是同一个
    • 服务器关闭后，内存中的Session对象肯定会被全部销毁。服务器重启后，获取到的Session对象显然不是同一个
  2. 虽然session对象不是同一个，但是要确保Session中的数据不丢失，可以采用：
    • session的钝化：序列化

      • 在服务器启动后，将session文件转化为内存中的session对象即可。

1 Spring概述

1.1 是什么

  • 轻量级 Java 开发框架。
  • 目的是为了解决企业级应用开发中的业务逻辑层和其他各层的耦合问题。
  • 它是一个分层的 JavaSE/JavaEE 一站式轻量级开源框架，为开发Java应用程序提供全面的基础架构支持。
  • Spring 负责基础架构，因此Java开发者可以专注于应用程序的开发。
  • Spring 最根本的使命是解决企业级应用开发的复杂性，即简化 Java 开发。
  • Spring可以做很多事情，它为企业级开发提供给了丰富的功能，但是这些功能的底层都依赖于它的两个核心特性，也就是依赖注入（dependency injection，DI）和面向切面编程（aspectoriented programming，AOP）。

1.2 两大核心

  1. IoC（Inverse of Control）：控制反转
  2. AOP（Aspect Oriented Programmimg）：面向切面编程

IOC

IOC 就是：

  • 不会在代码中直接使用 new 关键字创建对象，只是把对象声明出来；
  • 在代码中不直接让对象与服务进行连接，但是在配置文件中描述了哪一项组件需要哪一项服务，由容器将它们组装起来。
  • 在一般的 IOC 场景中容器创建了所有的对象，并设置了必要的属性将他们联系在一起，等到需要使用的时候才把他们声明出来；
  • 或者使用注解就跟，容器会自动根据注解把对象组合起来。

AOP

  • AOP 是一种编程模式，他允许开发人员通过自定义的横切点进行模块化，将那些影响多个类的行为封装到可重用的模块中。
  • 比如日志输出，不使用 AOP 的话就需要把日志的输出语句放在所有类、方法中，但是有了 AOP 就可以把日志输出语句封装一个可重
    用模块，在以声明的方式将它们放在类中，每次使用类就自动完成了日志输出。

1.3 Spring 的设计目标、设计理念、和核心

  • Spring设计目标：为开发者提供一个一站式轻量级应用开发平台；
  • Spring设计理念：在JavaEE开发中，支持POJO和JavaBean开发方式，使应用面向接口开发，充分支持OOP（面向对象）设计方法；Spring通过IOC容器实现对象耦合关系的管理，并实现依赖反转，将对象之间的依赖关系交给IOC容器，实现解耦；
  • Spring框架的核心：IOC 容器和 AOP 模块。通过 IOC 容器管理 POJO 对象以及他们之间的耦合关系；通过 AOP 以动态非侵入的方式增强服务。
  • IOC 让相互协作的组件保持松散的耦合，而 AOP 编程允许你把遍布于应用各层的功能分离出来形成可重用的功能组件。

1.4 优缺点

优点

  1. 降低耦合度，简化开发
    • 通过Spring提供的IoC容器，可以将对象间的依赖关系交由Spring进行控制，避免硬解码所造成的程序过度耦合。
    • 用户也不必再为单例模式、属性文件解析这些很底层的需求写代码，可以更专注于上层的应用。
  2. 支持面向切面编程（AOP）
    • Spring 提供面向切面编程，可以方便的实现对程序进行权限拦截、运行监控等功能。
  3. 声明式事务的支持
    • 只需要通过配置就可以完成对事务的管理，而无需手动编写代码。
  4. 方便程序的测试
    • Spring 对 Junit4 的支持，可以通过注解方便的测试 Spring 程序。
  5. 方便集成各种优秀框架
    • Spring 不排斥各种优秀的开源框架，其内部提供了对各种优秀框架的直接支持（如：Struts、Hibernate、MyBatis等）。
  6. 降低 JavaEE API 的使用难度
    • Spring 对 JavaEE 开发中非常难用的一些 API（JDBC、JavaMail、远程调用等），都提供了封装，使这些 API 应用难度大大降低。

缺点

  1. 配置复杂（相较于 SpringBoot 而言）
  2. Spring 依赖反射，反射影响性能

2.1 程序的耦合和解耦

  • 耦合性（Coupling），是对模块间关联程度的度量。

  • 划分模块的一个准则就是高内聚低耦合。

  1. 程序的耦合，就是程序间的依赖关系。包括：
    • 类之间的依赖
    • 方法之间的依赖
  2. 解耦，就是降低程序间的依赖关系。在实际开发中，应该做到：
    • 编译期不依赖，运行时才依赖。
  3. 解耦的思路：
    • step1：通过反射来创建对象，而避免使用new关键字。
    • step2：通过读取配置文件来获取想要创建的对象的全限定类名。

例子

在开发中，有些依赖关系必须是显式的，有些依赖关系可以通过优化代码来避免。

问：在使用 JDBC 注册驱动时，为什么不使用DriverManger中的静态方法 registerDriver()，而是采用Class.forName()的方式？

答：

  1. DriverManager.registerDriver(Driver driver)是 java.sql 包下的一个静态方法，接收的参数是一个数据库驱动实现类实例，作用是注册数据库驱动程序。如果我们直接调用该方法来注册驱动当然是可以的，需要我们创建一个数据库驱动实现类的实例并将对象传递给registerDriver()方法即可。但是此时：
    • 如果更换了数据库驱动，假设从MySql数据库驱动换为Oracle数据库驱动，那么必须修改源代码，重新创建一个Oracle数据库驱动对象并传递给registerDriver()方法。
    • 如果需要的数据库驱动类不存在，那么在编译期就会直接报错。
  2. 使用Class.forName("com.mysql.jdbc.Driver")反射机制将MySql数据库驱动加载进内存，其原理是：MySql 数据的驱动类中有一行静态代码块的内容是 DriverManager.registerDriver(new Driver());，即在当前Driver类中创建一个Driver类对象并使用 DriverManager.registerDriver()方法注册驱动。
    • 使用反射机制的好处是，消除了编译期依赖，Class.forName()方法中接收的形参是String类型，在编译期传递任意错误的值都不会报错，当然运行时必定会出现错误。
    • 但是更换数据库驱动时仍然要修改源代码。
  3. 但是第2种方法仍然不是耦合度最低方法。耦合度最低的方式是将数据库配置信息保存在外部配置文件(如.properties)中，在源码中通过读取配置文件的方式将数据库驱动加载进内存中。当我们更换了数据库品牌的时候，只需要修改外部的配置文件即可，而不用对源码做任何修改。

2.2 容器

Spring 作者 Rod Johnson 设计了两个接口用来表示容器。

  1. BeanFactory
  2. ApplicationContext

  • BeanFactory 简单粗暴，可以理解为就是个 HashMap，Key 是 BeanName，Value 是 Bean 的Class对象实例。通常只提供注册（put），获取（get）这两个功能。我们可以称之为 “低级容器”。

  • ApplicationContext 可以称之为 “高级容器”。它继承了包括 BeanFactory 在内的多个接口，具备了更多的功能。

    例如资源的获取，支持多种消息（例如 JSP tag 的支持），对 BeanFactory 多了工具级别的支持。所以它的名字已经不是 BeanFactory 之类的工厂了，而是 “应用上下文”， 代表着整个大容器的所有功能。

    该接口源码中定义了一个 refresh() 方法（高级容器特有的方法），此方法是所有阅读 Spring 源码的人的最熟悉的方法，用于刷新整个容器，即重新加载/刷新所有的 bean。

低级容器的IOC

IOC 在 Spring 里，只需要低级容器就可以实现：

  1. 加载配置文件，解析成 BeanDefinition 放在 Map 里。
  2. 调用 getBean 的时候，从 BeanDefinition 所属 Map 里，拿出 Class 对象进行实例化，同时，如果有依赖关系，将递归调用 getBean 方法完成依赖注入。

注意：

  • Spring 应用程序在启动时，就会加载配置文件到内存中，然后会根据配置文件中的全限定类名创建对应的 Class 类实例并保存到容器中。
  • 懒加载是指需要实例对象的时候，将 Class 对象拿出来创建对象的实例对象。而 Class 对象是需要即时加载的，并且会将 Class 实例保存到容器中作为 Value。

2、工厂

在开发中，我们可以把所有定义的类的全限定类名保存在配置文件中，当启动服务器的时候，让一个类中的方法读取配置文件，通过反射机制把这些类实例化并保存到一个容器中。

在需要使用实例对象的地方直接从容器中拿出来使用即可，而不用自己去创建(new)。

我们把这个读取配置文件，创建容器和获取容器中对象的类叫做工厂。

1、BeanFactory 和 ApplicationContext 的区别

  • BeanFactory才是spring容器中的顶层接口。
  • ApplicationContext是BeanFactory的子接口。

BeanFactory 和 ApplicationContext 是 Spring 的两大核心接口，都可以当做 Spring 的容器。其中 ApplicationContext 是BeanFactory 的子接口。

依赖关系

  • BeanFactory 是 Spring 中最底层的接口，类内部维护了一系列顶级方法，包含各种 Bean 的定义，读取 bean 配置文档，管理 bean 的加载、实例化，控制 bean 的生命周期，维护 bean 之间的依赖关系等。
  • ApplicationContext 接口作为 BeanFactory 的派生类，除了提供 BeanFactory 所具有的功能外，还提供了更完整的框架功能：
    • 继承 MessageSource，因此支持国际化。
    • 统一的资源文件访问方式。
    • 提供在监听器中注册bean的事件。
    • 同时加载多个配置文件。
    • 载入多个（有继承关系）上下文 ，使得每一个上下文都专注于一个特定的层次，比如应用的 web 层。

加载方式

  • BeanFactroy 采用的是懒加载的形式来注入 Bean 的，即只有在使用到某个Bean时（调用 getBean()）才对该 Bean 进行加载实例化。
  • 这样，我们就不能第一时间发现一些存在的 Spring 的配置问题。如果 Bean 的某一个属性没有注入，BeanFacotry 加载后，直至第一次使用调用 getBean() 方法才会抛出异常。
  • ApplicationContext，它是在容器启动时，一次性创建了所有的 Bean。这样，在容器启动时，如果存在的配置错误就会直接报错，这样有利于检查所依赖属性是否注入。
  • ApplicationContext 启动后预载入所有的单实例 Bean，当你需要使用的时候，无需等待，因为它们已经创建好了，可以直接使用。
  • 相对于基本的 BeanFactory，ApplicationContext 唯一的不足是占用内存空间。当应用程序配置 Bean 较多时，程序启动较慢。

2.4 IoC中bean标签和管理对象细节

bean.xml是一个配置文件，该文件内所有的数据都被用来根据反射创建对象。

1 bean标签

  1. 作用：

    • 用于配置要添加到容器中的对象的信息（id和全限定类名）。
    • 默认情况下，工厂创建实例对象时使用的是类的无参构造函数，如果没有无参构造函数则不能创建成功。

  2. 属性：

    • id：是对象在spring容器中的key，用于获取对象。
    • class：指定类的全限定类名。[全类名被用于通过反射创建对象，默认情况下调用无参构造函数初始化。]
    • scope：指定对象的作用范围。
      1. singleton：单例的。默认值
      2. prototype：多例的
      3. request：一次请求范围内
      4. session：一次会话范围内
      5. global session：全局session，一个集群范围内。如果没有集群，等价于session
    • init-method：指定类中初始法方法的名称。
    • destroy-method：指定类中销毁方法的名称

2 Bean的作用范围和生命周期

  1. 单例对象 [scope=”singleton”]

    • 一个容器中只有一个该类的实例对象。它的作用范围就是整个应用[即无论哪里用到该对象，都是同一个对象]。

    • 生命周期：

      1. 对象出生：当应用程序加载时，读取了配置文件，创建了Spring容器时，单例对象就被创建了。

      2. 对象活着：只要容器还存在，单例对象就一直活着。

      3. 对象死亡：当应用程序关闭，容器被销毁时，对象就被销毁了。

  2. 多例对象 [scope=”prototype”]

    • 一个容器中可以有无数个该类的实例对象。每次需要使用该对象时，就会重新创建实例对象。

    • 生命周期：

      1. 对象创建：spring容器被创建时，对象并没有被创建。当程序中的某一步骤需要使用该对象时，才会创建对象的实例。[懒加载]
      2. 对象活着：只要程序运行中某一步骤还需要使用该对象，多例对象就一直活着。
      3. 对象死亡：当对象长时间不用时，由Java的垃圾回收器回收。

1、依赖注入的概念

DI的设计原则允许我们移除硬编码依赖来降低程序间的耦合度。

我们的程序在编写时，通过控制反转，把对象的创建交给了spring，但是代码中不可能出现没有依赖的情况。IoC解耦只是降低它们的依赖关系，而不会消除。

4.2.1、junit单元测试的原理

  1. 所有程序的入口：
    • main方法
  2. junit单元测试中，没有main方法也能执行的原理：
    • junit单元测试集成了一个main方法
    • 该集成方法会判断当前测试类中哪些方法有@Test注解
    • junit就会让有@Test注解的方法执行。
  3. junit不会关注我们是否使用了spring框架。
    • 也就是说，在执行测试方法时，junit根本不知道我们是否使用了spring框架。
    • 所以也就不会为我们读取配置文件/配置类，创建spring容器。
  4. 由以上3点可知：
    • 当测试方法执行时，就算添加@Autowired注解，由于没有spring核心容器，也无法实现注入。
  5. 解决方案：
    • 但是junit给我们提供了一个注解，让我们可以替换掉它的运行器。
    • 只要把junit的运行器替换成可以自动读取配置文件，创建spring容器的运行器，就可以解决上述问题了。
    • Spring框架正好提供了一个运行器，可以自动读取配置文件/配置类来创建spring容器。我们只需要告诉它配置文件在哪里就行了。

5.1.1、存在的问题

  • 存在的问题：

    1. 事务被自动控制了[默认]。换言之，我们使用了connection对象的setAutoCommit(true)。

    2. 此方法控制事务，如果我们每次只执行一条SQL语句，出现异常时可以实现回滚操作。

    3. 但是如果一个业务方法包含多条SQL语句，当业务出现异常时，就无法实现全部业务的回滚操作了。

      • 当我们执行转账业务时，由于执行过程中出现异常，会导致转账失败。但是因为我们每次执行的持久层方法都是独立事务，导致无法实现事务控制（不符合事务的一致性）。
      • 每次自动提交，都会执行一遍开启事务、提交事务、回滚事务、关闭事务的过程

      1. accountDao.findByName(sourceName);：完成了按姓名查找的功能，并成功提交事务。
      2. accountDao.findByName(targetName);：完成了按姓名查找的功能，并成功提交事务。
      3. accountDao.update(sourceAccount);：完成了更新sourceAccount账户的功能，该账户中的余额=原本的余额-money，并成功提交事务。
      4. int i = 1/0；：除0异常，程序终止。
      5. accountDao.update(targetAccount);：更新targetAccount账户的事务未能执行，但是由于之前的三个事务与该事务独立，所以回滚的时候并不会撤销之前的操作。这就造成了转出账户余额减少，而转入账户余额未增加的问题。

5.2.1、是什么

AOP；Aspect Oriented Programing；面向切面编程

  • 面向切面编程：通过预编译方式和运行期动态代理实现程序功能的统一维护的一种技术。
  • AOP 是 OOP 的延续，是软件开发中的一个热点，也是Spring框架中的一个重要内容，是函数式编程的一种衍生泛型。利用 AOP 可以对业务逻辑的各个部分进行隔离。从而使得业务逻辑各部分之间的耦合度降低，提高程序的可重用性，同时提高了开发效率。
  • 简单的说，AOP就是把我们程序中重复的代码抽取出来，在需要执行的时候，使用动态代理的技术，在不修改源码的基础上，对我们已有的方法进行增强。

我更愿意称 面向切面编程 为面向业务编程，何为面向业务编程？每个大的业务都可以划分为粒度更小的业务，在很多个大的业务中，肯能有一些小的业务需要被重复使用，比如：日志业务、登录验证业务等。由于这些小的业务模块需要在很多大业务模块中被使用，为了避免在每个大业务模块中都编写这类重复的代码，我们可以把这些可能会多次用到的小业务模块抽取出来单独管理，在大业务模块需要使用时动态的“织入”进去，这就是所谓的 AOP（面向切面编程）。

听完上面的话，你是不是觉得这跟抽取一个公共方法没有什么两样？那么 AOP 和抽取公共方法相比最大的优势是什么呢？就是动态代理！！！抽取公共方法的方式虽然也提高了利用率，但是需要修改原先业务逻辑中的源代码，即如果某个模块需要使用到日志功能，那么它必须将日志模块抽取成的公共方法写到业务逻辑中，而动态代理则没有这个缺点，它不需要修改原先业务的代码逻辑，而是通过一个代理对象来调用原先的业务逻辑，并将增强的方法写在代理对象中，不会修改原业务逻辑

6 Spring中的AOP

​ AOP 就是纵向的编程，比如业务 1 和业务 2 都需要一个共同的操作（使用同一段代码），甚至可能未来的业务 n 也会使用到这段代码，为了提高代码的复用性，就会把这段代码抽取出来，让多个业务共同使用这段代码。AOP 就是把所有公共的代码抽取出来，放置到某个地方集中管理，然后在具体运行时，再由容器动态织入这些共有代码。

6.1 Spring中AOP的细节

6.1.1、说明

  • Spring中的AOP，就是通过配置的方式，实现动态代理的功能。

6.1.2、AOP中的相关术语

1、Joinpoint（连接点）

  • 所谓连接点就是指那些可以被拦截到的点。
  • 在Spring中，这些点指的是方法，因为Spring只支持方法类型的连接点。
  • 【所谓的拦截点(方法)，就是指可以被增强的方法，只有拦截到这些方法后才能对其增强】【可以理解为被代理对象中的方法】

2、Pointcut（切入点）

  • 所谓切入点是指我们要对哪些Joinpoint进行拦截。
  • Pointcut一定是Joinpoint，但是joinpoint不一定是Pointcut。
  • 【在代理对象中，我们可以对代理的方法进行增强，也可以选择不增强，按照代理前的方法输入。但是这些方法都是被拦截到的点。 我们把被拦截并增强的方法叫做Pointcut，而只是完成了拦截的点叫做Joinpoint】

3、Advice（通知/增强）

  • 所谓通知就是指拦截到Joinpoint之后所要做的事情就是通知。
  • 通知的类型：
    1. 前置通知
    2. 后置通知
    3. 异常通知
    4. 最终通知
    5. 环绕通知
  • 【通知就是我们编写的增强方法，在不修改源码的基础上新增功能的代码】

4、Introduction（引介）

  • 引介是一种特殊的通知，在不修改原代码的前提下，Introduction可以在运行期为类动态的添加一些方法或Field。

5、Target（目标对象）

  • 被代理的对象

6、Weaving（织入）

  • 是指把增强方法应用到目标对象（被代理对象）来创建代理对象的过程。
  • Spring采用动态代理织入，而AspectJ采用编译期织入和类转载期织入。

7、Proxy（代理）

  • 一个类被AOP织入增强后，就产生一个代理类。

8、Aspect（切面）

  • 是切入点和通知（引介）的结合。
  • 【只有切入点没有通知或只有通知没有切入点都无法实现增强，只有切入点+通知，才能完成一次织入。完成织入的方法叫做一个切面】

6.1.3、学习Spring中的AOP要明确的事

  1. 开发阶段
    • 编写核心业务代码；
    • 把公用代码抽取出来，制作成通知；
    • 在配置文件中，声明切入点与通知间的关系，即切面。
  2. 运行阶段（Spring框架完成的）
    • Spring框架监控切入点方法的执行。
    • 一旦监控到切入点方法被调用，就使用代理机制，动态创建目标对象的代理对象，根据通知类别，在代理对象的对应位置，将通知对应的功能织入，完成完成的代码逻辑运行。

6.1.4、关于代理的选择

在Spring中，框架会根据目标是否实现了接口来决定采用哪种动态代理的方式。【动态代理分为基于接口的动态代理和基于类的动态代理】

      void cn.xsyu.service.impl.AccountServiceImpl.saveAccount()

    • 返回值可以用通配符：

      * cn.xsyu.service.impl.AccountServiceImpl.saveAccount()

    • 方法名可以使用通配符

    • 参数列表可以使用..表示有无参数均可。有参数可以是任意类型。

      * cn..*.*(..)

    • 全通配方式

      * *..*.*(..)

    • 通常情况下，我们都是对业务层的方法进行增强，所以切入点表达式都是切到业务层实现类。

      execution(* cn.xsyu.service.impl.*.*(..))

数组 vs. 集合

1 集合与数组存储数据概述

  • 集合、数组都是对多个数据进行存储操作的结构，简称 Java 容器。
  • 说明：此时的存储，主要指的是内存层面的存储，不涉及到持久化的存储（.txt,.jpg,.avi，数据库中)

2 为什么要有集合？

  • 为了解决数组存储数据方面的弊端。

3 数组存储的弊端

  1. 一旦初始化以后，其长度就不可修改；
  2. 数组中提供的方法非常限，对于添加、删除、插入数据等操作，非常不便，同时效率不高；
  3. 获取数组中实际元素的个数的需求，数组没有现成的属性或方法可用；
  4. 数组存储的数据是可重复的。对于不可重复的需求，不能满足。

总结：长度固定、类型固定、功能少。

4 集合和数组的区别：

数组 集合
1 长度是固定的 长度是可变的
2 只能存储同一类型的元素 可以存储不同类型的元素[但不建议]
3 可以存储任意类型的数据 只能存储引用类型数据对象（即存储基础类型数据时会自动装箱）

HashSet

特点

数据特点具体分析：

以 HashSet 为例说明：

【有序性是数据存储时按照插入的先后顺序排列的】

  1. 无序性：不等于随机性。存储的数据在底层数组中并非照数组索引的顺序添加，而是根据数据的哈希值决定的。
  2. 不可重复性：保证相同的元素在添加时，根据 equals() 判断不能返回 true。即相同的元素只能添加一个。

底层数据结构：

HashSet 底层是通过 HashMap 实现的，底层也是一个数组（HashTable），初始容量16，当如果使用率超过 0.75（160.75=12）就会扩大容量为原来的 2 倍。*

  • 在 JDK7中，HashMap 的底层还是通过：数组+链表的结构 实现的。
  • 从 JDK8 开始，HashMap 的底层结构更改为链表长度大于等于 8 时，开始由链表转换为红黑树。

元素添加过程

  1. 我们向 HashSet 中添加元素 a 时，首先调用元素 a 所在类的 hashCode() 方法，计算元素 a 的哈希值，
  2. 此哈希值接着通过某种算法计算出在 HashSet 底层数组中的存放位置（即：索引位置）
  3. 判断数组此位置上是否已经元素：
    • 如果此位置上没有其它元素，则元素 a 添加成功。【Case1】
    • 如果此位置上其他元素 b（或以链表形式存在的多个元素，则比较元素 a 与元素 b 的 hash 值）：
      • 如果 hash 值不相同，则元素 a 添加成功。【Case2】
      • 如果 hash 值相同，进而需要调用元素 a 所在类的 equals() 方法：
        • equals() 返回 true，元素 a 添加失败。【Case3】
        • equals() 返回 false，则元素 a 添加成功。【Case4】

HashMap 底层结构

JDK7

  1. 执行 new HashMap() 的指令，底层会创建一个长度为16 的一维数组 Entry[] table；
  2. ….可能已经执行过多次 put …
  3. 执行 map.put(key1,value1) 方法：
    • 首先，调用 key1 所在类的 hashCode() 计算 key1 的哈希值，此哈希值经过某种算法计算以后，得到在 Entry 数组中的存放位置。
      • 如果此位置上的数据为空，此时的 key1-value1 添加成功。【Case1】
      • 如果此位置上的数据不为空（意味着此位置上存在一个或多个数据(以链表形式存在)），比较 key1 和已经存在的一个或多个数据的哈希值：
        • 如果 key1 的哈希值与已经存在的数据的哈希值都不相同，此时 key1-value1 添加成功。【Case2】
        • 如果 key1 的哈希值和已经存在的某一个数据 (key2-value2) 的哈希值相同，继续比较.调用 key1 所在类的equals(key2)方法，比较：
          • 如果 equals()返回 false，此时 key1-value1 添加成功。【Case3】
          • 如果 equals() 返回 true，使用 value1 替换 value2。【Case4】

Tip：

  • 在不断的添加过程中，会涉及到扩容问题，当超出临界值 0.75 (且要存放的位置非空)时，扩容。
  • 默认的扩容方式：扩容为原来容量的 2 倍，并将原的数据复制过来。

JDK8：HashMap 在 JDK8 中相较于 JDK7 在底层实现方面的不同：

  1. new HashMap() 底层没有创建一个长度为16的数组，而是使用一个类加载阶段就创建好的空数组；
  2. JDK8 底层的数组是：**Node[]**，而非Entry[]；
  3. 首次调用put()方法时，底层才会创建长度为16的数组；
  4. JDK7 底层结构：数组+链表。JDK8 中底层结构：数组+链表+红黑树。
  5. 当数组的某一个索引位置上的元素以链表形式存在的数据个数 > 8 且当前数组的长度 > 64时，此时此索引位置上的所数据改为使用红黑树存储。[链表长度 >= 8 && 数组长度 =< 64]

HashMap 底层结构典型属性的说明（源码中属性的说明）：

  • DEFAULT_INITIAL_CAPACITY ：HashMap的默认容量，16
  • DEFAULT_LOAD_FACTOR：HashMap的默认加载因子，0.75；
  • threshold：扩容的临界值， 等于 容量*填充因子：16 * 0.75 => 12；
  • TREEIFY_THRESHOLD：Bucket 中链表长度大于该默认值时转化为红黑树，8；
  • MIN_TREEIFY_CAPACITY：桶中的Node被树化时最小的 hash 表容量，64

1、内存的概念

内存是用来存储临时数据的（即断电后数据消失），它主要负责：硬盘等硬件上的数据和CPU之间的数据交换处理，也起到缓存系统中的临时数据的作用。在计算机中，它直接和CPU交换数据。

内存，外存和CPU的关系

  • 我们编写的程序是存放在硬盘中的，在硬盘中的程序是不会运行的，必须先加载进内存中，然后由CPU运行程序，运行完毕后会清空内存。
  • JVM虚拟机要运行程序，必须要对内存进行空间的分配和管理。

为什么4个字节的 float 表示数值的范围比8个字节的 long 还大？

整数在计算机底层采用补码的形式表示，除去首位的符号位，剩余的位数即表示数值的范围，也就是位数越多，范围越大。

指数位有 8 位，即最高取到 128，底数固定为 2，即最大范围约等于 2^-128^ ~ 2^128^ 之间，显然大于 long 类型的数据。

方法重载指在同一个类中，允许存在一个以上的同名方法，只要它们的参数列表不同即可，与修饰符和返回值类型无关。

方法重载的条件：

  2. 相同方法名
  3. 参数列表不同

    • 数据类型不同；
    • 参数类型排列顺序不同。
  4. 跟方法的权限修饰符、返回值类型、形参变量名、方法体都没关系！

7.5 重写

方法重写描述的是子类和父类之间的方法关系，而重载指的是同一类中方法的关系。

定义：

子类继承父类以后，可以对父类中同名同参数的方法，进行覆盖操作。

重写以后，当创建子类对象以后，通过子类对象调用子父类中的同名同参数的方法时，实际执行的是子类重写父类的方法。

方法重写的原则：“两同两小一大”

  • 子类重写方法的 方法名 和 形参列表 必须和父类被重写的方法的方法名和形参列表相同。【两同】
  • 子类重写的方法的权限修饰符大于父类被重写的方法的权限修饰符。【一大】
    • 特殊情况：子类不能重写父类中声明为 private 权限的方法【继承都继承不到，自然谈不上重写了】
  • 返回值类型： - 父类被重写的方法的返回值类型是 void，则子类重写的方法的返回值类型只能是 void； - 父类被重写的方法的返回值类型是基本数据类型(比如：double)，则子类重写的方法的返回值类型必须是相同的基本数据类型(必须也是double) - 父类被重写的方法的返回值类型是 A 类型，则子类重写的方法的返回值类型可以是 A 类或 A 类的子类（一小）
  • 子类重写的方法抛出的异常类型小于父类被重写的方法抛出的异常类型（两小）

12 继承

是什么？

定义

多个类中存在相同属性或行为时，将这些内容抽取到单独一个类中，那么多个类无需再定义这些属性和行为，只要继承那一个类即可。

继承：

  • 子类继承父类的非私有和非静态的属性和方法，使得子类对象具有与父类对象相同的属性、方法。
  • 子类可以直接访问父类中的非私有和非静态的属性和行为。

好处：

  1. 提高代码的复用性
  2. 类与类之间产生了关系，是多态的前提。

继承的特点

  1. 一个类可以被多个子类继承。
  2. 单继承：一个类只能有一个父类
  3. 子类直接继承的父类，称为 直接父类；间接继承的父类称为 间接父类。
  4. 子类继承父类以后，就获取了直接父类以及所间接父类中声明的属性和方法。

为什么要有？

  1. 减少了代码的冗余，提高了代码的复用性。【像Java中很多顶级父类，一直从JDK1.1、1.2的版本沿用至今，很多都是利用类的继承性】
  2. 便于功能的扩展
  3. 多态性的前提

继承后的特点

1. 成员变量

  • 子父类中成员变量无重名：无影响

  • 子父类中成员变量有重名：有影响，通过子类对象访问父类中的非私有成员变量时，需要使用super关键字。

2. 普通方法

  • 子父类中成员方法无重名：无影响

  • 子类的无参构造器中会隐式地调用 super() 方法来调用父类的构造方法，这个隐式调用在编译后位于构造方法的第一行。

13 抽象类

13.1 概述

父类中的方法，往往被多个子类重写，子类各自的实现方式不尽相同。所以父类中方法的方法声明和方法体，大多数时候只有方法声明还有意义，而方法主体则没有实际意义了。为了方便，大多数时候我们在父类中只编写方法声明，而把具体的方法体交给子类去实现。

  • 我们把没有方法体并使用abstract修饰的方法，叫做抽象方法。
  • 把使用abstract修饰的类，叫做抽象类。[抽象类中可以没有抽象方法]

13.2 abstract

可以修饰的结构：类、方法

抽象类

  • 抽象类不能实例化。
  • 但是抽象类中一定有构造器，便于子类实例化时调用。
  • 开发中，都会提供抽象类的子类，让子类对象实例化，完成相关的操作。
  • 抽象类中，不一定包含抽象方法，但是包含抽象方法的类必定是抽象类。
  • 抽象类的子类必须重写父类中所有的抽象方法。否则，该子类也必须声明为抽象类。

抽象类结构

必须有：

  1. 构造方法

可以有：

  1. 抽象方法
  2. 普通属性、方法
  3. 静态属性、静态方法。

总结：抽象类和普通类只有两点不同：

  • 抽象类必须用 abstract 修饰外；
  • 抽象类中可以包含抽象方法。

抽象方法

注意

  1. abstract 不能用来修饰：属性、构造器等结构
  2. abstract 不能用来修饰私有方法、静态方法、final 的方法、final 的类。

理解：

abstract 的作用就是定义顶级的接口和方法，一般只起一个声明的作用，而不做具体实现，具体实现交由不同的子类去完成。而我们上面的这些方法：构造方法、私有方法、静态方法、final 方法，它们都有一个共同的特点，就是每个方法都不能被重写。这些方法都不能被子类去重新实现，也就是说如果你把这些方法声明为抽象的，它们就会永久成为抽象的了，这样就失去了定义方法的意义。

总结：abstract 修饰的方法，必须要能被重写。

14 接口

  • 接口用来定义一种对外的规范。
  • 接口是一种引用类型，是方法的集合。

14.1 接口中的成员

JDK8 之前：接口中只能定义全局常量和抽象方法。

  • 全局常量：public static final的.但是书写时，可以省略不写
  • 全局抽象方法：public abstract 的方法
  • 注意：一定要是全局 public 的，不是全局的就会产生编译时异常。
  1. 从 JDK8 开始：接口中除了定义全局常量和抽象方法之外，还可以定义静态方法、默认方法。
  • 注意：无论是静态方法还是默认方法，前提都必须是全局方法。
  1. 从 JDK9 开始：接口中新增可以定义私有（private）方法了。
  • 接口中的私有方法用于改善接口内部的代码可重用性。比如多个 default 方法中存在重复的代码可以抽取出一个 private 方法供接口内部调用。但是这个方法是不会对外暴露的，所以说接口对外暴露的方法都要求必须是全局(public)的。
  • 可以说正是因为 JDK8 的改动才导致 JDK9 需要在接口内部新增可以定义私有方法。因为在 JDK8 之前，接口内部是不存在方法实现的，从 JDK8 开始新增了 默认方法 和 静态方法 后，接口内部才可以写一些方法实现了。

总结

  1. 接口中对外暴露的方法必须是全局方法。

  2. 接口中只能定义方法和常量，变量和代码块都不能在接口中定义。

  3. 接口中没有构造方法，不能被实例化。

  4. 接口中的默认方法（default 修饰的方法）与类中的默认权限修饰符方法是不同的。

  5. 如果实现类实现了接口中的所有抽象方法，则此实现类就可以实例化了；如果实现类没有实现接口中的所有抽象方法，则此实现类仍为一个抽象类，仍然不能被实例化。

  6. Java类可以实现多个接口。【弥补了Java单继承性的局限性】

  7. 接口与接口之间可以继承，而且可以多继承。

14.3 接口中的成员详解

1 抽象方法

  1. 使用public abstract abstract 修饰，没有方法体，该方法供子类实现使用。
  2. 接口中有多个抽象方法并且抽象方法有重名时，只需重写一次。

2 默认方法（JDK8++）

  1. 使用public default修饰，供子类调用或重写。

  2. 实现类可以继承，可以重写默认方法，但是只能通过实现类的对象来调用（因为接口不能创建实例对象）。[实现类的对象又可以使用接口来表示，这就是多态，父类引用指向子类对象]

3 静态方法（JDK8++）

  1. 使用public static 修饰，只能通过 接口名.方法名 直接调用。【因为接口不能创建实例对象】
  2. 静态方法保存在内存中的方法区，只能使用接口名调用，不能通过实现类名或实现类对象调用。
  • 普通类中的静态方法，既可以通过类名调用，也可以通过实例对象调用。
  • 接口中的静态方法，不能够通过 实现类类名 和 实现类对象 调用。
  • 普通类中，静态属性和静态方法只保留一份在方法区中，实例对象中保存着 Class 对象的地址，所以可以通过实例对象调用 Class 对象中的属性和方法。
  • 但是在接口中，子类实现接口的时候 私有方法 和 静态方法 不会继承给子类，所以子类成员中就已经失去了私有方法和静态方法，显然实例化时也没有私有方法和静态方法的地址信息，所以不能通过 实现类类名 和 实现类对象 调用。*
  1. 接口中，存在同名的静态方法并不会冲突，因为静态方法不会继承给实现类，只能通过各自的接口名访问。

4 私有方法

  1. 使用private关键字修饰，在接口中可以省略，供接口中的默认方法或静态方法调用。

    • 私有方法：只有默认方法可以调用。
    • 私有静态方法：默认方法和静态方法都可以调用。

  2. 如果一个接口中有多个默认方法，并且方法中有重复的内容，那么就可以抽取出来，封装到私有方法中，供默认方法调用【类似于工具类】。

14.4 接口的实现

类与接口的关系叫做实现关系。即类实现接口。

该类可以称为接口的实现类，也可以称为接口的子类。

1 非抽象子类实现接口

  1. 必须重写接口中所有抽象方法。
  2. 继承了接口中的默认方法，既可以直接调用，也可以重写。
  3. 无法继承接口中的私有方法和静态方法。

2. 接口的多实现

  • 一个类只能继承一个父类。
  • 但是，一个类可以实现多个接口。
  • 而且，一个类可以继承一个父类的同时，实现多个接口。

优先级问题

  • 当一个类，既继承一个父类，又实现若干个接口时，父类中的成员方法与接口中的默认方法重名，子类就近选择执行父类的成员方法。

    extends > 接口1 > 接口2 > ...

14.5 接口的多继承

  • 一个类只能直接继承一个父类，
  • 但是，一个接口能继承另一个或多个接口。
  • 接口的继承使用extends关键字。
  • 子接口继承多个父接口，如果父接口中的方法有重名的，那么子接口需要重写一次。

附1：

  1. 子接口重写默认方法时，default关键字可以保留。
  2. 子类(实现类)重写默认方法时，default关键字不可以保留。

14.6 接口中的注意事项

  1. 接口中，无法定义成员变量，但是可以定义成员常量，其值不可以改变，使用public static final修饰。
  2. 接口中，没有构造方法，不能创建对象。
  3. 接口中，只能有方法和常量，不能有代码块（包括静态代码块和非静态代码块）。
  4. 类优先原则：如果子类同时继承了父类和实现了接口中声明了同名同参数的默认方法，那么子类在没重写此方法的情况下，默认调用的是父类中的同名同参数的方法。
  5. 接口冲突原则：如果实现类实现了多个接口，而这多个接口中定义了同名同参数的默认方法，那么在实现类没重写此方法的情况下会报错。这就需要我们必须在实现类中重写此方法
  6. 通过实现类的对象，可以调用接口中的默认方法。如果实现类重写了接口中的默认方法，调用时，仍然调用的是重写以后的方法。
  7. 如果想要在实现类中调用接口中的默认方法，可以通过 super.方法名 来调用。

2. 向下转型

是什么？

  • 向下转型：父类类型向子类类型转换的过程，这个过程是强制的。

  • 将父类实例转换为子类实例可以使用强制类型转换的格式。

为什么要向下转型？

  2. 为了避免在向下转型时出现 ClassCastException 的异常，我们在向下转型之前，会先进行 instanceof 的判断，只有返回为 true 时才进行向下转型；如果返回 false 则不进行向下转型。

Object 类

  1. java.lang.Object类的说明：
  1. Object 类是所有类的根类。
  2. 如果在类的声明中未使用 extends 关键字指明其父类，则默认父类为 java.lang.Object 类。
  3. Object 类中的功能(属性、方法)具有通用性。
    • 属性：无
    • 方法：equals() | toString() | getClass() | hashCode() | clone() | finalize() | wait() | notify() | notifyAll()
  4. Object类只声明了一个空参的构造器
  1. equals() 方法

  1. 只适用于引用数据类型。

  2. Object 类中定义的 equals() 和 == 的作用是相同的，都是比较两个对象的地址值是否相同。

  3. 诸如 String、Date、File、包装类等都重写了 Object#equals() 方法。重写后比较的不是两个引用的地址是否相同，而是比较两个对象的”实体内容”是否相同。

  4. 如果我们自定义的类需要使用 equals() 比较两个对象的”实体内容”是否相同的话。那么我们就需要重写继承自 Object类中的equals() 方法。

  5. 在实际开发中，可以使用 IDE 工具的提供的自动生成功能。

  6. 回顾 == 运算符的使用：

    • 可以用在基本数据类型变量和引用数据类型变量中
    • 如果比较的是基本数据类型变量：比较两个变量保存的数据是否相等。（不一定类型要相同）
    • 如果比较的是引用数据类型变量：比较两个对象的地址值是否相同。
    • 补充：== 符号使用时，必须保证符号左右两边的变量类型一致。
  1. toString()方法

  1. 当我们输出一个对象的引用时，实际上就是调用当前对象的 toString()。

  2. Object 类中 toString() 的定义：

  3. 像 String、Date、File、包装类 等都重写了 Object 类中的 toString() 方法。使得在调用对象的 toString() 时，返回”实体内容”信息。

  4. 自定义类也可以重写toString()方法，当调用此方法时，返回对象的”实体内容”

1、Listener概念

监听器就是监听某个对象状态变化的组件。

常用于监听web应用中某些对象的创建、销毁、增加，修改，删除等动作的发生，然后作出相应的响应处理。

当对象的状态发生变化的时候，服务器自动调用监听器对象中的方法。

  1. 概念：Listener，监听器。Web三大组件之一。
  2. 事件监听机制
    • 事件 ：一件事情
    • 事件源 ：事件发生的地方
    • 监听器 ：一个对象
    • 注册监听：将事件、事件源、监听器绑定在一起。 当事件源上发生某个事件后，执行监听器代码

2、Listener的实现

2.1 实现步骤

  1. 自定义一个监听器类，实现ServletContextListener接口。
  2. 重写接口中的方法，完成监听的逻辑。
  3. 在配置文件web.xml中（或使用注解）配置监听对象。

2.2 使用web.xml配置文件配置

  • 在配置文件中将自定义的监听器类注册到监听器列表中去。

2.3 注解配置

  • 在自定义的监听器类上使用@WebListener注解将该类注册到监听器列表中去。

3、ServletContextListener

  • 作用：监听ServletContext对象的创建和销毁

  • 方法：

    1. void contextDestroyed(ServletContextEvent sce)
      • ServletContext对象被销毁之前会自动调用该方法
    2. void contextInitialized(ServletContextEvent sce)
      • ServletContext对象创建之后会自动调用该方法。

1. Filter原理

Filter是对客户端访问资源的过滤，符合条件放行，不符合条件不放行，并且可以对目标资源访问前后进行逻辑处理。

Filter主要用于对用户请求（request）进行预处理，也可以对服务器响应（response）进行后处理。

  • 当浏览器访问服务器的资源时，过滤器可以将请求拦截下来，完成一些特殊的功能。
  • 过滤器一般用于完成通用的操作。如，登录验证、统一编码处理、敏感字符的操作等。

2. Filter实现

2.1 实现步骤

    • 注解配置方式

2.2 过滤条件

2.2.1、通过访问路径拦截请求消息

  1. 拦截具体资源
    • /index.jsp 只有访问index.jsp时，才会执行该过滤器。
  2. 通过目录拦截
    • /user/* 访问/user目录下的所有资源时，过滤器都会被执行。
  3. 通过后缀名拦截

3. 过滤器细节

3.1 过滤器执行流程

  1. 请求消息到达过滤器后首先执行过滤器中放行前的业务逻辑；
  2. 放行后，跳转到Servlet转去执行Servlet中的业务逻辑；
  3. Servlet给出的响应消息返回给客户端时再次经过过滤器，执行过滤器中放行后的业务逻辑。

3.2 过滤器生命周期方法

  1. init()
    • 服务器在启动时，会自动创建Filter实例，该实例调用init()方法。只执行一次。用于加载资源。
  2. doFilter()
    • 每一次请求被拦截资源时，都会执行。执行多次。
  3. destroy()
    • 服务器正常关闭时，会调用destroy()方法，Filter对象被销毁。只执行一次，用于释放资源。

3.5 过滤器链（配置多个过滤器）

3.5.1. 过滤器执行顺序：栈式原则。

​ 如果有2个过滤器，过滤器1和过滤器2。其执行顺序为：

  1. 过滤器1
  2. 过滤器2
  3. 访问资源
  4. 过滤器2
  5. 过滤器1

3.5.2. 过滤器先后顺序问题

  • 如果有2个过滤器，过滤器1和过滤器2，如何判断是过滤器1先执行还是过滤器2先执行？

1. 注解配置

  • 按照类名的字符串比较规则，值小的先执行。

2. web.xml配置

  • 在配置文件中，谁定义在前面谁先执行。
