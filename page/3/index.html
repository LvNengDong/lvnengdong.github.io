Spring TX；Spring事务管理

基于 XML 配置文件的事务管理

1 创建事务管理器对象

  • Spring 框架为我们提供了管理事务的对象 DataSourceTransactionManager，这个对象中集成了事务的开启、提交、回滚、关闭等操作。
  • 使用前需要先将 DataSourceTransactionManager 对象添加到 IoC 容器中。
  • DataSourceTransactionManager 对象依赖于 DataSource 对象。

2 配置事务管理器的通知（增强）

事务管理器的通知（简称“事务的通知”）是事务管理器的引用，如果不对其进行动态代理增强的话，就等价于事务管理器。

事务的通知的主要作用就是在通知中动态地添加一些额外的功能。比如：可以在事务的通知中定义事务的属性。

3 配置事务的属性

事务的属性被配置在事务的通知中，被 <tx:attributes> 标签包裹。

<tx:method>

<tx:method> 标签主要用于管理指定的方法。包含如下属性：

  • name：指定要被事务管理的方法的方法名，可以模糊指定或精确指定。【优先级：精准匹配>部分匹配>完全模糊匹配】
  • isolation：用于指定事务的隔离级别。默认值是 Default，表示使用数据库的默认隔离级别。【MySQL 和 Oracle 不同】
  • propagation：用于指定事务的传播行为。默认值是 REQUIRED。
  • read-only：用于指定事务是否只读。只有查询方法才能设置事务为 true。默认值是false，表示读写。
  • timeout：用于指定事务的超时时间。默认值是-1，表示永不超时，如果指定了数值，以秒为单位。
  • rollback-for：用于指定一个异常，当产生该异常时，事务回滚，当产生其它异常时，事务不会滚。没有默认值，表示任何异常都会滚。
  • no-rollback-for：用于指定一个异常，当产生该异常时，事务不会滚。产生其它异常时，事务回滚，没有默认值，表示任何异常都回滚。

  • 即要对哪个（些）方法施加配置好的事务。

5 将事务通知织入到切入点

Tip：

  • 一般情况下，AOP 中的切面使用 <aop:aspect> 标签；

6 总结（重要）

  • 为了保证代码的可读性和易维护性，我们在 Service 层编写的对数据库的操作都是屏蔽了事务操作的、简单易懂的代码。

    • 所以 Spring 配置文件中的切入点就是 Service 层中实现类，再通过事务的属性 <tx:method name=""> 定位到实现类中的具体方法，给对应的方法添加配置文件中配置好的事务；

@Transactional 注解

作用：该注解的属性与 XML 配置文件中的事务属性的含义一致。

作用范围：该注解可以在接口上、类上和方法上使用。

  • 在接口上使用，表示该接口的所有实现类都支持该事务；
  • 在类上使用，表示该类中的所有方法都支持该事务；
  • 在方法上使用，表示该方法支持该事务。
  • 以上 3 个位置的优先级：方法> 类> 接口

事务的传播行为

是什么？

事务的传播行为：当一个方法运行在另一个事务方法中时，必须指定事务该如何传播。

例如：方法可能继续在现有事务中运行，也可能开启一个新事务，在新事务中运行。

事务的传播行为可以由传播属性指定。Spring 中定义了 7 种事务传播行为。

传播属性 描述
REQUIRED（默认） 如果有事务正在运行，当前方法就在这个事务内运行，否则，就启动一个新的事务，并在新的事务内运行。
REQUIRES_NEW 当前方法必须启动新事务，并在它自己的事务（新事务）内运行。如果有事务正在运行，应该将其挂起。
SUPPORTS 如果有事务正在运行，当前方法就在这个事务内运行；否则它可以不运行在事务中。
NOT_SUPPORTED 当前方法不应该运行在事务中，如果有运行中的事务，就将其挂起。
MANDATORY 当前方法必须运行在事务内部，如果没有正在运行的事务，就抛出异常。
NEVER 当前方法不应该运行在事务中，如果有正在运行的事务，就抛出异常。
NESTED 如果有事务在运行，当前方法就应该在这个事务的嵌套事务内运行。否则，就启动一个新的事务，并在它自己的事务内运行。

怎么用？

  1. 事务的传播属性可以在 xml 配置文件中配置。
  1. 事务的传播属性可以在 @Transactional 注解的 propagation 属性中定义。

Demo

我们来模拟一个场景：

现有数据库中有 100 本书，每本书 60 元，模拟客户买书，每次只能买一本，每次买书都需要付一次钱，可以买多次。客户共有 100 元，一直买书直到买不起为止。

​ 假设买书方法 buyBook() 和支付方法 pay() 都使用默认的事务传播行为 Required。在这种情况下，内部方法 pay() 和外层方法 buyBook() 用的是同一个事务，这种情况下会发生：多次买书的行为要么全部成功，要么全部失败（事务的原子性）。

​ 就比如在本例中，第一次买书后，客户余额 -60，库存 -1，pay() 方法正常结束，但是第二次买书时由于余额不足以支付，所以支付失败，这时候需要回滚，由于内层方法和外层方法使用的是同一个事务，所以会直接回滚到库存 100，用户余额 100 的状态。

​ 如果把事务传播行为改成 REQUIRES_NEW，在这种情况下，用户第一次买书后，就已经完成了一次事务，已经将数据库中的数据修改并提交了，即使第二次买书失败，也只会回滚到第二次买书前的状态（事务的隔离性），即库存 99，用户余额 40 的状态。

事务的隔离级别

前置知识

数据库事务并发导致的问题：

  1. 脏读。事务 A 读到了事务 B 更新但未提交的数据。
  2. 不可重复读。（多次获取）事务两次读到的数据不一致。
  3. 幻读。（多次获取）事务两次读到表数据的数量不一致。

隔离级别

  1. 读未提交。允许事务 A 读取事务 B 未提交的修改。
  2. 读已提交。要求事务 A 只能读取事务 B 已提交的修改。
  3. 可重复读 。确保一个事务可以多次从一个字段中读到相同的值。即事务 A 执行期间禁止其它事务对这个字段进行更新。【字段上锁】
  4. 串行化。确保一个事务可以多次从一张表中读到相同数量的行。即在事务 A 执行期间，禁止其它事务对这张表进行添加、更新、删除操作。可避免任何并发问题，但性能差。【表上锁】

怎么用？

在 Spring 中，通过 isolation 属性来设置事务的隔离级别。

  • 可以在 xml 配置文件中设置；
  • 也可以在注解中设置。

事务的隔离级别与使用的数据库产品相关联，比如 Oracle 就只支持两种隔离级别，而 MySQL 支持 4 种隔离级别。Spring 中默认使用的是数据库产品默认的隔离级别。如：

  • Isolation.REPEATABLE_READ，可重复读。MySQL 默认的隔离级别。
  • Isolation.READ_COMMITTED，读已提交。Oracle 默认的隔离级别。

是什么

代理模式；Proxy；

代理模式是：为目标对象提供一个代理对象，通过代理对象操作对目标对象，即通过代理对象访问目标对象。通俗的来讲代理模式就是我们生活中常见的中介。

这样做的好处是：可以在目标对象实现的基础上，增强额外的功能，即扩展目标对象的功能。

被代理的对象可以是：远程对象、创建开销大的对象或需要安全控制的对象。

代理模式有不同的形式，主要有两种：

  1. 静态代理
  2. 动态代理：
    • JDK 代理（接口代理）
    • Cglib 代理：可以在内存动态的创建对象，而不需要实现接口

为什么要使用代理模式

  • 中介隔离作用：在某些情况下，如果客户类不方便直接引用一个委托对象，而代理类对象可以在客户类和委托对象之间起到中介的作用，其特征是代理类和委托类实现相同的接口。

  • 开闭原则，增加功能：代理类除了是客户类和委托类的中介之外，我们还可以通过给代理类增加额外的功能来扩展委托类的功能，这样做我们只需要修改代理类而不需要再修改委托类，符合代码设计的开闭原则。代理类主要负责为委托类预处理消息、过滤消息、把消息转发给委托类，以及事后对返回结果的处理等。代理类本身并不真正实现服务，而是同过调用委托类的相关方法，来提供特定的服务。真正的业务功能还是由委托类来实现，但是可以在业务功能执行的前后加入一些公共的服务。例如我们想给项目加入缓存、日志这些功能，我们就可以使用代理类来完成，而没必要打开已经封装好的委托类。

  • 客户端对象（Client）
  • 目标对象/委托对象（TargetObject）
  • 代理对象（ProxyObject）

AQS 原理概述（乞丐版）（以 ReentrantLock 为例）

ReentrantLock 和 AQS 的关系

ReentrantLock 类有一个内部类 Sync 继承了 AbstractQueuedSynchronizer 抽象类，说白了就是 ReentrantLock 中包含了一个 AQS 对象，AQS 对象是 ReentrantLock 实现加锁和释放锁的关键性核心组件。

ReentrantLock 加锁和释放锁的底层原理

  1. 线程 T1 加锁

如果现在有一个线程尝试调用 ReentrantLock 的 lock() 方法进行加锁，会发生什么事情呢？

很简单。这个 AQS 对象内部有一个核心变量 private volatile int state，代表了加锁的状态。初始状态下，state == 0，另外，这个 AQS 内部还有一个关键变量，用来记录当前加锁的是哪个线程，初始化状态下，这个变量是 null。

接着线程 T1 调用 ReentrantLock#lock() 方法尝试进行加锁，这个加锁的过程，就是用 CAS 操作将 state 值从 0 变成 1。

如果 T1 之前没有其它线程加锁锁，那么 state==0 恒成立。此时线程 T1 就可以加锁成功。

一旦线程 T1 加锁成功后，就可以设置当前加锁线程是自己。如下图所示，就是线程 T1 加锁的过程。

说白了，AQS 其实就是 JUC 包中的一个核心组件，里面有 state 变量、加锁线程变量等核心的东西，维护了加锁的状态。

而ReentrantLock 只是一个外层的 API，内核中的锁机制实现都是依赖于 AQS 组件的。

ReentrantLock 顾名思义是一个可重入锁，即可以使用一个 ReentrantLock 对象执行多次 lock() 和 unlock() 方法。

那么 AQS 是如何实现可重入加锁的呢？线程 T1 每调用一次 lock() 方法，如果 CAS 比较发现 state != 0，接着会再判断当前加锁线程是不是自己，如果是就把 state 的值累加 1，其余不变。同理，调用 unlock() 释放锁的时候就是把 state 值累减 1。

  1. 线程 T2 加锁

如果线程 T1 加锁之后，线程 T2 想要争夺锁会怎么样？（互斥锁）

  1. 线程 T2 尝试获取锁，调用 lock() 方法，首先会去检查 ReentrantLock.AQS.state 变量是否为 0，发现不为 0，说明有线程正在使用锁，CAS 校验失败；
  2. 接着线程 T2 会继续查看当前加锁的线程是不是自己，“加锁线程”这个变量明确记录了是线程 T1 占用了这个锁，所以此时线程 T2 会加锁失败。

  1. 加锁失败

接着，线程 T2 会将自己放入 AQS 的同步等待队列中，因为自己尝试加锁失败了，此时就要将自己放入等待队列中，直到线程 T1 释放锁之后，自己就可以重新尝试加锁了。

所以 AQS 内部还有一个同步等待队列，专门存放那些加锁失败的线程。

  1. 释放锁

接着，线程 T1 在执行完自己的业务逻辑代码之后，就会释放锁。释放锁的过程非常简单，就是将 AQS 内的 state 变量的值递减 1，如果 state 为 0，则彻底释放锁，会将“加锁线程”变量的值也设置为 null。

  1. 释放锁之后

接下来，会从等待队列的队头唤醒线程 T2 重新尝试加锁。

线程 T2 重新尝试加锁，还是使用 CAS 操作将 state 从 0 变成 1，此时就会加锁成功，并将 state 设置为 1，同时把 “加锁线程” 设置为线程 T2 自己，同时线程 T2 从同步等待队列中出队。

AQS 初步

​ 多线程环境下，为保证线程安全经常需要加锁，加锁就会导致线程阻塞，有阻塞就需要排队，实现排队必然需要队列（抢不到锁的线程被阻塞后就得去排队）。

​ 抢到资源（锁）的线程直接使用资源处理业务逻辑，抢不到资源的线程必然涉及一种排队等候机制。抢占资源失败的线程继续等待，但等候的线程仍然保留获取锁的可能性。

​ 如果共享资源被占用，就需要一定的阻塞等待和唤醒机制来保证锁分配。这个机制主要是通过 CLH（CLH 是三个科学家的首字母缩写）队列的变体实现的，将暂时获取不到锁（被阻塞）的线程加入到队列中，这个队列就是 AQS 的抽象表现。它将请求共享资源（锁）的线程封装成队列的节点（Node），通过 CAS、自旋以及 LockSupport#part() 等方式，维护 state 变量的状态，使并发达到同步的控制效果。

​ AQS 管理的是没有抢到锁的线程。

AQS 数据结构

AQS 使用一个 volatile int 类型的成员变量 state 来表示同步状态，通过内置的 FIFO 队列来完成资源获取的排队工作，将每个要去抢占资源的线程封装成一个 Node 节点来实现锁的分配，通过 CAS 操作完成对 state 值的修改。

AQS 是一个队列：

  • private volatile int state：同步状态，有 3 种状态。0 表示无人占用，1 表示有线程正在占用，大于 1 表示可重入锁。
  • static final class Node：节点是线程的抽象
  • private transient volatile Node head：代表头结点
  • private transient volatile Node tail：代表尾节点

1 为什么使用线程池？（优势）

​ 线程池的主要工作是控制运行线程的数量，处理过程中将任务放入队列。在线程创建后启动这些任务，如果任务数量超过了最大线程数量，超出数量的任务排队等候，等其它线程执行完毕，再从队列中取出任务来执行。

使用线程池的优势：

  1. 线程复用：创建/销毁线程需要消耗系统资源，线程池可以复⽤已创建的线程。
  2. 控制最大并发数：控制并发的数量。并发数量过多，可能会导致资源消耗过多，从⽽造成服务器崩溃。（主要原因）
  3. 管理线程：线程是稀缺资源，如果无限制的创建，不仅会消耗系统资源，还会降低系统的稳定性，使用线程池可以进行统一的分配、调优和监控
  4. 提高响应速度。当任务到达时，任务可以不要等待线程创建就能立即执行。

2 怎么用

  • 当我们使用线程池时，就不再需要把任务交给某个具体的线程了，我们只需要把任务交给线程池即可，线程池内部会分配一个空闲的线程去执行该任务。

2.1 架构说明

  • 线程池是通过 Executor 框架实现的，该框架用到了 Executors、Executor、ExecutorService、ThreadPoolExecutor 这几个类。

2.2 代码实现（3/5种）

  2. Executors.newSingleThreadExecutor()

2.3 源码分析

查看底层源码发现，这三个创建线程池的方法底层都使用了 ThreadPoolExecutor 类。

  1. newFixedThreadPool
  • 创建一个定长的线程池，可控制线程的最大并发数，超出的线程数量的任务会在队列中等待。
  • newFixedThreadPool 创建的线程池 corePoolSize 和 maximumPoolSize 值是相等的，
  • 底层使用的数据结构是 LinkedBlockingQueue。
  • 执行长期的任务，性能较好

  1. newSingleThreadExecutor
  • 创建一个只包含一个线程的线程池，只会用唯一的线程来执行任务，保证所有任务按照指定顺序执行。
  • newSingleThreadExecutor 将 corePoolSize 和 maximumPoolSize 都设置为 1，
  • 底层使用的数据结构是 LinkedBlockingQueue。
  • 适用于一个任务一个任务执行的场景

  1. newCachedThreadPool
  • 创建一个自适应长度的线程池，即多退少补。
  • newCachedThreadPool 将 corePoolSize 设置为 0，将 maximumPoolSize 设置为 Integer.MAX_VALUE，也就是说来了任务就创建线程执行任务，当线程空闲超过 60s，就销毁线程。
  • 底层使用的数据结构是 SynchronousQueue。
  • 适用于执行很多短期异步的小程序或者负载较轻的服务器。

3 ThreadPoolExecutor 构造方法中的 7 大参数

ThreadPoolExecutor 有多个重载的构造方法，方法中的参数为 5~7 个不等，我们直接看一下最多的 7 个参数。

  1. corePoolSize：线程池中常驻核心线程数
  • 线程池中的常驻线程数，即使没有任务时也需要去维护的线程数量。
    • 当有请求任务进来后，首先会安排常驻线程去执行任务；
    • 当线程池中正在使用的线程数量达到 corePoolSize 后，再有新的任务进来，就会把新进来的任务放到缓存队列中。
  1. maximumPoolSize：线程池能够容纳的最大线程数，此值必须大于等于 1。
  2. keepAliveTime：多余的空闲线程的最大存活时间。[多余线程就是除了常驻线程之外的线程]
  • 如果线程池中的线程数量超过 corePoolSize，并且线程的空闲时间达到 keepAliveTime 值后，多余的空闲线程将会被销毁直到只剩下 corePoolSize 个线程。
    • 只有当前线程池中的线程数大于 corePoolSize 时 keepAliveTime 才会生效，直到线程池中的线程数不大于 corePoolSize。
  1. unit：keepAliveTime 的时间单位。
  2. workQueue：任务队列，被提交但尚未执行的任务。
  3. threadFactory：生成线程池中线程实例的工厂，一般使用默认的即可。不同的工厂都能生产线程实例，就类似于联想和戴尔工厂都能生产电脑，但具体的生产过程是不一样的。
  4. handler：拒绝策略。当阻塞队列满了并且工作线程大于等于线程池的最大线程数时如何拒绝请求任务的策略。
  • 线程池的所有线程都已经在工作了，并且等待执行的任务队列也满了，这时候就需要拒绝新的任务加入，采用的策略就叫做拒绝策略。

4.2 工作流程

工作流程图

在创建了线程池后，线程池会等待提交过来的任务请求。每次调用 execute() 方法添加新的任务请求时，线程池中会进行如下判断：

  1. 如果线程池中正在运行的线程数量小于 corePoolSize，那么马上分配一个空闲线程执行这个任务；
  2. 如果线程池中正在运行的线程数量大于等于 corePoolSize，会将这个任务放入阻塞队列；
  3. 如果这时候阻塞队列也满了且正在运行的线程数量小于 maximumPoolSize，就会创建非核心线程来执行这个任务；
  4. 如果队列满了且正在运行的线程数量大于等于 maximumPoolSize，那么线程池会执行饱和拒绝策略。
  5. 当一个线程执行完一个任务时，它会从阻塞队列中选取下一个任务进行执行。
  6. 当一个线程无事可做超过一定的时间（keepAliveTime）时，线程池会判断：
    • 如果当前线程池中运行的线程数量大于 corePoolSize ，那么这个线程就会被销毁；
    • 所以线程池的所有任务完成后它最终会收缩到 corePoolSize 的大小。

小总结：

​ 任务首先放到线程池的核心池中，放不下就放在阻塞队列，再放不下就对线程池扩容，直到扩容到最大。这时候如果继续有很多线程要进入，启动拒绝策略；如果这时候线程开始逐渐减少，就把线程池中超过空闲时间的线程销毁掉，直到线程池中线程数等于核心线程数。

5 线程池的4种拒绝策略

  1. AbortPolicy（默认）：直接抛出 RuntimeExecutionException 异常，阻止系统正常运行。

  2. DiscardOldestPolicy：抛弃阻塞队列中等待最久的任务，然后把当前任务加入等待队列中尝试再次提交当前任务。

  3. DiscardPolicy：直接丢弃任务，不予任何处理也不抛出异常。如果允许任务丢失，这是最好的一种方案。

  4. CallerRunsPolicy：“调用者“运行一种调节机制，该策略既不会抛弃任务，也不会抛出异常，而是将某些任务回退到调用者，让调用者线程去处理这个任务，从而降低新任务的流量。

    注解：假设 main 线程是线程的调用者，main 线程将一个任务 A 分配给线程池处理，但此时线程池中的运行着的线程已经达到了 maximumPoolSize，即无法再给任务 A 分配一个新的线程了，需要拒绝这个任务。如果采用的拒绝策略是 CallerRunsPolicy，那么任务 A 不会被丢弃，也不会出现异常，而是会将任务 A 会退给 main 线程，让 main 线程去执行任务 A 。

以上 4 种内置的拒绝策略的实现类均实现了 RejectedExecutionHandler 接口。

6 线程在实际开发中的使用（阿里巴巴 Java 开发手册）

  1. 在实际开发中，线程资源必须通过线程池提供，不允许开发人员在程序中自行显式地创建线程。

    注解：使用线程池的好处是减少在创建和销毁线程上所消耗的时间以及系统资源的开销，解决资源不足的问题。如果不使用线程池，有可能造成系统创建大量同类线程而导致消耗完内存或者“过度切换上下文线程”的问题。

  2. 实际开发中，线程池不允许通过 Executors 工具类去创建，而是要通过 ThreadPoolExecutor 的方式去创建，这样的处理方式让开发人员更加明确线程池的规则，规避资源耗尽的风险。

    注解：Executors 创建的线程池对象的弊端如下：

    1. FixedThreadPool 和 SingleThreadPool：

      这两种线程池维护的请求（阻塞）队列虽然是有界队列，但是允许的最大长度为 Integer.MAX_VALUE，这个值实际上非常大，可能会堆积大量的请求，从而导致 OOM。

    2. CachedThreadPool 和 ScheduledThreadPool

      这两种线程池虽然使用的请求队列是同步阻塞队列，最大容量为1，但是允许创建的最大线程数量为 Integer.MAX_VALUE，也可能会创建大量的线程，从而导致 OOM。

不会抛出异常，而是将阻塞队列中等待时间最长的任务丢弃掉。可以看到请求任务有 12 个，而程序只执行的 10 个，阻塞队列中等待时间最长的两个被抛弃掉了。

  1. DiscardPolicy 策略

不会抛出异常，同样是把超出数量的任务抛弃掉，但是在这里抛弃的是容纳不下的最新的任务请求。

如何合理配置线程池参数

CPU密集型任务

  • CPU 密集的意思是该任务需要大量的运算，而没有阻塞，CPU 一直全速运行。
  • CPU 密集任务只有在真正的多核 CPU 上才能得到加速（通过多线程）
  • 而在单核CPU上，无论你开几个模拟的多线程该任务都不可能得到加速，因为CPU总的运算能力有限。
  • CPU 密集型任务的配置尽可能少的线程数量。
  • 一般公式：CPU 核数 + 1个线程的线程池

IO密集型

  • IO 密集型，即该任务需要大量的 IO，即大量的阻塞。
  • 在单线程上运行 IO 密集型的任务会导致大量的 CPU 运算能力浪费在等待。
  • 所以在 IO 密集型任务中使用多线程可以大大的加速程序运行。即在单核 CPU 上，这种加速主要就是利用了被浪费掉的阻塞时间。
  • 由于 IO 密集型任务并不是一直在执行任务，则应该配置尽可能多的线程，如 CPU核数×2

乐观锁 & 悲观锁

乐观锁

​ 乐观锁 对应于生活中乐观的人，乐观的人总是想着事情往好的方向发展。

​ 在并发情况下，对于一个线程来说，最好的情况就是在自己操作共享数据期间没有其它线程去更新这个数据。而对于乐观锁来说，它就很乐观的认为确实会发生这种情况。所以乐观锁操作数据时不会上锁，只是在更新的时候会判断一下在此期间是否有其他线程去更新这个数据。

​ 乐观锁可以使用 版本号机制 和 CAS算法 实现。在 Java 中 java.util.concurrent.atomic 包下的原子类就是使用CAS 乐观锁实现的。

悲观锁

​ 与乐观锁相反，悲观锁 考虑的则是最坏的情况。

​ 一个共享数据加了悲观锁，那么线程每次想操作这个数据前都会假设其他线程也可能会操作这个数据，所以每次操作前都会上锁，这样其他线程想操作这个数据拿不到锁只能阻塞了。

​ 在 Java 中，synchronized 和 ReentrantLock 等就是典型的悲观锁，还有一些使用了 synchronized 关键字的容器类如 HashTable 等也是悲观锁的应用。

总结

  1. 乐观锁 = 不真正加锁 + CAS
  2. 悲观锁：加独占锁，加锁后其它线程的读写操作都会被阻塞

两种锁的使用场景

  1. 乐观锁适用于写比较少（冲突比较小）的场景，因为不用上锁、释放锁，省去了锁的开销，从而提升了吞吐量。
  2. 如果是写多读少的场景，即冲突比较严重，线程间竞争激烈，使用乐观锁就是导致线程不断进行重试，这样可能还降低了性能，这种场景下使用悲观锁就比较合适。

公平锁 & 非公平锁

是什么

公平锁：

指多个线程按照申请锁的顺序来获取锁，即谁先申请锁谁就优先获得锁。类似于生活中的排队，先到先得。

非公平锁：

只多个线程获得锁的顺序并不是按照申请锁的顺序，有可能后申请锁的线程比先申请锁的线程优先获得锁。在高并发情况下，有可能造成优先级反转或者饥饿现象。

  • 优先级反转现象：就是后申请锁的线程比先申请锁的线程优先获得锁的现象，后申请锁的线程本该是一个低优先级的线程，但是出现了优先级反转，它就提前获得了锁对象。（类似于插队）

  • 饥饿现象：饥饿现象是指如果一直存在插队的线程获得锁，就会导致本该获得锁的线程一直处于等待状态，这就叫做饥饿状态。

二者区别

  • 公平锁

在并发环境中，每个线程在争夺公平锁时会先查看此锁对象维护的等待队列，如果为空，或者当前线程是等待队列中的第一个，才占有锁，否则就会加入到等待队列中，按照先进先出的规则等到自己获取锁。

  • 非公平锁

如果锁对象是一个非公平锁，在并发环境中，每当有新线程进入争夺锁对象时，这个线程会直接去尝试占有锁，如果尝试失败，才会再采用类似公平锁的方式，让线程去锁对象维护的等待队列中排队。

在这种情况下，存在这样一种情况：线程 A 是当前持有锁的对象，线程 A 释放锁之后，按照优先级顺序本该线程 B 得到这个锁对象，但是此时有一个新线程 Z 进来了，这个 Z 不管三七二十一首先就去尝试占有锁了，如果这时候刚好处于 A 释放完锁而 B 有没有获取到锁的时间段，那么 Z 就会直接占有锁，而 B 获取不到锁后，只能继续在等待队列中排队了。这就是 B 线程被插队了。

可重入锁（递归锁）

是什么

可重入锁指的是同一线程的外层方法获得锁之后，如果内层的递归方法仍然需要获得该锁，则可以直接使用，不会因为外层方法的锁对象还没释放而阻塞。（前提是锁对象得是同一个对象）

两（多）个需要锁的递归的同步方法，只使用了一把锁，这把锁在进入了最外层方法后，还可以继续在内存的同步方法中使用，即一锁多用

可重入锁就是说某个线程获得某把锁的使用权，在没有归还锁的之前，这把锁可以一直使用！！！

可重入锁的一个优点是可以一定程度避免死锁。

死锁则是一个外层同步方法需要 A 锁才能进入，进入之后内层方法则需要 B 锁才能进入，使用的不是同一把锁。

自旋锁

自旋锁；SpinLock

是什么

概念

自旋锁 是指尝试获取锁的线程在没有获得锁时不会被直接挂起，而是采用循环的方式去不断地尝试获取锁，这个循环就是所谓的自旋。【CAS 思想】

Tip：处于自旋中的线程并不会一直等待获取锁后才结束循环。如果持有锁的线程对锁的占用时间过长，自旋锁在自旋一段时间后仍然无法获取到锁，还是会进入阻塞状态，这时候锁也就不再是自旋锁了。

  • 自旋锁的目的是为了减少线程被挂起的几率，因为线程的挂起和唤醒也都是耗资源的操作。【好处：减少上下文切换的消耗】
  • 但是如果锁被另一个线程占用的时间比较长，即使自旋了之后当前线程还是会被挂起，忙循环就会变成浪费系统资源的操作，因为循环会消耗 CPU，反而降低了整体性能。因此自旋锁是不适应锁占用时间长的并发情况的。【缺点：自旋时间过长时浪费 CPU 资源较多，反而会拖慢程序的整体性能】

案例

​ 有两个线程A、B在竞争一个锁，假设 A 拿到了锁对象，这个时候 B 就会被挂起阻塞，一直等到 A 释放了锁 B 才得到使用权。在操作系统中阻塞和唤醒其实也是一个耗时操作，如果线程 A 在很短的时间内就使用完并释放了锁，当这个时间比 阻塞B线程 + 唤醒B 线程 的总耗时更短的时候，我们将线程 B 挂起，其实并不是一个最优的选择。

扩展：自适应自旋锁

​ 在 JDK1.6 引入了自适应自旋，这个就比较智能了，自旋时间不再固定，由前一次在同一个锁上的自旋时间以及锁的拥有者的状态来决定。如果虚拟机认为这次自旋也很有可能再次成功那就会次序较多的时间，如果自旋很少成功，那以后可能就直接省略掉自旋过程，避免浪费处理器资源。

自旋锁类分析

首先要明确一点，自旋锁在自旋期间并不会挂起任何一个线程。即不像 synchronized 一样同一时间只有一个线程是活跃的。

​ 每个线程执行 myLock() 方法时首先都会尝试获取锁。假设 T1 先执行 myLock()，T1 会先查看锁对象是否被占用。同理，T2 线程执行 myLock() 时也会查看锁对象是否被占用。

​ 如果 T2 查看锁状态时发现锁对象正在被使用，由于自旋锁的特性，T2 不会直接挂起，而是会一直自旋，所以线程的本地内存也就不会被释放。

​ 一段时间后 T1 释放了锁对象后（释放锁对象会修改锁状态的值），T2 要第一时间能得到锁的最新状态。所以自旋锁对象要使用 volatile 关键字修饰，保证多个线程间的可见性。

myLock() 方法

myLock() 方法的作用是根据锁对象的状态，选择让当前线程继续执行还是陷入循环。判断的依据是：

  1. 如果锁对象 atomicReference 是第一次被使用，由于锁对象是使用无参构造函数 new AtomicReference() 创建的，所以主内存中的值 atomicReference.value = null 恒成立，代表此时没有其它线程正在使用该锁对象，当前线程可以获取到锁【其实就是不进入忙循环】。
  2. 在线程 T1 使用锁的 5s 内，线程 T2 也进来了，通过 myLock 方法尝试获取锁对象，由于此时主内存中 value 值已经不为空了（变成了 T1 线程的 currentThread），所以 compareAndSet 方法返回值为 false，取反为 true，线程 T2 进入循环体，不停地做 CAS 判断。
  3. 当线程 T1 执行结束后，会调用 myUnLock() 方法，由于 T1 执行过程中 T2 始终没有更新主内存中的 value 成功，所以此时主内存中的 value 还是与 T1 本地内存中的 currentThread 快照的值相同，T1 可以成功将主内存中的 value 修改为 null。
  4. 由于 value 使用 volatile 修饰，所以当线程 T1 更新 value 后，线程 T2 能马上监听到，这段时间内 T2 一直在执行 myLock 方法，监听到 value 变为 null 后，终于 compareAndSet 方法可以退出循环了，这时候它会把 T2 线程的当前线程对象写回主内存中的 value 位置上，重新由 T2 来占用这把锁。
  5. 在 T2 持有锁的时间段内，如果有其它线程进入也只能不断进行自旋直到 T2 执行完毕，修改主内存中的 value 为 null 后才能继续得到锁。

独占锁 & 共享锁

独占锁

独占锁 是指锁一次只能被一个线程所持有。

如果一个线程对数据加上排他锁后，那么其他线程不能再对该数据加任何类型的锁。获得独占锁的线程即能读数据又能修改数据。

JDK中的synchronized 和 ReentrantLock 就是独占锁。

共享锁

共享锁 是指锁可被多个线程所持有。【共享数据被共享锁锁住，共享锁可被多个线程获取，获取到锁的线程都可以访问共享数据】

如果一个线程对数据加上共享锁后，那么其他线程只能对数据再加共享锁，不能加独占锁。

获得共享锁的线程只能读数据，不能修改数据。

  2. 偏向锁

无锁

  • 乐观锁就是一种无锁状态。

偏向锁

​ 偏向锁(Biased Locking) 是指它会偏向于第一个访问锁的线程。如果在运行过程中，只有一个线程访问加锁的资源，不存在多线程竞争的情况，那么线程是不需要重复获取锁的，这种情况下，就会给线程加一个偏向锁。

​ 偏向锁的实现是通过控制对象 Mark Word 的标志位来实现的，如果当前是 可偏向状态，需要进一步判断对象头存储的线程 ID 是否与当前线程 ID 一致，如果一致直接进入。

轻量级锁

​ 当线程竞争变得比较激烈时，偏向锁就会升级为 轻量级锁，轻量级锁认为虽然竞争是存在的，但是理想情况下竞争的程度很低，通过 自旋方式 等待上一个线程释放锁。

重量级锁

​ 如果线程并发进一步加剧，线程的自旋超过了一定次数，或者一个线程持有锁，一个线程在自旋，又来了第三个线程访问时（反正就是竞争继续加大了），轻量级锁就会膨胀为 重量级锁，重量级锁会使除了此时拥有锁的线程以外的线程都阻塞。

​ 升级到重量级锁其实就是互斥锁了，一个线程拿到锁，其余线程都会处于阻塞等待状态。

​ 在 Java 中，synchronized 关键字内部实现原理就是锁升级的过程：无锁 --> 偏向锁 --> 轻量级锁 --> 重量级锁。

2 CAS

为什么要有 CAS？

CAS 是一种思想，可以保证在多线程环境下，不加锁就能保证共享数据的一致性。

  • 在 JDK5 之前，Java 语言是靠 synchronized 关键字来保证同步的，也就是加锁。锁机制在同一时间段只允许一个线程来修改变量，一致性得到了保障，但并发性下降；
  • 而 CAS 实现了在不加锁的情况下保证共享变量的一致性，并发性高。

CAS 是什么？

CAS；Compare And Swap；比较并交换

​ 在线程从本地内存向主内存写回数据时，如果线程的期望值与主内存的真实值相等，则执行更新（交换）操作；如果线程的期望值与主内存的真实值不相等，不执行更新操作。

在 CAS 中，有这样三个值：

  • V：要更新的变量(var)
  • E：期望值(expected)
  • N：新值(new)

⽐较并交换的过程如下：

  • 判断 V 是否等于 E，如果等于，将 V 的值设置为 N；如果不等，说明已经有其它线程更新了 V，则当前线程放弃更新，什么都不做。所以这⾥的预期值 E 本质上指的是“旧值”。

详细描述

  • CAS 不是真正的加锁，也就是说允许多个线程同时访问一个变量。在 JMM 内存模型中我们已经了解到，线程在工作时不会直接操作主内存中的数据，而是会将主内存中的数据拷贝回线程本地内存中进行操作，操作完成之后再将数据写回主内存中。
  • 但是在多线程环境下，这一过程是不满足原子性的。根据 JMM 模型，每个线程操作主内存中的变量都需要经历 从主内存中复制到本地内存 -->> 本地内存中对变量进行运算 -->> 将运算结果写回主内存中 三个步骤。原子性就是这三个步骤读、更新、写回是一体的、不可分割的，要么全部执行成功，要么全部执行失败。如果原子性不能得到保证，在多个线程同时操作一份本地变量 X 时，就会出现 T1 线程在经过读、更新后被挂起，然后 T2 线程完整的执行了 读、更新、写回 三个步骤，修改了主内存中的变量 X 为 Y。这时候 T1 线程继续执行，这时候写回主内存的数据是错误的，因为它没有基于主内存中最新的值来进行计算。
  • CAS 的思想就是要求每个线程在拷贝主内存中的数据时在本地内存中保存一份主内存数据的快照，在本地内存中执行完运算任务要写回主内存时，首先会拿着这个快照与当前时刻主内存中数据进行对比，如果快照数据与主内存当前数据一致，则校验成功，再将线程本地内存中的数据写回主内存中；反之，如果校验失败，则不会将本地内存数据写回主内存。而是让线程重新获取主内存中的最新数据，重新运算，重新判断是否可以写回主内存中，直到 CAS 校验成功，将本地内存数据写回主内存中为止。
  • 线程的期望值
  • 主内存中的真实值
  • 在 CAS 的具体操作中，我们习惯将快照值称为线程的期望值，因为多线程中每个线程肯定期望自己在本地内存中运算的这段时间内没有其它线程去更新主内存中的值，这样就不会产生线程不一致。
  • 但实际上在多线程环境下，由于多个线程操作同一变量，且每个线程执行的时间长短不一致，时间较短的线程肯定会先于时间较长的线程修改主内存中的共享变量，当长时间的线程执行完线程任务准备将更新后的数据写回主内存中时，这一时刻主内存中的共享变量可能发生了改变，也可能没发生改变，这个值就叫做主内存中的真实值。
  • CAS 就是当主内存中的真实值和线程的期望值相同时，才执行交换（即将线程本地内存中值写回主内存）动作；如果真实值和期望值不相同，则拒绝执行交换操作。
  • 这优点类似于 Git 或 Svn 中代码的提交，多个人协同办公就相当于多线程，假设最初的时候每个人都拿到了 1.0 版本的代码，之后在这个版本上进行迭代更新，小A每工作一小时就提交一次代码，而小B则是每天下班之前才提交一次代码。当第一天工作结束后，小B直接提交代码时就会报错，因为他没有兼容最新版的代码，只有在拉取最新的代码、并检查与自己的代码无冲突时，它才可以进行提交。

3 CAS 底层原理

CAS 底层原理：自旋锁 + UnSafe类

CAS 保证原子性

  1. 如果有⼀个多个线程共享的变量 i 原本等于5，我现在在线程 A 中，想把它设置为新的值6；
  2. 我们使⽤ CAS 来做这个事情；
  3. ⾸先我们⽤ i 去与 5 对⽐，发现它等于 5 ，说明没有被其它线程改过，那我就把它设置为新的值6，此次CAS成功，i 的值被设置成了6；
  4. 如果不等于5，说明 i 被其它线程改过了（⽐如现在 i 的值为2），那么我就什么也不做，此次 CAS 失败，i 的值仍然为 2。

在这个例⼦中， i 就是 V，5 就是 E，6 就是 N。那有没有可能我在判断了 i 为 5 之后，正准备更新它的新值的时候，被其它线程更改了 i 的值呢？

  • 这是不会发生的，因为 CAS 是⼀种原⼦操作，它是⼀种系统原语，是⼀条 CPU 的原⼦指令，从CPU层⾯保证它的原⼦性。
  • 当多个线程同时使⽤ CAS 操作⼀个变量时，只有⼀个会胜出，并成功更新，其余均会失败，但失败的线程并不会被挂起，仅是被告知失败，并且允许再次尝试，当然也允许失败的线程放弃操作。

  • 原语是由若干条指令组成的、用于完成某个功能的一组指令集，这组指令集的执行顺序是确定的，且在执行过程中不允许被中断，具有原子性。可以直接将原语视为一条大的、执行时间较长的指令。
  • 系统原语就是操作系统把一些常用的原语直接实现在硬件上，以便遇到时方便调用。
  • CAS（Compare-And-Swap）是一条 CPU 并发原语。它的功能是判断内存中某个位置的值是否为预期值，如果是则更改为新的值，这个过程是原子的。

    4. 这时线程 A 恢复，执行 compareAndSwapInt 方法比较，发现自己手里的值 3 和主内存中的值 4 不一致（因为变量 value 被 volatile 修饰，所以其它线程对它的修改，线程 A 总是能够看到），说明该值已经被其它线程抢先一步修改过了，那 A 线程本次修改失败，只能再重新从主内存中读取最新的值重新计算一遍了.
    5. 线程 A 重新获取 value 值，线程A继续执行 compareAndSwapInt 进行比较替换，直到成功。

一句话总结

  • atomicInteger#getAndIncrement 方法就是 20000 个线程执行 num++，但是每次执行完要写会主内存时要先经过 CAS 校验，校验成功了就写回去，校验不成功则重写读取主内存中的值，重新执行 num++，重写写回主内存，反复执行，直到成功为止。

4 CAS 缺点

  1. 循环时间长开销大

​ 如果 CAS 失败（数据从线程本地内存写回主内存时一直校验不成功），就会一直进行尝试（自旋）。如果 CAS 长时间一直不成功，可能会给 CPU 带来很大的开销。

  1. 只能保证一个共享变量的原子操作（源码中就只有一个）

​ 当对一个共享变量执行操作时，可以使用 CAS 循环的方式来保证变量的原子性。但是，对于多个共享变量进行操作时，CAS 循环就无法保证操作的原子性，这个时候只能用锁来保证原子性。

  1. CAS 会导致 ABA 问题

​ CAS 算法只关注头和尾，只要头和尾是正确就判断为真，但是中间数据可能被改过多次。正是因为这样，才会出现由于时间差引起的 ABA 问题。

5 ABA 问题

5.1 是什么？

“狸猫换太子”

ABA 问题是什么？

  • ABA 问题是 CAS 算法的遗留问题。
  • CAS 算法的一个重要前提是需要取出主内存中某时刻的数据并在当前时刻比较并替换，由于当前时刻与某一时刻存在时间差，这个时间差就会导致数据的变化。
  • 比如说一个线程 T1 从主内存位置 V 取出 A，这时候另一个线程 T2 也从主内存中取出 A，并且线程 T2 进行了一些操作将值变成了 B，然后线程 T2 又将 V 位置的数据变成 A，这时候线程 T1 再进行 CAS 操作时发现内存中仍然是 A，然后线程 T1 操作成功。尽管线程 T1 的 CAS 操作成功，但不代表这个过程是没有问题的。

5.3 如何解决 ABA 问题？

时间戳的原子引用

  • CAS 算法中，每次线程从本地内存将数据写回主内存中时，只比较线程的期望值与主内存中的当前值是否相同，而不关心主内存中的当前值是否与线程的期望值是不是同一个值，所以会存在 ABA 问题。
  • 但是如果我们给每个值都加一个版本号，在线程从本地内存将数据写回主内存时除了比较数据值还要额外增加版本号的比较，就可以解决 ABA 问题了。

我们来看一个简单的加了时间戳原子引用的例子：

0s 2s~4s 4s~6s 10s
T1 100（1） 100（1） -> 101（2） 101（2） 101（2）->100（3） 100（3）
T2 100（1） 100（1） -> 990（2） 100(1)？100（3）
  1. 线程 T1、T2 并发执行，都得到了主内存中变量 100（1） 的副本拷贝，并且在拷贝数据时同时将版本号也拷贝到本地内存中。
  2. 由于 T2 执行时间较长（假设为 10s），T1 执行时间较短（假设每次为 2s），在 T1 执行的时间段内，T2 已经完成了：
    • 将 100（1）修改为101（2），更新版本号，写回主内存中；
    • 再从主内存中将 101（2）读取到本地内存中；
    • 将 101（2）修改为 100（3），更新版本号，写回主内存中。
  3. 又过了 2s ，T2 线程终于执行完毕了，将100（1）更新为 999（2），准备将 999（2）写回主内存中。由于当前数据是加了时间戳原子引用，所以写回主内存前需要经过 CAS 验证（包含时间戳的 CAS 验证），即比较 100（1）和 100（3），发现这时候虽然这两个变量值相等，但是版本号却不同，也即期望版本号与主内存中当前数据的版本号不同，所以验证不能通过，即写回主内存的操作失败，这个线程只能再重新读取主内存中的最新值重新在本地内存中计算了。这就可以很好的解决 ABA 问题。

现代计算机内存模型

  • 早期计算机中 CPU 和内存的速度是差不多的，CPU 执行指令的速度与内存读写数据的速度基本可以达到一种平衡。
  • 但是在现代计算机中，CPU 执行指令的速度要远高于内存的存取速度，当 CPU 运算结束后，只能空闲等待，直到再次读取到内存中的数据，这是非常浪费新能的。
  • 所以现代计算机系统一般都会加入一层读写速度尽可能接近 CPU 运算速度的高速缓存（Cache）来作为内存和处理器之间的缓冲。将运算需要使用到的数据复制到缓存中，让运算能快速进行，当运算结束后再从缓存同步回内存之中，这样处理器就无需等待缓慢的内存读写了。

为什么不把数据直接存在 CPU 中呢？这样读写速度岂不是更快了？

  • 现代计算机中，硬件性能的速度从高到低为：CPU > Cache > 内存 > 硬盘。而在同等价值下，内存大小则是刚好反过来的，也就是说，运算速度越快，可存储的数据量就越小。所以说使用 CPU 来存储程序运行时需要使用到的所有数据是不现实的，在计算机系统中，CPU 主要用于数据计算，其它硬件则用于数据存取。
  • 为了防止 CPU 被阻塞。如果直接将数据放在 CPU 中，当运算阻塞时整个 CPU 就废了，而放在高速缓存中，如果某个运算发生阻塞超过一定时间后，CPU 可以放弃该运算转去执行其它缓存中的运算。

磁盘中的数据什么时候加载到高速缓存（Cache）中呢？

  1. 程序开始运行的时候，需要把运行时需要使用到的数据加载到内存中。在这个阶段就可以把一些高频的、可能会经常用到的数据（如加减乘除等运算符）直接加载到高速缓存中。
  2. 程序在运行过程中，如果发现有一部分数据是被反复使用的（如通过阈值来筛选高频出现的数据），就可以把这一部分数据在运行期再加载到高速缓存中。
  3. 高速缓存中放的只是主内存中的一部分高频使用的数据，并不能把全部数据都复制到高速缓存中，但即使这样，也大大增加了 CPU的利用率。

缓存一致性问题

​ 绝大多数的运算任务都不可能只靠处理器“计算”就能完成，处理器至少要与内存交互，如读取内存数据、存储运算结果等，这个 I/O 是很难消除的。即程序运行时 CPU 需要经历 从内存取数据 --> CPU 运算 --> 将运算结果写回内存，由于 CPU 的运算速度远快于内存的读写速度，就会出现这样一种情况，内存花了 10s 才将要运算的数据传递到 CPU 中，但是 CPU 只用了 1s 就计算完了，那么中间的 9s CPU 就一直没有活干。写出到内存中也存在类似的情况。基于高速缓存的存储很好地解决了处理器与内存的速度矛盾，但是引入了一个新的问题：缓存一致性。

​ 在多处理器系统中，每个处理器都有自己的高速缓存，而它们又共享同一主内存（物理意义上的内存，可以看做是内存条）（比如 3 个 CPU，一根内存条）。当多个处理器的运算任务都涉及同一块主内存区域时，将可能导致各自的缓存数据不一致。举例说明就是某个变量在多个 CPU 之间共享，并且这些 CPU 都对这个变量进行了运算，那么当缓存数据同步回主内存时以谁的缓存数据为主呢？

​ 为了解决缓存一致性的问题，需要各个处理器访问缓存时都要遵循一些协议，在读写的时候要根据协议来进行操作。

JMM 概述

JMM；Java Memory Model；Java内存模型

什么是 Java 内存模型？

​ Java 内存模型就是按照功能将内存划分为几个不同的区域，这些区域在内存中不是真实存在的，而是为了方便理解将某个（或某些）真实的内存区域抽象成了概念上不同的区域。

​ 其实 JVM 也是一种内存模型，因为真实的物理内存空间并未划分什么栈空间、堆空间、方法区等，只不过 JVM 在实现时利用一部分物理空间存储具有相似性的数据，所以才把内存抽象成一块一块不同的区域。

​ 从抽象的角度看，JMM 定义了线程和主内存之间的抽象关系：线程之间共享的变量存储在主内存（Main Memory）中，每个线程都有一个私有的本地内存（Local Memory），本地内存中存储了该线程读/写共享变量的副本。

​ 本地内存是 JMM 的一个抽象概念，并不真实存在，它由缓存、写缓冲区、寄存器以及其它的编译期优化共同实现的。它描述的是一组规则或规范，通过这组规范定义了程序中各个变量（包括实例字段、静态字段和构成数组对象的元素）的访问方式。

运行时内存的划分

  • 对于每一个线程来说，栈都是私有的，而堆是共有的。
  • 也就是说在栈中的变量（局部变量、⽅法定义参数、异常处理器参数）不会在线程之间共享，也就不会有内存可见性的问题，也不受内存模型的影响。
  • 而在堆中的变量是共享的，称为共享变量。所以，内存可见性是针对的共享变量。

既然堆是共享的，为什么在堆中会有内存不可见问题？

  • 这是因为现代计算机为了高效，往往会在高速缓存区中缓存共享变量，因为 CPU 访问缓存区比访问内存要快得多。
  • 线程之间的共享变量存在主内存中，每个线程都有一个私有的本地内存，存储了该线程读、写共享变量的副本。
  • 本地内存是 Java 内存模型的一个抽象概念，并不真实存在。它涵盖了缓存、写缓冲区、寄存器等。
  • Java 线程之间的通信由 Java 内存模型（简称JMM）控制，从抽象的角度来说，JMM 定义了线程和主内存之间的抽象关系。

JMM与Java内存区域划分的区别与联系

上面两小节分别提到了JMM 和 Java 运行时内存区域的划分，这两者既有差别又有联系：

区别

两者是不同的概念层次。JMM是抽象的，他是⽤来描述一组规则，通过这个规则来控制各个变量的访问方式，围绕原子性、有序性、可见性等展开的。而 Java 运行时内存的划分是具体的，是 JVM 运行 Java 程序时，必要的内存划分。

联系

  • 都存在私有数据区域和共享数据区域。⼀般来说，JMM中的主内存属于共享数据区域，他是包含了堆和方法区；
  • 同样，JMM 中的本地内存属于私有数据区域，包含了程序计数器、本地方法栈、虚拟机栈。
  • 实际上，他们表达的是同⼀种含义，这⾥不做区分。

JMM 工作原理

​ 上面我们讲了：JMM 定义了线程与主内存之间的抽象关系。

​ 在一个运行的 Java 程序中，JVM 是一个进程，运行时包含多个线程，而每个线程创建时 JVM 都会为其创建一个工作内存，工作内存是每个线程私有的数据区域，而 JMM 中规定所有变量都要存储在主内存，主内存是共享内存区域，所有线程都可以访问，但 线程对变量的操作（读取、赋值等）必须要在工作内存中进行，首先要将变量从主内存拷贝到自己的工作内存空间，然后对变量进行操作，操作完成后再将变量写回主内存，而不能直接操作主内存中的变量，各个线程中的工作内存中存储着主内存中的变量副本拷贝，因此不同的线程间无法访问对方的工作内存。若多个线程间想要通信，必须通过主内存来完成。其简要访问过程如下图：

从图中可以看出：

  1. 所有的共享变量都存在主内存中。
  2. 每个线程都保存了⼀份该线程使用到的共享变量的副本。
  3. 如果线程A与线程B之间要通信的话，必须经历下面2个步骤：
  • 线程A将本地内存A中更新过的共享变量刷新到主内存中去。
  • 线程B到主内存中去读取线程A之前已经更新过的共享变量。

所以，线程A无法直接访问线程B的工作内存，线程间通信必须经过主内存。
注意，根据JMM的规定，线程对共享变量的所有操作都必须在自己的本地内存中进行，不能直接从主内存中读取。

所以线程B并不是直接去主内存中读取共享变量的值，⽽是先在本地内存B中找到这个共享变量，发现这个共享变量已经被更新了，然后本地内存B去主内存中读取这个共享变量的新值，并拷贝到本地内存B中，最后线程B再读取本地内存B中的新值。

那么怎么知道这个共享变量的被其他线程更新了呢？这就是JMM的功劳了，也是 JMM 存在的必要性之一。JMM通过控制主内存与每个线程的本地内存之间的交互，来提供内存可见性保证。

  • Java中的 volatile 关键字可以保证多线程操作共享变量的可见性以及禁止指令重排，
  • synchronized 关键字不仅保证可见性，同时也保证了原子性（互斥性）。
  • 在更底层，JMM通过内存屏障来实现内存的可见性以及禁止指令重排。

JMM 三大特性

  1. 可见性
  2. 原子性
  3. 有序性

​ JMM 三大特性是大多数多线程开发需要遵守的规定。但是 volatile 只支持可见性、有序性（禁止指令重排），而不支持原子性。

可见性

  • 每个线程对主内存中共享变量的读写操作都是先将主内存中的共享变量拷贝到自己的工作内存进行运算，然后再写回到主内存中的。
  • 这就可能会造成：线程 A 修改了共享变量 X 的值但还未写回主内存时，另一个线程 B 又对主内存中同一个共享变量 X 进行操作，但此时 A 线程工作内存中的共享变量 X 对线程 B 来说并不可见。这种工作内存与主内存同步延迟现象造成了可见性问题。

原子性

什么是原子性？

  •      不可分割性，完整性，即某个线程正在做某个具体业务时，中间不可以被加塞或者被分隔，需要整体完整。

  •      要么同时成功，要么同时失败。有点类似于关系型数据库中的事务。

有序性

JMM 关于同步的规定

  1. 线程加锁前，必须把共享变量的值刷新回主内存；
  2. 线程加锁前，必须读取主内存的最新值到自己的工作内存；
  3. 加锁解锁是同一把锁。

ThreadLocal

使用 ThreadLocal 进行线程本地存储

  • ThreadLocal 是 Java 中提供的线程本地存储机制，可以利用该机制将数据缓存在某个线程内部，该线程可以在任意时刻、任意方法中获取缓存中的数据。

线程池中的 ThreadLocal 存在的内存泄露问题

  • 如果在线程池中使用 ThreadLocal 会造成内存泄露，因为正常情况下，线程死亡后，需要把线程对象使用过的资源全部释放。即 ThreadLocal 使用完之后，应该要把使用过的 Entry<K, V> 对象进行回收。
  • 但是由于线程池中的部分线程是不会被回收的，而线程池中的线程对象通过强引用指向了 ThreadLocalMap，ThreadLocalMap 又通过强引用指向 Entry 对象。所以如果线程对象不被回收的话，Entry 对象也就不会被回收。
  • 如果反复使用线程池中线程对象的 ThreadLocalMap 缓存数据，就会导致内存泄露。
  • 解决方法是：在使用完 ThreadLocal 对象后，手动调用 ThreadLocal 中的 remove() 方法，手动清除 Entry 对象。
  • ThreadLocal 经典的应用场景就是连接管理（一个线程持有一个连接，该连接对象可以在不同的方法之间进行传递，线程之间共享一个连接）。

写时复制技术

  • 顾名思义，“写时复制”就是写的时候才复制，也就是说“读的时候不复制”。
  • 我们都知道，多个线程读同一份资源不会有线程安全的问题，只有多个线程对同一份资源进行写操作的时候才可能出现线程安全问题。
  • 一般的加锁机制（重锁）不会区分读还是写，只要是某一资源被上锁了，不管是读还是写都需要进行资源的争夺。
  • 而如果某份资源在大多数情况下都是多线程读，只有少数情况下才会进行多线程写的话，使用不区分读写的锁显然对性能的浪费很大。而如果将读和写的操作分开，在某些情况下反而能得到更好的性能。写时复制技术就是这样一种技术。
  • 写时复制技术用到了读写分离的思想。
  • 读的时候共享，写的时候复制一份数据的副本对副本进行操作，写好以后，用副本覆盖原件。

放在本例中就是，线程 A、B、C 同时执行 list.add() 操作，向集合中添加元素。

  1. 如果线程 A、B、C 同时要读 list 中的数据，可以任意读，无锁。
  2. 如果线程 B 要修改 list 中的数据，如调用 add() 方法添加元素。不好意思，不能直接在原 list 集合中添加，线程 B 需要将原来的 list 集合的内容复制一份，将要添加的数据加入到 list 副本中。如果添加成功了，再将指向原 list 的指针指向新的 list，这就完成了一次集合中元素的添加。
