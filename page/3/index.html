MapReduce概述

MapReduce定义

MapReduce 是一个分布式运算程序的编程框架，是用户开发“基于Hadoop的数据分析应用”的核心框架。

MapReduce核心功能是将用户编写的业务逻辑代码和自带默认组件整合成一个完整的分布式运算程序，并发运行在一个Hadoop集群上。

MapReduce优缺点

优点——简单

缺点——慢

如何对数据进行切片？

Shuffle

shuffle：洗牌

从 Mapper 出去到 Reducer 接收之前全部都是 Shuffle 的范围。

Shuffle 阶段数据发生了关键的变化：

  1. 从 Mapper 出去的 <K，V> 值是无序的，而到了 Reducer 中数据则是有分组的。 也就是说，在 Shuffle 阶段会将数据整理、分组。
  2. Mapper 出来的数据是无序的，在 Shuffle 阶段首先对数据进行排序，排完序之后，相等的数据就挨在一起了，再进行分组就相对容易了。
  3. 排序方法：快排 + 归并
    • 对于快速排序来说，需要将所有的数据全部加载进内存中然后再进行比较；但是对于大数据而言，显然是不合理的。所以会先将大数据的切分，对每一块数据进行快排。
    • 然后再将多个有序块再进行归并排序。
  4. 数据会被分片（逻辑上），每一个片对应一个 MapTask 。每个 MapTask 都会进行排序，最终形成一个归并排序后的有序数据块，这些数据块最终又会汇总到一个 Reducer 中。Reducer 在拿到多个有序文件后，再进行一次归并让多个有序文件变成一个有序文件，最终输入到 Reducer 中。
  5. 这就是 Shuffle 阶段如何把数据进行排序、分组的过程。
  • 三次排序：
    1. 快排
    2. 归并
    3. 再次归并

大数据

大数据（Big Data）：

大数据指无法在一定时间范围内用常规软件工具进行捕捉、管理和处理的数据集合。是需要新处理模式才能具有更强的决策力、洞察发现力和流程优化能力的海量、高增长率和多样化的信息资产。

  • 多样化：传统的数据库如 MySQL 只能处理结构化的数据类型，而大数据不仅可以处理结构化的数据，还可以处理非结构化、半结构化的数据。

大数据技术/大数据框架：

主要解决，海量数据的存储和海量数据的分析计算问题。

大数据特点：

  1. Volume（大量）

  2. Velocity（高速）

  3. Variety（多样）

  4. Value（低价值密度）

大数据应用场景

Hadoop

Hadoop 设计的初衷是采用大量廉价的机器，组成一个集群！完成大数据的存储和计算！[通过数量的堆叠来提高算力]

Hadoop 是什么

  • Hadoop 是一个由 Apache 基金会所开发的分布式系统基础架构。
  • 主要解决，海量数据的存储和海量数据的分析计算问题。
  • 狭义上来说，单指 Hadoop 技术。
  • 广义上来说，Hadoop 通常是指一个更广泛的概念——Hadoop 生态圈

Hadoop 的优点（4高）

  1. 高可靠性

Hadoop 底层维护多个数据副本，所以即使 Hadoop 某个计算元素或存储出现故障（即使 Hadoop 集群中的某个数据副本所在的节点出现故障），也不会导致数据的丢失。

而且 Hadoop 会实时动态的监测集群中的副本，如果集群中的某个节点发生故障导致副本数小于集群预设的副本数，那么 Hadoop 就会自动创建一个新的副本，维护副本数量等于预设的副本数量。

  1. 高扩展性

在集群间分配任务数据，可方便的扩展数以千计的节点。

在处理数据时如果发现集群中的节点不够使用了，可以很方便的增加一些节点扩展整个 Hadoop 集群。

  1. 高效性

在 MapReduce 的思想下，Hadoop 是并行工作的，以加快任务处理速度。

  1. 高容错性

能够自动将失败的任务重新分配。

Hadoop 在 Map 阶段会将大的任务分成若干个小的任务进行运算，如果在运算过程中某个小的任务因某些不可抗力因素发生失败，那么 Hadoop 就会重新分配一个线程来运行未成功的任务，直至成功。

Hadoop 中的组件

  • 在 Hadoop1.x 时代，Hadoop 中的 MapReduce 同时处理业务逻辑运算和资源的调度，耦合性较大；
  • 在Hadoop2.x时代，增加了 Yarn 。Yarn 只负责资源的调度，MapReduce 只负责运算。

Hadoop 1.0

  • HDFS：负责大数据的存储
  • Common：HDFS 和 MR 共有的常用的工具包模块。
  • MapReduce：
    1. 负责计算。
    2. 负责计算资源的申请和调度。

Hadoop 2.0

  • HDFS（框架）：负责大数据的存储
  • YARN（框架）：负责大数据的资源调度。[只有在计算时才会用到 YRAN ，因为在计算时需要申请很多资源，由 YARN 统一分配]
  • Common：HDFS 和 MR 共有的常用的工具包模块。
  • MapReduce（编程模型）：使用 Hadoop 制定的编程规范要求，来编写程序，完成大数据的计算。

Hadoop 1.x 和 Hadoop 2.x 的区别

在 1.x 版本中，MR 既负责运行 MR 程序还负责为 MR 程序申请资源！Hadoop 只能为自身的 MR 程序提供服务！

在 2.x 版本中，MR 只负责 MR 程序的计算，资源的调度和管理由 YARN 负责！Hadoop 集群不仅能为自身的 MR 程序提供服务，还可以为第三方计算引擎，例如 TeZ，Spark，Flink 等提供计算资源的调度服务！

HDFS 架构概述

HDFS（Hadoop Distribute File System）：负责大数据的存储。

元数据：文件的属性信息。如文件名、文件目录结构、文件属性（生成时间、副本数、文件权限），以及每个文件的块列表和块所在的DataNode等。

在本地文件系统中存储文件块数据，以及块数据的校验和。

  • 职责：
    1. 负责接收 NM 分配的任务。
    2. 负责数据块（block）的管理（读、写）
  1. Secondary NameNode（2nn）：负责辅助 NameNode 工作！

每隔一段时间对NameNode元数据备份。

MapReduce 架构概述

  • MapReduce 是一种编程规范，要求程序中有 Mapper（简单处理） 和 Reducer（合并）。
  • 遵循 MapReduce 编程规范编写的程序，打包后，称为一个 Job（任务）。
  • Job 需要提交到 YARN 上，向 YARN 申请计算资源，运行 Job 中的 Task（进程）！
  • Job 会先创建一个进程 MrAppMaster（MapReduce 应用管理者）， 由 MrAppMaster 向 YARN 申请资源！
  • MrAppMaster 还负责监控 Job 中各个 Task 运行情况，进行容错管理！

MapReduce将计算过程分为两个阶段：Map和Reduce

  • Map阶段并行处理输入数据
  • Reduce阶段对Map结果进行汇总

Yarn 架构概述

Yarn 负责集群中所有计算资源的管理和调度。

核心进程

  1. ResourceManager（1个）：负责整个集群所有资源的管理！
  • 职责
    1. 负责接收客户端提交的 Job 请求
    2. 负责向 NM 分配任务。
    3. 负责接收 NM 上报的信息。
  1. NodeManager（N个）：负责单台计算机所有资源的管理 ！
  • 职责：
    1. 负责和 RM 进行通信，上报本机中的可用资源
    2. 负责领取 RM 分配的任务！
    3. 负责为 Job 中的每个 Task 分配计算资源！

概念：

Container（容器）：

比如：NodeManager 为 Job 的某个 Task 分配了 2 个 CPU 和 2G 内存的计算资源！

为了防止当前 Task 在使用这些资源期间，被其他的 Task 抢占资源。 Hadoop 框架就会将这些计算资源，封装到一个 Container 中，在 Container 中的资源，会被暂时隔离，无法被其它进程所抢占。

当前 Task 运行结束后，当前 Container 中的资源就会被释放！允许其它 task 来 使用！

  1. ResourceManager（RM）
  1. 处理客户端请求
  2. 监控NodeManager
  3. 启动或监控ApplicationMaster
  4. 资源的分配与调度
  1. NodeManager（NM）
  1. 管理单个节点上的资源
  2. 处理来自ResourceManager的命令
  3. 处理来自ApplicationMaster的命令
  1. ApplicationMaster（AM）
  1. 负责数据的切分
  2. 为应用程序申请资源并分配给内部的任务
  3. 任务的监控与容错
  1. Container

Container 是YARN中的资源抽象，它封装了某个节点上的多维度资源，如内存、CPU、磁盘、网络等。

HDFS 概述

产生背景

随着数据量越来越大，在一个机器上存不下所有的数据，就需要把数据分配到更多的机器中，但是这样又不方便管理和维护，所以我们迫切需要一种可以通过一台机器来管理多台机器上的文件的系统，这就是分布式文件管理系统。

HDFS 定义

  • HDFS（Hadoop Distribute FileSystem）：Hadoop 的分布式文件系统，本质上还是一个文件系统。
  • 文件系统即用来存储管理计算机中文件的系统。
  • HDFS 通过目录树来定位文件；其次，它是分布式的，由很多服务器联合起来实现其功能，集群中的服务器有各自的角色。
  • HDFS 只是分布式文件管理系统中的一种。

HDFS 的优缺点

HDFS 的使用场景

适合一次写入，多次读出的场景，且不支持文件的修改。适合用来做数据分析。

优点

  1. 高容错性

    1）数据自动保存多个副本。它通过增加副本的形式，提高容错性。【副本数量由客户端指定】

    2）某一个副本丢失以后 ，它可以自动恢复。

  2. 适合处理大数据

    1）数据规模：能够处理数据规模达到GB、TB、甚至 PB 级别的数据；

    2）文件规模：能够处理百万规模以上的文件数量，数量相当之大。

  3. 可构建在廉价机器上，通过多副本机制，提高可靠性。

缺点

  1. HDFS 不支持对文件的随机写

也就是说：仅支持数据 append（追加），不支持文件的随机修改。

原因：文件在 HDFS 上存储时，以 block 为基本单位存储！也就是说：

（1）没有提供对文件的在线寻址/在线打开功能！

（2）文件以块形式存储，如果修改了一个块中的内容，就会影响当前块之后的所有块，效率很低。

  1. 不适合低延时数据的访问。

意思就是：如果我们要求能尽可能快的读取数据（即数据访问时延较低），使用 HDFS 是不合适的。比如毫秒级的存储数据，是做不到的。一般处理很小量的数据时，要求速度要快，HDFS就不适合处理这些数据。

  1. 不适合存储大量的小文件。

原因：（1）如果 HDFS 存储了大量的小文件，就会降低 NN 的服务能力！

​ （2）小文件存储的寻址时间会超过读取时间，它违反了 HDFS 的设计目标。

  1. 不支持并发写入、文件随机修改

同一文件在同一时刻只能由一个线程写入，不允许多个线程同时写；。

假设 线程A 正在上传ss.txt文件，那么在线程A 上传开始到结束之间，其它所有的线程不能再上传ss.txt文件

HDFS 组成架构

  1. NameNode（nn）：就是Master，它是一个主管、管理者。

    1）管理HDFS的名称空间；

    2）配置副本策略；

    3）管理数据块（Block）映射信息；

    4）处理客户端读写请求

  2. DataNode（dn）：就是Slave。NameNode下达命令，DataNode执行实际的操作

    1）存储实际的数据块；

    2）执行数据块的读/写操作。

  3. Client：就是客户端。

    1）文件切分。文件上传HDFS的时候，Client将文件切分成一个一个的Block，然后进行上传；

    2）与NameNode交互，获取文件的位置信息；

    3）与DataNode交互，读取或者写入数据；

    4）Client提供一些命令来管理HDFS，比如NameNode格式化；

    5）Client可以通过一些命令来访问HDFS，比如对HDFS增删查改操作；

  4. Secondary NameNode（2nn）：并非NameNode的热备。当 NameNode 挂掉的时候，它并不能马上替换NameNode并提供服务。1）辅助NameNode，分担其工作量，比如定期合并 Fsimage 和 Edits ，并推送给 NameNode；

    2）在紧急情况下，可辅助恢复NameNode。

HDFS 块大小

  • HDFS 中的文件在物理上是分块存储（Block），块的大小由hdfs-default.xml配置文件中的参数 dfs.blocksize 来决定。

  • 在 Hadoop 2.x 中，默认值是134217728（128M）；

    在 Hadoop 1.x 中，默认值是 64M 。

  1. Hadoop 2.x 块大小默认为 128M 的原因：基于最佳传输损耗理论！

最佳传输损耗理论：在一次传输中，寻址时间占总传输时间1%时，本次传输的损耗最小，为最佳性价比传输！

Lambda表达式

所谓的 Lambda 表达式，就是对接口中抽象方法的实现。

Lambda是一个匿名函数，我们可以把Lambda表达式理解为一段可以传递的代码（将代码像数据一样进行传递）。可以写出更简洁、更灵活的代码。

Lambda表达式，也可以称为闭包，是Java8的新特性。

Lambda允许把函数作为一个方法的参数（函数作为参数传递进方法中）。

Lambda 表达式语法

格式：

Java8 中引入了一个新的操作符->，它将 Lambda 分为了两个部分：

  • 左侧：Lambda 表达式的参数列表；对应接口中抽象方法的参数列表；
  • 右侧：Lambda 表达式的方法体；对应接口中抽象方法的实现类的方法体。

Lambda 表达式对接口的要求

  • 需要函数式接口的支持。即@FunctionalInterface接口

  • 因为函数式接口中有且仅有一个抽象方法，所以使用 Lambda 表达式实现的就是该抽象方法。

Lambda表达式的特征：

  • 可选的类型声明：不需要声明参数类型，因为JVM编译器可通过上下文推断出数据类型。
  • 可选的参数圆括号：一个参数无需定义圆括号，但多个参数需要定义圆括号
  • 可选的大括号：如果主体只包含了一条语句，就不需要使用大括号。
  • 可选的返回关键字：如果主体只有一条语句且为返回值语句，则可以省略return关键字和大括号（return关键字和大括号要么全都省略，要么全都不省略）。

Lambda 语法优化

  1. 简化参数类型：可以不写参数类型，但是必须所有参数都不写参数类型
  2. 简化参数小括号，如果只有一个参数则可以省略参数小括号。
    • 有多个参数不能省略小括号；
    • 没有参数也不能省略小括号；
  3. 简化方法体大括号，如果方法体中只有一条语句，则可以省略方法体大括号
    • 如果方法体中唯一的一条语句是 return 语句，则进行省略时需要把方法体大括号和return关键字一起省略。

Lambda 表达式只是实现了方法，但是并没有调用方法。

Stream API

概念

Stream：流

Stream 是 Java8 中处理集合的关键抽象概念，它可以指定你希望对集合进行的操作，可以执行非常复杂的查找、过滤和映射数据等操作。使用 Stream API 对集合数据进行操作，就类似于使用 SQL 执行数据库查询。 也可以使用 Stream API 来并行执行操作。 简而言之，Stream API 提供了一种高效且易于 使用的处理数据的方式。

Stream 用于操作数据源（集合、数组等）所生成的元素序列。

包：java.util.stream.*

Tip：

  • Stream 自己不会存储元素。
  • Stream 不会改变源对象。相反，它们会返回一个持有结果的新的 Stream 对象。
  • Stream 操作是延迟 执行的。这意味着它们会等到需要结果的时候才执行。

Stream操作的三个步骤

  1. 创建Stream对象
    • 一个数据源（如：集合、数组）
    • 获取一个流对象
  2. 中间操作
    • 一个中间操作链，对包含数据源数据的流对象进行处理
  3. 终止操作（终端操作）
    • 一个终止操作 ，执行中间操作链，并产生结果。

创建Stream对象

创建Stream对象的4种方法

  1. Java8 中的 Collection 接口被扩展，提供了两个获取流的方法。
  2. 通过 Arrays 中的的静态方法 stream() 可以获取数组流。
  3. 通过 Stream 类中的静态方法 of() 获取流对象
  4. 通过 Stream 类中的静态方法iterate或generate创建无限流。
    • 无限流就是如果不加限制 ，将会一直生成数据的流。
类 方法 说明
Collection default Stream<E> stream() 返回一个顺序流
- default Stream<E> parallelStream() 返回一个并行流
Arrays static <T> Stream<T> stream(T[] array) 返回一个流
Stream<T> public static<T> Stream<T> of(T... values) 返回一个流
Stream<T> public static<T> Stream<T> iterate(final T seed, final UnaryOperator<T> f) 迭代
- public static<T> Stream<T> generate(Supplier<T> s) 生成
  • 只创建Stream对象并不会输出任何结果，只有在进行了终止操作后才会输出结果。

多个中间操作可以连起来形成一个流水线，除非流水线上触发终止操作，否则中间操作不会执行任何的处理！而在终止操作时一次性全部处理，称为“惰性求值”。

  • findFirst()方法的作用是返回返回当前流中的第一个对象。

  • 而findAny()的作用则是返回当前流中的任意一个元素。

    • 如果与获取串行流的方法stream()配合使用，则每次获取的都是流中的第一个元素；

并行流与串行流

并行流就是把一个内容分成多个数据块，并用不同的线程分别处理每个数据块的流。

Java8 中将并行进行了优化，我们可以很容易的对数据进行并行操作。Stream API 可以声明性的通过parallel()与sequential()在并行流和串行流之间进行切换。
