斐波那契数列的多种解法

斐波那契数列

普通递归解法（性能最差）

递归优化解法

斐波那契数列递归的时候会造成大量的重复计算，比如就计算 fib(6) 为例来看：F(6) = F(5)+F(4)，当我们计算出 F(5) 的时候，F(4) 也已经被计算出来了，如果我们在 F(6) 的右子节点中能够调用已经计算出来的值的话，那么该节点就不用继续向下展开了。这在 n 非常大时，节省的效率很高，接近于一半。

我们看到上面相同颜色的都是重复计算，当 n 越大，重复的就会越多。

所以我们可以使用一个 map 把计算过的值存起来，每次计算的时候先查看 map 中有没有，

  • 如果有就表示计算过，直接从 map 中取；
  • 如果没有就先计算，计算完之后再把结果存到map中。

动态规划（非递归解法）

  • 想要计算 F(n) 的值，就需要知道 F(n-1) 和 F(n-2) 的值；
  • 想要计算 F(n-1) 的值，就要知道 F(n-2) 和 F(n-3) 的值；
  • 以此类推，一直到 F(0) 和 F(1)。

​ 正如上面讲的普通的递归算法的弊端一样，如果每次 F(n-1) 和 F(n-2) 都独立运算，那么重复计算的值就太多了。上面的优化方式是将已经计算过的值保存到一个 Map 中避免重复运算，而且采用的是自顶向下的计算方式，即从 F(n-1) 算到 F(1)。

​ 而所谓的斐波那契的动态规划解法，其实可以看做是一种穷举法。就是将从 F(1) 到 F(n-1) 的所有值都计算出来，保存到一个 List（或者 Map 均可）中，但是这里采用的是自底向上，即从 F(1) 计算到 F(n-1) 。

​ 所以，在计算 F(n) 之前，动态规划必定已经建立好了一个从 F(0) ~ F(n-1) 的数组。而结算结果就是从数组中取出对应的元素即可。

977. 有序数组的平方（双指针）

题目：977. 有序数组的平方

暴力解法

双指针 + 归并排序

思路

暴力解法没有利用【数组 nums 已经按照升序排列】这个条件。显然，

  • 如果数组 nums 中的所有数都是非负数，那么将每个数平方后，数组仍然保持升序；
  • 如果数组 nums 中的所有数都是负数，那么将每个数平方后，数组会保持降序。

这样一来，如果我们能够找到升序数组 nums 中负数与非负数的分界线，那么就可以用类似「归并排序」的方法了。

​ 具体地，我们设 neg 为数组 nums 中负数与非负数的分界线。也就是说，nums[0] ~ nums[neg] 均为负数，而 nums[neg+1] ~ nums[n−1] 均为非负数。当我们将数组 nums 中的数平方后，那么 nums[0] ~ nums[neg] 单调递减，nums[neg+1] ~ nums[n−1] 单调递增。

​ 由于我们得到了两个已经有序的子数组，因此就可以使用归并的方法进行排序了。具体地，使用两个指针分别指向位置 neg 和 neg+1，每次比较两个指针对应的数，选择较小的那个放入答案数组并移动指针。当某一指针移至边界时，将另一指针还未遍历到的数依次放入答案。

如果两个数组是有序的，则可以使用双指针的方法得到两个数组的交集。

  1. 首先对两个数组进行排序，然后使用两个指针遍历两个数组。

分治算法

分治算法介绍

​ 分治法是一种很重要的算法，顾名思义就是“分而治之”，即把一个复杂的问题分解成两个或更多相同或相似的子问题，再继续把子问题分成更小的子问题，直到最后子问题可以简单的直接求解，原问题的解即子问题的解的合并。

​ 分治思想是很多高效算法的基础，如排序算法（快速排序、归并排序），傅立叶变换（快速傅立叶变换）等。

分治算法的基本步骤

分治法在每一层递归上都有三个步骤：

  1. 分解：将原问题分解为若干个规模较小，相互独立，与原问题形式相同的子问题；
  2. 解决：若子问题规模较小而容易被解决则直接解，否则递归地解各个子问题；
  3. 合并：将各个子问题的解合并为原问题的解。

  • ADHOC(P) 是该分治法中的基本子算法，用于直接解决小规模的问题 P。因此，当 P 的规模不超过 n0 时直接用算法 ADHOC(P) 求解。
  • 算法 MERGE(y1,y2,…,yk) 是该分治法中的合并子算法，用于将 P 的子问题 P1 ,P2 ,…,Pk 的相应的解 y1,y2,…,yk 合并为 P 的解。

动态规划算法介绍

  1. 动态规划（Dynamic Programming）算法的核心思想是：将大问题划分为小问题进行解决，从而一步步获取最优解的处理算法。
  2. 动态规划算法与分治算法类似，其基本思想也是将待求解问题分解成若干个子问题，先求解子问题，然后从这些子问题的解得到原问题的解。
  3. 与分治法不同的是，适合于用动态规划求解的问题，经分解得到的子问题往往不是互相独立的。即下一个子问题的求解是建立在上一个子问题解的基础上得到的。
  4. 动态规划可以通过填表的方式来逐步推进，得到最优解.

MySQL 架构

1.1 MySQL逻辑架构

​ MySQL 可以在多种不同场景中应用并发挥良好作用。主要体现在存储引擎的架构上，插件式的存储引擎架构将查询处理和其它的系统任务以及数据的存储提取相分离。这种架构可以根据业务的需求和实际需要选择合适的存储引擎。其主要逻辑架构可分为四层，从上到下分别是：

  1. 连接层
  2. 服务层
  3. 引擎层
  4. 存储层

第三层的存储引擎主要负责数据存储和提取，存储引擎不会去解析 sql，不同存储引擎之间不会通讯，只会简单地响应上层服务器的请求。

1.1.1 连接层

连接层主要处理客户端与 MySQL 服务端的连接、授权认证、安全等问题。包含本地 socket 通信和大多数基于客户端/服务端工具实现的类似于 TCP/IP 的网络通信。主要完成一些类似于连接处理、授权认证、及相关的安全方案。在该层上引入了线程池的概念，为通过认证安全接入的客户端提供线程。同样在该层上可以实现基于 SSL 的安全链接。服务器也会为安全接入的每个客户端验证它所具有的操作权限。

Management Serveices & Utilities：系统管理控制工具

​ 可以看做是类似于 Hadoop 中的 Zookeeper，或者是 SpringCloud 中的配置中心，主要作用是协调整管理个系统的各个模块。

SQL Interface：SQL 接口

​ 类似于 MVC 三层架构中的 Controller 层，只是负责接收请求和返回最终的结果，具体的业务逻辑交给更底层的服务去做。比如 select ... from ... 就会调用 SQL Interface。

Parser：解析器

SQL Interface 中的 SQL 命令传递到解析器的时候会被解析器验证和解析。主要功能有：

  • 验证：对接收的 SQL 语句进行语义和语法的分析，如果在分解构成中遇到错误，那么就说明这个 SQL 语句是不合理的，验证失败，不继续网下执行，直接抛出错误。
  • 解析：如果验证成功，则根据语义和语法，将 SQL 语句划分为不同的操作类型，然后进行针对性的转发到后续步骤。比如查询语句就转发到查询对应的服务，更新操作就转发到更新对应的服务。

Optimizer：查询优化器

SQL 语句在查询之前会使用查询优化器对查询进行优化。就是优化客户端请求的 SQL 语句，根据客户端请求的 SQL 语句，和数据库中的一些统计信息，在一系列算法的基础上进行分析，得出一个最优的策略，告诉后面的程序如何执行这个 SQL 语句。

查询优化器就类似于指令重排，对于开发人员来说，写一条 SQL 可能自我感觉良好，性能天下第一，但是对于底层机器来说，它有自己的优化标准，经过查询优化器优化后的 SQL 语句的执行顺序并不一定与开发人员设计的 SQL 执行顺序相同。

Cache和Buffer：查询缓存

如果查询缓存有命中的查询结果，查询语句就可以直接去查询缓存中取数据。这个缓存机制是由一系列小缓存组成的。比如表缓存，记录缓存，key 缓存，权限缓存等。

1.1.3 引擎层

​ 存储引擎真正的负责了 MySQL 中数据的存储和提取，服务器通过 API 与存储引擎进行通信。

​ 存储引擎接口模块可以说是 MySQL 数据库中最有特色的一点了。目前各种数据库产品中，基本上只有 MySQL 可以实现其底层数据存储引擎的插件式管理。这个模块实际上只是 一个抽象类，但正是因为它成功地将各种数据处理高度抽象化，才成就了今天 MySQL 可插拔存储引擎的特色。

​ 可插拔的存储引擎的实现，让 MySQL 在处理不同的业务逻辑时，可以根据实际应用场景选择效率尽可能高的存储引擎。这就是 MySQL 相较于其它同类数据库产品的最大优势。

  • 注意：存储引擎是基于表的，而不是数据库。

1.3 大致查询流程

MySQL 查询的流程大致是：

  1. MySQL 客户端通过协议与 MySQL 服务器建连接；
  2. 发送查询语句，先查询缓存，如果命中，直接返回结果；
  3. 否则进行语句解析，也就是说，在解析查询之前，服务器会先访问查询缓存（Query Cache）——它存储了 SQL 语句以及
    相应的查询结果集。
  4. 如果某个查询结果已经位于缓存中，服务器就不会再对 SQL 语句进行解析、优化、以及执行。它仅仅将缓存中的结果返回给用户即可，这将大大提高系统的性能。

索引

是什么

官方定义：

索引（Index）是帮助 MySQL 高效获取数据的数据结构。

索引的目的在于提高查询效率。

​ 可以类比为字典。如果想要查询 “mysql” 这个单词，首先肯定要定位到 m 字母，然后在 m 字母的范围内继续找第 2 个 y 字母，以此类型，找到剩下的三个字母。

​ 如果没有索引，那么每找一个字母都需要全部查找一遍。

索引可以理解为“排好序的快速查找的数据结构”！！！

索引会影响到两大功能的执行效率：

  1. 排序：ORDER BY
  2. 查找：WHERE；HAVING

​ 在数据之外，数据库系统还维护着满足特定查找算法的数据结构，这些数据结构以某种方式指向数据（记录了真实真实数据的地址），在这些数据结构上我们可以实现高效的查找算法。这种数据结构，就是索引。

下图就是一种可能的索引方式示例：

  • 左边是索引表，一共有两列七条记录，最左边记录的是真实数据的物理地址，Col1 是主键字段，Col2 是索引字段。
  • 为了加快 Col2 的查找，可以维护一个右边所示的二叉查找树，每个节点分别包含索引键值和一个指向对应数据记录物理地址的指针，这样就可以运用二叉查找在一定的复杂度内获取到相应数据，从而快速的检索出符合条件的记录。
  • 一般来说索引本身也很大，不可能全部存储在内存中，因此索引往往以索引文件的形式存储的磁盘上。

优缺点

优势：

  • 索引能提高数据检索速率，对于一些经常被用于查询的字段，如果将其设置为索引，能大大地提高查询的速率。将频繁使用的字段就将其设置为索引，比不设索引时查询速度要快。

  • 通过索引列对数据进行排序，降低数据排序的成本。

劣势：

  • 实际上索引也是一张表，该表保存了主键与索引字段，并指向实体表的记录。所以索引也是要占用空间的。
  • 虽然索引大大提高了查询速度，但同时却会降低更新表的速度。比如在对表进行 INSERT、UPDATE 和 DELETE 操作时，因为更新表时 MySQL 不仅要更新数据，还要更新索引表。
    • 对于 UPDATE 操作，记录变更后索引字段在索引表中的位置就可能发生变化，需要更新。
    • 对于 INSERT 操作，添加记录后需要将新记录的索引字段加到索引表中。同理 DELETE 需要删除索引表中的记录。

索引分类

  • 单值索引：一个索引只包含单个列，一个表可以有多个单列索引。
  • 唯一索引：索引列的值必须唯一（即用于建立索引的列必须符合 UNIQUE），但允许有空值。
  • 复合索引：即一个索引包含多个列。

  3. 查看表的所有索引
  • 主键自动就是一种唯一索引。

检索原理

【初始化介绍】

一颗 B 树，浅蓝色的块我们称之为一个磁盘块，可以看到每个磁盘块包含了数据项（深蓝色所示）和指针（黄色所示）。如磁盘块1 包含数据项 17 和35，包含指针 P1、P2、P3。

  • P1 指向数据项小于 17 的磁盘块，P2 指向数据项在17~35 之间的磁盘块，P3 指向数据项大于 35 的磁盘块。
  • 真实的数据存储在于叶子节点中，即 3、5、9、10、13、15、28、29、36、60、75、79、90、99。
  • 非叶子节点只不存储真实的数据，只存储指引搜索方向的数据项。如 17、35 并不真实存在于数据表中。

【查找过程】

  1. 如果要查找数据项 29，那么首先会把磁盘块 1 由磁盘加载到内存中，此时发生一次 IO，在内存中用二分查找确定 29 在17~35 之间，锁定磁盘块 1 的 P2 指针；
  2. 通过磁盘块 1 的 P2 指针的磁盘地址把磁盘块 3 由磁盘加载到内存中，发生第二次 IO，在内存中用二分查找确定 29 在 26~30 之间，锁定磁盘块 3 的 P2 指针；
  3. 通过指针加载磁盘块 8 到内存中，发生第三次IO，同时内存中做二分查找找到29，结束查询，总计三次 IO。

在实际生产中，3 层的 b 树可以表示上百万的数据，如果上百万的数据查找只需要三次IO，性能提高将是巨大的，如果没有索引，每个数据项都要发生一次 IO，那么总共需要百万次的IO，显然成本非常非常高。

哪些情况需要创建索引

  1. 主键自动建立唯一索引；
  2. 频繁作为查询条件的字段应该创建索引；
  3. 查询中与其它表关联的字段，外键关系应该建立索引；
  4. 查询中排序的字段。排序字段若通过索引去访问将大大提高排序速度；
  5. 单键/组合索引的选择问题，组合索引性价比更高；
  6. 查询中统计或者分组字段；

哪些情况不需要创建索引

  1. 表记录太少

  2. 频繁更新的字段（经常增删改的表或者字段）不适合创建索引；【因为每次更新不单单要更新数据库表记录还会更新索引表记录】

  3. Where 条件里用不到的字段不创建索引（不用做查询条件的字段）

  4. 过滤性不好的不适合建索引。

    【假如一个表中有 10w 条记录，有一个字段 gender 只有 0 和 1 两种值，且每个值的分布概率大约为 50%，那么对这种字段建立索引一般不会提高数据库的查询速度】

    【索引的选择是指索引列中不同值的数量与表中记录数的比。如果一个表中有 2000 条记录，表索引列有 1980 个不同的值，那么这个索引的选择性就接近于 1，这个索引的效率越高。】

MySQL锁

是什么

定义

锁是计算机协调多个进程或线程并发访问某一资源时的机制。

在数据库中的数据是一种供许多用户共享的资源，所以就可能出现多个用户并发访问的情况。如何保证数据并发访问的一致性、有效性是所有数据库必须解决的一个问题。锁冲突也是影响数据库并发访问性能的一个重要因素。

生活案例

在淘宝上抢手机，库存只剩最后一件了，如果同时有多个用户同时点击了购买，到底谁买到？

  • 首先，这里肯定要使用到事务，我们先从库存表中取出商品数量，然后插入订单，付款后插入付款表信息，最后更新商品数量。在这个过程中，使用锁可以对有限的资源进行保护，解决隔离和并发的矛盾。

注意事项：

  • 数据库锁都是悲观锁！！！

锁的分类

从对数据操作的类型来分可分为：

  • 读锁（共享锁）：针对同一份数据，多个读操作可以同时读而不会互相影响。
  • 写锁（排它锁）：当前写操作没有完成之前，它会阻断其它写锁和读锁。【注意：写和读都被阻断了】

从对数据操作的粒度来分可分为：

  • 行锁（Row_Level_Lock）：开销大，加锁慢；会出现死锁；锁定粒度最小，发生锁冲突的概率最低,并发度也最高。
  • 表锁（Table_Level_Lock）：开销小，加锁快；不会出现死锁；锁定粒度大，发生锁冲突的概率最高,并发度最低。
  • 页锁：开销和加锁时间界于表锁和行锁之间；会出现死锁；锁定粒度界于表锁和行锁之间，并发度一般。

使用场景：

  • 表锁更适合以查询为主的表使用。

案例分析

给表加读锁

  1. 向上面建好的 tb_emp 表中插入数据
  1. 模拟两个线程同时操作 tb_emp 表

线程 T1 线程 T2
lock table tb_emp read;【OK】
select * from tb_emp; 【OK】
select * from tb_emp; 【OK】
UPDATE tb_emp SET NAME="张飞" WHERE id=1; 【Error】
UPDATE tb_emp SET NAME="张飞" WHERE id=1; 【阻塞】
UNLOCK TABLES;
UPDATE tb_emp SET NAME="张飞" WHERE id=1; 【OK】

分析：

两个线程同时操作数据库中的同一张表，

  1. 首先，线程 T1 先对 emp 表加表共享读锁；
  2. 然后，T1 对加锁的表执行查询操作，OK；【因为 T1 本身就是持有锁的】
  3. 再之，T2 对加锁的表进行查询操作，OK；【因为读锁是一种共享锁，共享锁可以有多个】
  4. T1 在未释放锁之前，想要读取其它表（非加锁的表），发现报错；
  5. T2 读取未加锁的表，OK；
  6. T1 在未释放锁之前，想要修改被加锁的表，报错；
  7. T2 想要修改未加锁的表，发现没有报错 ，但程序被阻塞【一直在转圈，但不能执行结束】
  8. T1 释放读锁；
  9. 这时 T2 立即执行未执行完的更新的操作。

给表加写锁

  • 当前线程对表的 查询+更新+插入 操作都可以执行；
  • 其它线程对表的查询被阻塞，需要等待锁被释放。

总结

  • MyISAM 在执行查询语句（SELECT）前，会自动给涉及的所有表加读锁，在执行增删改操作前，会自动给涉及的表加写锁。
  • 如果想要使用 InnoDB 给表加读/写锁，则需要显式地声明才可以。

结论：

对所有使用 MyISAM 引擎的表进行操作，会有以下情况：

  • 对 MyISAM 表的读操作，不会阻塞其它线程对同一表的读请求，但是会阻塞对同一表的写请求。只有当读锁释放后，才会执行其它线程的写操作。
  • 对 MyISAM 表的写操作，会阻塞其它线程对同一表的读和写操作，只有当写锁释放后，才会执行其它线程的读写操作。

简而言之，就是读的时候可以读，读的时候不能写，写的时候不能读，写的时候不能写（读就是共享读，其它的可以读，不能写；写是独占写，其它的不能读也不能写）

如何分析表锁定

可以通过检查 table_locks_waited 和 table_locks_immediate 状态变量来分析系统上的表锁定。

这里有两个变量记录 MySQL 内部表级锁定的情况，两个变量的说明如下：

  • table_locks_immediate：产生表级锁定的次数，即表锁被获取过的次数，每次获取锁该值+1。
  • table_locks_waited：由于表锁争用而发生等待的次数（不能立即获取锁的次数，每等待一次该值+1）。此值较高则说明存在着较严重的表级锁争用的情况。

MyISAM 不适合作为以写操作为主的表的引擎。因为对表中数据进行的写操作会造成表被自动加写锁，导致其它线程不能做任何操作，大量的更新会使得查询操作很难获取到锁，从而造成永远阻塞。

行锁

特点

  • InnoDB 引擎在执行 SQL 语句时默认加行锁（执行查询语句加默认读锁，执行写操作语句默认加写锁）；
  • 开销大，加锁慢。（需要给每行数据都加锁，自然开销比较大，加锁比较慢）
  • 会出现死锁；
  • 锁的粒度小，
  • 发生锁冲突的概率最低，
  • 并发度最高。

案例分析

建表

行锁的基本演示

  • 模拟两个线程同时操作 stu 表数据。建立两个会话分别为 Session1 和 Session2。
Session1 Session2
SET autocommit = 0 SET autocommit = 0
UPDATE tb_stu SET age=11 WHERE id=1;【OK】
SELECT * FROM tb_stu;【查看后发现更新成功】
SELECT * FROM tb_stu;【未读到Session1未提交的事务】
UPDATE tb_stu SET age=19 WHERE id=1;【写操作被阻塞】
COMMIT;【Session1 提交事务】
【阻塞解除，更新操作继续执行成功。此时，id=1对应的age=19】
  1. 分别关掉 Session1 和 Session2 事务的自动提交行为；
  2. 更新 id=1 的这一条记录，在 Session1 中更新成功，但在 Session2 中并未读取到 Session1 的更新。这是由于 MySQL 默认的隔离级别（可重复读）导致的。
  3. 此时，如果在 Session2 中想要更新同一行数据，发现 Session 的操作被阻塞，这是因为 InnoDB 默认给每行数据加了行锁。
  4. 使用 COMMIT 提交事务后，行锁也会被释放，这时 Session2 的更新操作得以继续执行。

索引失效行锁升级为表锁

行锁必须有索引才能实现，否则会自动锁全表，那么就不是行锁了。

所以在 InnoDB 表中有两种情况会导致行锁升级为表锁：

  • 一是条件字段压根没有建立索引；
  • 二是条件字段的索引在使用中失效了。

  3. 但是我们此时使用 Session2 想要插入一条 id=2 的数据，却发现被阻塞了。这时为什么呢？

    按理来说，行锁只会作用于当前行，在本例中也就是第 3、4、5 行，而 Session2 操作的明明是第 2 行，为什么也会被阻塞呢？这时因为此时 id=2 的那一行也被加了行锁，这就是传说中的间隙锁。

  4. 只有当 Session1 的事务提交后，所有作用于行上的锁被释放了，Session2 的插入操作才能继续完成。

什么是间隙锁？

当我们使用范围条件而不是相等条件检索数据，并请求共享或排它锁时，InnoDB 会给符合条件的已有数据记录的索引项加锁，这是显然的；

对于索引的键值在条件范围内但并不存在的记录，叫做“间隙（GAP）”；

InnoDB 也会对这个“间隙”加锁，这种锁机制就是所谓的间隙锁（Next-Key锁）。

总结：间隙锁 = 使用范围条件检索 + 请求锁 +检索的字段是一个索引

间隙锁的危害

因为 Query 执行过程中通过范围查找的话，它会锁定整个范围内所有的索引键值，即使这个键值在索引表中并不存在。

间隙锁有一个致命的弱点就是当锁定一个范围的键值之后，即使某些不存在的键值也会被无辜的锁定，从而造成在锁定的时候无法插入锁定键值范围内的任何数据。在某些场景下这可能会对性能造成很大的危害。

线索二叉树基本介绍

空指针域计算：

如果一棵二叉树有 n 个节点，那么其必然有（n-1）条连线（即必须使用的指针），按照每个节点有 2 个指针来算，未使用的空指针域共有：2n - (n-1) = (n+1) 个。

  1. n 个结点的完全二叉树中含有 n+1 个空指针域，利用二叉树中的空指针域，存放该结点在某种遍历次序下的前驱和后继结点的指针，这种附加的指针称为”线索“。
  2. 这种加上了线索的二叉数称为线索二叉树(Threaded BinaryTree)，根据线索性质的不同，线索二叉树可分为前序线索二叉树、中序线索二叉树和后序线索二叉树三种。
  3. 一个结点的前一个结点，称为前驱结点。
  4. 一个结点的后一个结点，称为后继结点。

堆排序

堆排序是利用堆这种数据结构而设计的一种排序算法。

  • 堆排序是一种选择排序；
  • 它的最坏，最好，平均时间复杂度均为 O(nlog^n^)；
  • 它也是不稳定排序。

基本思想

堆排序的基本思想是：

  1. 将待排序的无序序列构造成一个大顶堆，此时整个序列的最大值就是堆顶节点；
  2. 然后将堆顶元素与末尾元素进行交换，此时末尾元素就是最大值。
  3. 之后忽略掉末尾元素，将剩下的 n-1 个元素重新构造成一个大顶堆，得到其中的最大值（也就是全部 n 个元素中的次小值）。
  4. 如此反复执行，便能得到一个有序序列了。
  5. 如果一个堆有 n 个元素，那么只需要执行 n-1 次堆的重构即可，因为 n-1 次后就只剩下一个元素了，最后一个元素的位置是确定的。
  6. 可以看到在堆排序的过程中，每轮需要比较的元素的个数是在逐渐减少的，因为每次形成一个大顶堆后，就会将一个元素放到其最终位置上。

  • 根据堆的特性（以大顶堆为例），需要比较堆顶节点与子节点的大小，并将其中较大值提到堆顶位置。如果该节点是叶子节点的话，就没有比较的意义了。
  • 只要是非叶子节点，都有被比较的意义。因为这个节点都有子节点，所以都可能出现子节点大于根节点而导致需要调整的情况。一般情况下是从最后一个非叶子结点开始递减比较之前的所有节点。当然也可以从下标为 0 的根节点开始递增比较一直到最后一个非叶子节点。

赫夫曼树

是什么？

基本介绍

  1. 给定 n 个权值作为 n 个叶子结点，构造一棵二叉树，若该树的带权路径长度（WPL）达到最小，称这样的二叉树为最优二叉树，也称为赫夫曼树(Huffman Tree)。
  2. 赫夫曼树是带权路径长度最短的树，权值较大的结点离根较近。

一些概念

  • 路径
  • 路径长度
  • 结点的权
  • 带权路径长度

如何构建Huffman树？

给你一个数列 {13, 7, 8, 3, 29, 6, 1}，要求转成一颗赫夫曼树。

构成赫夫曼树的步骤：

  1. 将数列按从小到大的顺序进行排序。每个数据都是一个节点，每个节点可以看成是一颗最简单的二叉树。
  2. 取出数列中权值最小的两个节点与一个根节点组成一颗新的二叉树，该新的二叉树的根节点的权值是前面两颗二叉树根节点权值的和。

  1. 定长编码：按照字符的内容，逐一将其转化成二进制码，传递给计算机去执行的编码方式

1 创建事务管理器对象

  • 使用前需要先将 DataSourceTransactionManager 对象添加到 IoC 容器中。

2 配置事务管理器的通知（增强）

事务管理器的通知（简称“事务的通知”）是事务管理器的引用，如果不对其进行动态代理增强的话，就等价于事务管理器。

事务的通知的主要作用就是在通知中动态地添加一些额外的功能。比如：可以在事务的通知中定义事务的属性。

3 配置事务的属性

事务的属性被配置在事务的通知中，被 <tx:attributes> 标签包裹。

<tx:method>

<tx:method> 标签主要用于管理指定的方法。包含如下属性：

  • name：指定要被事务管理的方法的方法名，可以模糊指定或精确指定。【优先级：精准匹配>部分匹配>完全模糊匹配】
  • isolation：用于指定事务的隔离级别。默认值是 Default，表示使用数据库的默认隔离级别。【MySQL 和 Oracle 不同】
  • propagation：用于指定事务的传播行为。默认值是 REQUIRED。
  • read-only：用于指定事务是否只读。只有查询方法才能设置事务为 true。默认值是false，表示读写。
  • timeout：用于指定事务的超时时间。默认值是-1，表示永不超时，如果指定了数值，以秒为单位。
  • rollback-for：用于指定一个异常，当产生该异常时，事务回滚，当产生其它异常时，事务不会滚。没有默认值，表示任何异常都会滚。
  • no-rollback-for：用于指定一个异常，当产生该异常时，事务不会滚。产生其它异常时，事务回滚，没有默认值，表示任何异常都回滚。

事务的传播行为

是什么？

事务的传播行为：当一个方法运行在另一个事务方法中时，必须指定事务该如何传播。

例如：方法可能继续在现有事务中运行，也可能开启一个新事务，在新事务中运行。

事务的传播行为可以由传播属性指定。Spring 中定义了 7 种事务传播行为。

传播属性 描述
REQUIRED（默认） 如果有事务正在运行，当前方法就在这个事务内运行，否则，就启动一个新的事务，并在新的事务内运行。
REQUIRES_NEW 当前方法必须启动新事务，并在它自己的事务（新事务）内运行。如果有事务正在运行，应该将其挂起。
SUPPORTS 如果有事务正在运行，当前方法就在这个事务内运行；否则它可以不运行在事务中。
NOT_SUPPORTED 当前方法不应该运行在事务中，如果有运行中的事务，就将其挂起。
MANDATORY 当前方法必须运行在事务内部，如果没有正在运行的事务，就抛出异常。
NEVER 当前方法不应该运行在事务中，如果有正在运行的事务，就抛出异常。
NESTED 如果有事务在运行，当前方法就应该在这个事务的嵌套事务内运行。否则，就启动一个新的事务，并在它自己的事务内运行。

怎么用？

  1. 事务的传播属性可以在 xml 配置文件中配置。
  1. 事务的传播属性可以在 @Transactional 注解的 propagation 属性中定义。

Demo

我们来模拟一个场景：

现有数据库中有 100 本书，每本书 60 元，模拟客户买书，每次只能买一本，每次买书都需要付一次钱，可以买多次。客户共有 100 元，一直买书直到买不起为止。

​ 假设买书方法 buyBook() 和支付方法 pay() 都使用默认的事务传播行为 Required。在这种情况下，内部方法 pay() 和外层方法 buyBook() 用的是同一个事务，这种情况下会发生：多次买书的行为要么全部成功，要么全部失败（事务的原子性）。

​ 就比如在本例中，第一次买书后，客户余额 -60，库存 -1，pay() 方法正常结束，但是第二次买书时由于余额不足以支付，所以支付失败，这时候需要回滚，由于内层方法和外层方法使用的是同一个事务，所以会直接回滚到库存 100，用户余额 100 的状态。

​ 如果把事务传播行为改成 REQUIRES_NEW，在这种情况下，用户第一次买书后，就已经完成了一次事务，已经将数据库中的数据修改并提交了，即使第二次买书失败，也只会回滚到第二次买书前的状态（事务的隔离性），即库存 99，用户余额 40 的状态。

事务的隔离级别

前置知识

数据库事务并发导致的问题：

  1. 脏读。事务 A 读到了事务 B 更新但未提交的数据。
  2. 不可重复读。（多次获取）事务两次读到的数据不一致。
  3. 幻读。（多次获取）事务两次读到表数据的数量不一致。

隔离级别

  1. 读未提交。允许事务 A 读取事务 B 未提交的修改。
  2. 读已提交。要求事务 A 只能读取事务 B 已提交的修改。
  3. 可重复读 。确保一个事务可以多次从一个字段中读到相同的值。即事务 A 执行期间禁止其它事务对这个字段进行更新。【字段上锁】
  4. 串行化。确保一个事务可以多次从一张表中读到相同数量的行。即在事务 A 执行期间，禁止其它事务对这张表进行添加、更新、删除操作。可避免任何并发问题，但性能差。【表上锁】

怎么用？

在 Spring 中，通过 isolation 属性来设置事务的隔离级别。

  • 可以在 xml 配置文件中设置；
  • 也可以在注解中设置。

事务的隔离级别与使用的数据库产品相关联，比如 Oracle 就只支持两种隔离级别，而 MySQL 支持 4 种隔离级别。Spring 中默认使用的是数据库产品默认的隔离级别。如：

  • Isolation.REPEATABLE_READ，可重复读。MySQL 默认的隔离级别。
  • Isolation.READ_COMMITTED，读已提交。Oracle 默认的隔离级别。

设计模式——代理模式

是什么

代理模式；Proxy；

代理模式是：通过为目标对象提供一个替身，以控制对这个对象的访问，即通过代理对象访问目标对象。通俗的来讲代理模式就是我们生活中常见的中介。

这样做的好处是：可以在目标对象实现的基础上，增强额外的功能，即扩展目标对象的功能。

被代理的对象可以是：远程对象、创建开销大的对象或需要安全控制的对象。

代理模式有不同的形式，主要有两种：

  1. 静态代理
  2. 动态代理：
    • JDK 代理（接口代理）
    • Cglib 代理：可以在内存动态的创建对象，而不需要实现接口

生活中的代理模式

​ 假如说我现在想买一辆二手车，虽然我可以自己去找车源，做质量检测等一系列的车辆过户流程，但是这确实太浪费我得时间和精力了。我只是想买一辆车而已为什么我还要额外做这么多事呢？于是我就通过中介公司来买车，他们来给我找车源，帮我办理车辆过户流程，我只是负责选择自己喜欢的车，然后付钱就可以了。

为什么要使用代理模式

  • 中介隔离作用：在某些情况下，一个客户类不想或者不能直接引用一个委托对象，而代理类对象可以在客户类和委托对象之间起到中介的作用，其特征是代理类和委托类实现相同的接口。

  • 开闭原则，增加功能：代理类除了是客户类和委托类的中介之外，我们还可以通过给代理类增加额外的功能来扩展委托类的功能，这样做我们只需要修改代理类而不需要再修改委托类，符合代码设计的开闭原则。代理类主要负责为委托类预处理消息、过滤消息、把消息转发给委托类，以及事后对返回结果的处理等。代理类本身并不真正实现服务，而是同过调用委托类的相关方法，来提供特定的服务。真正的业务功能还是由委托类来实现，但是可以在业务功能执行的前后加入一些公共的服务。例如我们想给项目加入缓存、日志这些功能，我们就可以使用代理类来完成，而没必要打开已经封装好的委托类。

ReentrantLock 和 AQS 的关系

​ ReentrantLock 类有一个内部类 Sync 继承了 AbstractQueuedSynchronizer 抽象类，说白了就是 ReentrantLock 中包含了一个 AQS 对象，AQS 对象是 ReentrantLock 实现加锁和释放锁的关键性核心组件。

ReentrantLock 加锁和释放锁的底层原理

  1. 线程 T1 加锁

如果现在有一个线程尝试调用 ReentrantLock 的 lock() 方法进行加锁，会发生什么事情呢？

很简单。这个 AQS 对象内部有一个核心变量 private volatile int state，代表了加锁的状态。初始状态下，state == 0，另外，这个 AQS 内部还有一个关键变量，用来记录当前加锁的是哪个线程，初始化状态下，这个变量是 null。

接着线程 T1 调用 ReentrantLock#lock() 方法尝试进行加锁，这个加锁的过程，就是用 CAS 操作将 state 值从 0 变成 1。

如果 T1 之前没有其它线程加锁锁，那么 state==0 恒成立。此时线程 T1 就可以加锁成功。

一旦线程 T1 加锁成功后，就可以设置当前加锁线程是自己。如下图所示，就是线程 T1 加锁的过程。

说白了，AQS 其实就是 JUC 包中的一个核心组件，里面有 state 变量、加锁线程变量等核心的东西，维护了加锁的状态。

而ReentrantLock 只是一个外层的 API，内核中的锁机制实现都是依赖于 AQS 组件的。

ReentrantLock 顾名思义是一个可重入锁，即可以使用一个 ReentrantLock 对象执行多次 lock() 和 unlock() 方法。

那么 AQS 是如何实现可重入加锁的呢？线程 T1 每调用一次 lock() 方法，如果 CAS 比较发现 state != 0，接着会再判断当前加锁线程是不是自己，如果是就把 state 的值累加 1，其余不变。同理，调用 unlock() 释放锁的时候就是把 state 值累减 1。

  1. 线程 T2 加锁

如果线程 T1 加锁之后，线程 T2 想要争夺锁会怎么样？（互斥锁）

  1. 线程 T2 尝试获取锁，调用 lock() 方法，首先会去检查 ReentrantLock.AQS.state 变量是否为 0，发现不为 0，说明有线程正在使用锁，CAS 校验失败；
  2. 接着线程 T2 会继续查看当前加锁的线程是不是自己，“加锁线程”这个变量明确记录了是线程 T1 占用了这个锁，所以此时线程 T2 会加锁失败。

  1. 加锁失败

接着，线程 T2 会将自己放入 AQS 的同步等待队列中，因为自己尝试加锁失败了，此时就要将自己放入等待队列中，直到线程 T1 释放锁之后，自己就可以重新尝试加锁了。

所以 AQS 内部还有一个同步等待队列，专门存放那些加锁失败的线程。

  1. 释放锁

接着，线程 T1 在执行完自己的业务逻辑代码之后，就会释放锁。释放锁的过程非常简单，就是将 AQS 内的 state 变量的值递减 1，如果 state 为 0，则彻底释放锁，会将“加锁线程”变量的值也设置为 null。

  1. 释放锁之后

接下来，会从等待队列的队头唤醒线程 T2 重新尝试加锁。

线程 T2 重新尝试加锁，还是使用 CAS 操作将 state 从 0 变成 1，此时就会加锁成功，并将 state 设置为 1，同时把 “加锁线程” 设置为线程 T2 自己，同时线程 T2 从同步等待队列中出队。

AQS 初步

​ 多线程环境下，为保证线程安全经常需要加锁，加锁就会导致线程阻塞，有阻塞就需要排队，实现排队必然需要队列（抢不到锁的线程被阻塞后就得去排队）。

​ 抢到资源（锁）的线程直接使用资源处理业务逻辑，抢不到资源的线程必然涉及一种排队等候机制。抢占资源失败的线程继续等待，但等候的线程仍然保留获取锁的可能性。

​ 如果共享资源被占用，就需要一定的阻塞等待和唤醒机制来保证锁分配。这个机制主要是通过 CLH（CLH 是三个科学家的首字母缩写）队列的变体实现的，将暂时获取不到锁（被阻塞）的线程加入到队列中，这个队列就是 AQS 的抽象表现。它将请求共享资源（锁）的线程封装成队列的节点（Node），通过 CAS、自旋以及 LockSupport#part() 等方式，维护 state 变量的状态，使并发达到同步的控制效果。

​ AQS 管理的是没有抢到锁的线程。

AQS 数据结构

AQS 使用一个 volatile int 类型的成员变量 state 来表示同步状态，通过内置的 FIFO 队列来完成资源获取的排队工作，将每个要去抢占资源的线程封装成一个 Node 节点来实现锁的分配，通过 CAS 操作完成对 state 值的修改。

AQS 是一个队列：

  • private volatile int state：同步状态，有 3 种状态。0 表示无人占用，1 表示有线程正在占用，大于 1 表示可重入锁。
  • static final class Node：节点是线程的抽象
  • private transient volatile Node head：代表头结点
  • private transient volatile Node tail：代表尾节点

1 为什么使用线程池？（优势）

​ 线程池的主要工作是控制运行线程的数量，处理过程中将任务放入队列。在线程创建后启动这些任务，如果任务数量超过了最大线程数量，超出数量的任务排队等候，等其它线程执行完毕，再从队列中取出任务来执行。

使用线程池的优势：

  1. 线程复用：创建/销毁线程需要消耗系统资源，线程池可以复⽤已创建的线程。
  2. 控制最大并发数：控制并发的数量。并发数量过多，可能会导致资源消耗过多，从⽽造成服务器崩溃。（主要原因）
  3. 管理线程：线程是稀缺资源，如果无限制的创建，不仅会消耗系统资源，还会降低系统的稳定性，使用线程池可以进行统一的分配、调优和监控
  4. 提高响应速度。当任务到达时，任务可以不要等待线程创建就能立即执行。

2 怎么用

  • 当我们使用线程池时，就不再需要把任务交给某个具体的线程了，我们只需要把任务交给线程池即可，线程池内部会分配一个空闲的线程去执行该任务。

2.1 架构说明

2.3 源码分析

查看底层源码发现，这三个创建线程池的方法底层都使用了 ThreadPoolExecutor 类。

  1. newFixedThreadPool
  • 创建一个定长的线程池，可控制线程的最大并发数，超出的线程数量的任务会在队列中等待。
  • newFixedThreadPool 创建的线程池 corePoolSize 和 maximumPoolSize 值是相等的，
  • 底层使用的数据结构是 LinkedBlockingQueue。
  • 执行长期的任务，性能较好

  1. newSingleThreadExecutor
  • 创建一个只包含一个线程的线程池，只会用唯一的线程来执行任务，保证所有任务按照指定顺序执行。
  • newSingleThreadExecutor 将 corePoolSize 和 maximumPoolSize 都设置为 1，
  • 底层使用的数据结构是 LinkedBlockingQueue。
  • 适用于一个任务一个任务执行的场景

  1. newCachedThreadPool
  • 创建一个自适应长度的线程池，即多退少补。
  • newCachedThreadPool 将 corePoolSize 设置为 0，将 maximumPoolSize 设置为 Integer.MAX_VALUE，也就是说来了任务就创建线程执行任务，当线程空闲超过 60s，就销毁线程。
  • 底层使用的数据结构是 SynchronousQueue。
  • 适用于执行很多短期异步的小程序或者负载较轻的服务器。

3 ThreadPoolExecutor 构造方法中的 7 大参数

ThreadPoolExecutor 有多个重载的构造方法，方法中的参数为 5~7 个不等，我们直接看一下最多的 7 个参数。

  1. corePoolSize：线程池中常驻核心线程数
  • 线程池中的常驻线程数，即使没有任务时也需要去维护的线程数量。
    • 当有请求任务进来后，首先会安排常驻线程去执行任务；
    • 当线程池中正在使用的线程数量达到 corePoolSize 后，再有新的任务进来，就会把新进来的任务放到缓存队列中。
  1. maximumPoolSize：线程池能够容纳的最大线程数，此值必须大于等于 1。
  2. keepAliveTime：多余的空闲线程的最大存活时间。[多余线程就是除了常驻线程之外的线程]
  • 如果线程池中的线程数量超过 corePoolSize，并且线程的空闲时间达到 keepAliveTime 值后，多余的空闲线程将会被销毁直到只剩下 corePoolSize 个线程。
    • 只有当前线程池中的线程数大于 corePoolSize 时 keepAliveTime 才会生效，直到线程池中的线程数不大于 corePoolSize。
  1. unit：keepAliveTime 的时间单位。
  2. workQueue：任务队列，被提交但尚未执行的任务。
  3. threadFactory：生成线程池中线程实例的工厂，一般使用默认的即可。不同的工厂都能生产线程实例，就类似于联想和戴尔工厂都能生产电脑，但具体的生产过程是不一样的。
  4. handler：拒绝策略。当阻塞队列满了并且工作线程大于等于线程池的最大线程数时如何拒绝请求任务的策略。
  • 线程池的所有线程都已经在工作了，并且等待执行的任务队列也满了，这时候就需要拒绝新的任务加入，采用的策略就叫做拒绝策略。

4.2 工作流程

工作流程图

在创建了线程池后，线程池会等待提交过来的任务请求。每次调用 execute() 方法添加新的任务请求时，线程池中会进行如下判断：

  1. 如果线程池中正在运行的线程数量小于 corePoolSize，那么马上分配一个空闲线程执行这个任务；
  2. 如果线程池中正在运行的线程数量大于等于 corePoolSize，会将这个任务放入阻塞队列；
  3. 如果这时候阻塞队列也满了且正在运行的线程数量小于 maximumPoolSize，就会创建非核心线程来执行这个任务；
  4. 如果队列满了且正在运行的线程数量大于等于 maximumPoolSize，那么线程池会执行饱和拒绝策略。
  5. 当一个线程执行完一个任务时，它会从阻塞队列中选取下一个任务进行执行。
  6. 当一个线程无事可做超过一定的时间（keepAliveTime）时，线程池会判断：
    • 如果当前线程池中运行的线程数量大于 corePoolSize ，那么这个线程就会被销毁；
    • 所以线程池的所有任务完成后它最终会收缩到 corePoolSize 的大小。

小总结：

​ 任务首先放到线程池的核心池中，放不下就放在阻塞队列，再放不下就对线程池扩容，直到扩容到最大。这时候如果继续有很多线程要进入，启动拒绝策略；如果这时候线程开始逐渐减少，就把线程池中超过空闲时间的线程销毁掉，直到线程池中线程数等于核心线程数。

5 线程池的4种拒绝策略

  1. AbortPolicy（默认）：直接抛出 RuntimeExecutionException 异常，阻止系统正常运行。

  2. DiscardOldestPolicy：抛弃阻塞队列中等待最久的任务，然后把当前任务加入等待队列中尝试再次提交当前任务。

  3. DiscardPolicy：直接丢弃任务，不予任何处理也不抛出异常。如果允许任务丢失，这是最好的一种方案。

  4. CallerRunsPolicy：“调用者“运行一种调节机制，该策略既不会抛弃任务，也不会抛出异常，而是将某些任务回退到调用者，让调用者线程去处理这个任务，从而降低新任务的流量。

    注解：假设 main 线程是线程的调用者，main 线程将一个任务 A 分配给线程池处理，但此时线程池中的运行着的线程已经达到了 maximumPoolSize，即无法再给任务 A 分配一个新的线程了，需要拒绝这个任务。如果采用的拒绝策略是 CallerRunsPolicy，那么任务 A 不会被丢弃，也不会出现异常，而是会将任务 A 会退给 main 线程，让 main 线程去执行任务 A 。

以上 4 种内置的拒绝策略的实现类均实现了 RejectedExecutionHandler 接口。

6 线程在实际开发中的使用（阿里巴巴 Java 开发手册）

  1. 在实际开发中，线程资源必须通过线程池提供，不允许开发人员在程序中自行显式地创建线程。

    注解：使用线程池的好处是减少在创建和销毁线程上所消耗的时间以及系统资源的开销，解决资源不足的问题。如果不使用线程池，有可能造成系统创建大量同类线程而导致消耗完内存或者“过度切换上下文线程”的问题。

  1. 实际开发中，线程池不允许通过 Executors 工具类去创建，而是要通过 ThreadPoolExecutor 的方式去创建，这样的处理方式让开发人员更加明确线程池的规则，规避资源耗尽的风险。

    注解：Executors 创建的线程池对象的弊端如下：

    1. FixedThreadPool 和 SingleThreadPool：

      这两种线程池维护的请求（阻塞）队列虽然是有界队列，但是允许的最大长度为 Integer.MAX_VALUE，这个值实际上非常大，可能会堆积大量的请求，从而导致 OOM。

    2. CachedThreadPool 和 ScheduledThreadPool

      这两种线程池虽然使用的请求队列是同步阻塞队列，最大容量为1，但是允许创建的最大线程数量为 Integer.MAX_VALUE，也可能会创建大量的线程，从而导致 OOM。

如何合理配置线程池参数

CPU密集型任务

  • CPU 密集的意思是该任务需要大量的运算，而没有阻塞，CPU 一直全速运行。
  • CPU 密集任务只有在真正的多核 CPU 上才能得到加速（通过多线程）
  • 而在单核CPU上，无论你开几个模拟的多线程该任务都不可能得到加速，因为CPU总的运算能力有限。
  • CPU 密集型任务的配置尽可能少的线程数量。
  • 一般公式：CPU 核数 + 1个线程的线程池

IO密集型

  • IO 密集型，即该任务需要大量的 IO，即大量的阻塞。
  • 在单线程上运行 IO 密集型的任务会导致大量的 CPU 运算能力浪费在等待。
  • 所以在 IO 密集型任务中使用多线程可以大大的加速程序运行。即在单核 CPU 上，这种加速主要就是利用了被浪费掉的阻塞时间。
  • 由于 IO 密集型任务并不是一直在执行任务，则应该配置尽可能多的线程，如 CPU核数×2
