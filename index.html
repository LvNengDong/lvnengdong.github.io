是什么

  • Java中，被 enum 关键字修饰的类型就是枚举类型。
  • 枚举类型是一种特殊的数据类型。它是一种与 Class、Interface、Annotation 平级的数据类型。

  • 如果枚举类中只一个对象，则可以作为一种单例模式的实现方式。
  • 枚举类型(enum)、类(Class)、接口(Interface)、注解类(Annotation)处于同一级。
  • 注解类不是注解，注解类的作用是定义注解的业务逻辑

泛型的引入背景

JDK5 之前

​ 集合类在设计阶段阶段不能确定这个集合到底实际存的是什么类型的对象，所以在 JDK5 之前只能把元素类型设计为 Object，如果我们对集合不做任何限定，它就可以存储任何类型的对象。如果一个集合中存储了多种数据类型的元素，那么在编译后这些元素都会向上转型为Object类型。这样的话，当我们取出任一元素想执行其特有的方法时，就必须采用向下转型。然而由于集合中元素的数据类型不同，所以没有一个统一的标准可以一次性将集合中的所有元素向下转型。

从 JDK5 开始

​ 在JDK1.5++，新增了泛型（Generic）语法。让你在设计 API 时可以指定类或方法支持泛型，这样我们使用API时也变得更为简洁，并得到了编译时期的语法检查。

​ JDK5 之后使用泛型来解决这个问题。因为这个时候除了元素的类型不确定，其它的部分是确定的，例如关于这个元素如何保存，如何管理等是确定的，因此此时把元素的类型设计成一个参数，这个类型参数叫做泛型。

泛型的概念

定义

  • 泛型：可以在类中或方法中预支的使用未知的类型。
  • 所谓泛型，就是允许在定义类、接口时通过一个标识表示类中某个属性的类型或者是某个方法的返回值及参数类型。这个类型参数将在使用时确定。比如：①继承或实现这个接口时； ②创建对象时； ③传入实际的类型参数时，也称为类型实参；
  • 泛型的使用：定义时不指定数据类型，使用时再指定数据类型。

小总结

  1. 泛型是在定义时使用的；
  2. 使用时才指定泛型的具体类型；
  3. 当没有指定泛型时，默认类型为 Object 类型。
  4. Collection虽然可以存储不同类型的对象，但实际使用时我们建议一般只存储同一类型的对象。

使用泛型的好处

  1. 将运行时期的ClassCastException，转移到了编译时期变成了编译失败。
  2. 避免了类型强转的麻烦。

异常

Posted on 2021-07-31 In JavaSE

1 是什么

定义：异常是指程序在执行过程中出现的非正常情况，最终会导致程序的非正常停止。

【异常并不是语法错误，语法错误的话编译时就会直接报错，不会产生字节码文件，根本不能运行。】

  • 在 Java 等面向对象的编程语言中，异常本身就是一个类，产生异常就是创建异常对象并抛出了一个异常对象。
  • Java处理异常的方式是中断处理。

2 异常的体系结构

  • Throwable 是类，不是接口
  • 同样的，Error 和 Exception 继承了 Throwable ，所以它俩也是类
  • RuntimeException 继承了 Exception，也是一个类；
  • 但是 CheckedException 并没有真正地继承 Exception 类，并且它是一个接口。

​ 在 Java 中，所有的异常都有一个共同的祖先 java.lang.Throwable 类。Throwable 类有两个重要的子类 Exception（异常） 和 Error（错误） ，二者都是 Java 异常处理的重要子类，各自都包含大量子类。

Error

java.lang.Error

Error 错误是程序无法处理的错误。大多数是 JVM 层面的问题。即代码在逻辑上是正确的，但是由于 JVM 内存不足或者某些对象申请的内存空间过大导致等原因导致程序出现错误。 这些错误是在编译期是不可查的，因为它们在应用程序的控制和处理能力之外。

​ 例如，Java虚拟机运行错误（Virtual MachineError），当 JVM 不再有继续执行操作所需的内存资源时，将出现 OutOfMemoryError。这些异常发生时，Java虚拟机（JVM）一般会选择线程终止。

Exception

java.lang.Exception

Exception 是程序本身可以处理的异常。Exception 又可分为 编译时异常 与 运行时异常 两大类。

  • 编译时异常：执行 javac.exe 命名时，可能出现的异常；
  • 运行时异常：执行 java.exe 命名时，出现的异常；

4 异常处理方式

异常的两种处理方式：1、“抛” 2、“抓”

  1. “抛”
  • 程序在正常执行的过程中，一旦出现异常，就会在异常代码处产生一个对应异常类的对象，并将此对象抛出。
  • 一旦抛出异常对象以后，其后的代码就不再执行。

关于异常对象的产生：

  1. 系统自动生成的异常对象；
  2. 手动的生成一个异常对象，并抛出（throw）。
  1. “抓”

可以理解为异常的处理方式：①try-catch-finally ②throws

异常处理方式一：try-catch-finally

使用说明：

说明：

  1. finally 是可选的。
  2. 使用 try 将可能出现异常代码包起来，在执行过程中，一旦出现异常，就会生成一个对应异常类的对象，根据此异常对象的类型，去 catch 中进行匹配；
  3. 一旦 try 中的异常对象匹配到某一个 catch 时，就进入 catch 中进行异常的处理。一旦处理完成，就跳出当前的 try-catch 结构，继续执行 try 中后续的代码，最后再执行 finally 中的代码。
  4. catch 中的异常类型如果没子父类关系，则谁声明在上，谁声明在下无所谓。但是 catch 中的异常类型如果满足子父类关系，则要求子类异常类型一定要声明在父类异常类型的上面，否则报错。【因为 catch 是从上向下匹配的，如果父类异常类型声明在先，则该父类异常的所有子类异常都会被父类异常拦截（多态），根本执行不到子类异常处 】
  5. 常用的异常对象处理的方式： ① String getMessage() ② printStackTrace()
  6. 在 try 结构中声明的变量，再出了 try 结构以后，就不能再被调用。
  7. try-catch-finally 结构可以嵌套。

如何看待代码中的编译时异常和运行时异常？

  1. 使用 try-catch-finally 处理编译时异常，使得程序在编译时就不再报错，但是运行时仍可能报错。相当于我们使用 try-catch-finally 将一个编译时可能出现的异常，延迟到运行时出现。
  2. 开发中，由于运行时异常比较常见，所以我们通常就不针对运行时异常编写 try-catch-finally 了。针对于编译时异常，我们则一定要考虑异常的处理。

异常处理方式二：throws

  • throws + 异常类型 写在方法的声明处。指明此方法执行时，可能会抛出的异常类型。
  • 一旦当方法体执行时出现异常，仍会在异常代码处生成一个异常类的对象，此对象满足 throws 抛出的异常类型时，就会被抛出。异常代码后续的代码，就不再执行。

对比两种处理方式

  • try-catch-finally：真正的将异常给处理掉了。
  • throws：只是将异常抛给了方法的调用者，并没真正将异常处理掉。

体会开发中应该如何选择两种处理方式？

  • 如果父类中被重写的方法没 throws 方式处理异常，则子类重写的方法也不能使用 throws。这意味着如果子类重写的方法中异常，必须使用 try-catch-finally 方式处理。
  • 执行的方法 methodA() 中，先后又调用了另外的几个方法，这几个方法是递进关系执行的。我们建议这几个方法使用 throws 的方式进行处理，而执行的方法 methodA() 可以考虑使用 try-catch-finally 方式进行处理。
  • 使用 throws 抛出的异常最外层还是要落实到使用 try-catch-finally 处理。

补充：

  • 方法重写的规则之一：子类重写的方法抛出的异常类型不大于父类被重写的方法抛出的异常类型。【因为父类是子类的顶级抽象，如果子类的异常类型范围大于父类类型的话，则父类就会无法处理超出范围的异常类型】

  • 如果没有就先计算，计算完之后再把结果存到map中。

动态规划（非递归解法）

  • 想要计算 F(n) 的值，就需要知道 F(n-1) 和 F(n-2) 的值；
  • 想要计算 F(n-1) 的值，就要知道 F(n-2) 和 F(n-3) 的值；
  • 以此类推，一直到 F(0) 和 F(1)。

​ 正如上面讲的普通的递归算法的弊端一样，如果每次 F(n-1) 和 F(n-2) 都独立运算，那么重复计算的值就太多了。上面的优化方式是将已经计算过的值保存到一个 Map 中避免重复运算，而且采用的是自顶向下的计算方式，即从 F(n-1) 算到 F(1)。

​ 而所谓的斐波那契的动态规划解法，其实可以看做是一种穷举法。就是将从 F(1) 到 F(n-1) 的所有值都计算出来，保存到一个 List（或者 Map 均可）中，但是这里采用的是自底向上，即从 F(1) 计算到 F(n-1) 。

​ 所以，在计算 F(n) 之前，动态规划必定已经建立好了一个从 F(0) ~ F(n-1) 的数组。而结算结果就是从数组中取出对应的元素即可。

分治算法

分治算法介绍

​ 分治法是一种很重要的算法，顾名思义就是“分而治之”，即把一个复杂的问题分解成两个或更多相同或相似的子问题，再继续把子问题分成更小的子问题，直到最后子问题可以简单的直接求解，原问题的解即子问题的解的合并。

​ 分治思想是很多高效算法的基础，如排序算法（快速排序、归并排序），傅立叶变换（快速傅立叶变换）等。

分治算法的基本步骤

分治法在每一层递归上都有三个步骤：

  1. 分解：将原问题分解为若干个规模较小，相互独立，与原问题形式相同的子问题；
  2. 解决：若子问题规模较小而容易被解决则直接解，否则递归地解各个子问题；
  3. 合并：将各个子问题的解合并为原问题的解。

  • ADHOC(P) 是该分治法中的基本子算法，用于直接解决小规模的问题 P。因此，当 P 的规模不超过 n0 时直接用算法 ADHOC(P) 求解。

我们用 OPT(i) 函数表示包含第 i 个任务的情况下报酬最高的任务组合。【OPT：最优解】

动态规划算法介绍

  1. 动态规划（Dynamic Programming）算法的核心思想是：将大问题划分为小问题进行解决，从而一步步获取最优解的处理算法。
  2. 动态规划算法与分治算法类似，其基本思想也是将待求解问题分解成若干个子问题，先求解子问题，然后从这些子问题的解得到原问题的解。
  3. 与分治法不同的是，适合于用动态规划求解的问题，经分解得到的子问题往往不是互相独立的。即下一个子问题的求解是建立在上一个子问题解的基础上得到的。
  4. 动态规划可以通过填表的方式来逐步推进，得到最优解.

1.1 MySQL逻辑架构

​ MySQL 可以在多种不同场景中应用并发挥良好作用。主要体现在存储引擎的架构上，插件式的存储引擎架构将查询处理和其它的系统任务以及数据的存储提取相分离。这种架构可以根据业务的需求和实际需要选择合适的存储引擎。其主要逻辑架构可分为四层，从上到下分别是：

连接层主要处理客户端与 MySQL 服务端的连接、授权认证、安全等问题。包含本地 socket 通信和大多数基于客户端/服务端工具实现的类似于 TCP/IP 的网络通信。主要完成一些类似于连接处理、授权认证、及相关的安全方案。在该层上引入了线程池的概念，为通过认证安全接入的客户端提供线程。同样在该层上可以实现基于 SSL 的安全链接。服务器也会为安全接入的每个客户端验证它所具有的操作权限。

Parser：解析器

SQL Interface 中的 SQL 命令传递到解析器的时候会被解析器验证和解析。主要功能有：

  • 验证：对接收的 SQL 语句进行语义和语法的分析，如果在分解构成中遇到错误，那么就说明这个 SQL 语句是不合理的，验证失败，不继续网下执行，直接抛出错误。
  • 解析：如果验证成功，则根据语义和语法，将 SQL 语句划分为不同的操作类型，然后进行针对性的转发到后续步骤。比如查询语句就转发到查询对应的服务，更新操作就转发到更新对应的服务。

Optimizer：查询优化器

SQL 语句在查询之前会使用查询优化器对查询进行优化。就是优化客户端请求的 SQL 语句，根据客户端请求的 SQL 语句，和数据库中的一些统计信息，在一系列算法的基础上进行分析，得出一个最优的策略，告诉后面的程序如何执行这个 SQL 语句。

查询优化器就类似于指令重排，对于开发人员来说，写一条 SQL 可能自我感觉良好，性能天下第一，但是对于底层机器来说，它有自己的优化标准，经过查询优化器优化后的 SQL 语句的执行顺序并不一定与开发人员设计的 SQL 执行顺序相同。

Cache和Buffer：查询缓存

如果查询缓存有命中的查询结果，查询语句就可以直接去查询缓存中取数据。这个缓存机制是由一系列小缓存组成的。比如表缓存，记录缓存，key 缓存，权限缓存等。

索引

是什么

官方定义：

索引（Index）是帮助 MySQL 高效获取数据的数据结构。

索引的目的在于提高查询效率。

​ 可以类比为字典。如果想要查询 “mysql” 这个单词，首先肯定要定位到 m 字母，然后在 m 字母的范围内继续找第 2 个 y 字母，以此类型，找到剩下的三个字母。

​ 如果没有索引，那么每找一个字母都需要全部查找一遍。

索引可以理解为“排好序的快速查找的数据结构”！！！

索引会影响到两大功能的执行效率：

  1. 排序：ORDER BY
  2. 查找：WHERE；HAVING

​ 在数据之外，数据库系统还维护着满足特定查找算法的数据结构，这些数据结构以某种方式指向数据（记录了真实真实数据的地址），在这些数据结构上我们可以实现高效的查找算法。这种数据结构，就是索引。

下图就是一种可能的索引方式示例：

  • 左边是索引表，一共有两列七条记录，最左边记录的是真实数据的物理地址，Col1 是主键字段，Col2 是索引字段。
  • 为了加快 Col2 的查找，可以维护一个右边所示的二叉查找树，每个节点分别包含索引键值和一个指向对应数据记录物理地址的指针，这样就可以运用二叉查找在一定的复杂度内获取到相应数据，从而快速的检索出符合条件的记录。
  • 一般来说索引本身也很大，不可能全部存储在内存中，因此索引往往以索引文件的形式存储的磁盘上。

优缺点

优势：

  • 索引能提高数据检索速率，对于一些经常被用于查询的字段，如果将其设置为索引，能大大地提高查询的速率。将频繁使用的字段就将其设置为索引，比不设索引时查询速度要快。

  • 通过索引列对数据进行排序，降低数据排序的成本。

劣势：

  • 实际上索引也是一张表，该表保存了主键与索引字段，并指向实体表的记录。所以索引也是要占用空间的。
  • 虽然索引大大提高了查询速度，但同时却会降低更新表的速度。比如在对表进行 INSERT、UPDATE 和 DELETE 操作时，因为更新表时 MySQL 不仅要更新数据，还要更新索引表。
    • 对于 UPDATE 操作，记录变更后索引字段在索引表中的位置就可能发生变化，需要更新。
    • 对于 INSERT 操作，添加记录后需要将新记录的索引字段加到索引表中。同理 DELETE 需要删除索引表中的记录。

索引分类

  • 单值索引：一个索引只包含单个列，一个表可以有多个单列索引。
  • 唯一索引：索引列的值必须唯一（即用于建立索引的列必须符合 UNIQUE），但允许有空值。
  • 复合索引：即一个索引包含多个列。

检索原理

【初始化介绍】

一颗 B 树，浅蓝色的块我们称之为一个磁盘块，可以看到每个磁盘块包含了数据项（深蓝色所示）和指针（黄色所示）。如磁盘块1 包含数据项 17 和35，包含指针 P1、P2、P3。

  • P1 指向数据项小于 17 的磁盘块，P2 指向数据项在17~35 之间的磁盘块，P3 指向数据项大于 35 的磁盘块。
  • 真实的数据存储在于叶子节点中，即 3、5、9、10、13、15、28、29、36、60、75、79、90、99。
  • 非叶子节点只不存储真实的数据，只存储指引搜索方向的数据项。如 17、35 并不真实存在于数据表中。

【查找过程】

  1. 如果要查找数据项 29，那么首先会把磁盘块 1 由磁盘加载到内存中，此时发生一次 IO，在内存中用二分查找确定 29 在17~35 之间，锁定磁盘块 1 的 P2 指针；
  2. 通过磁盘块 1 的 P2 指针的磁盘地址把磁盘块 3 由磁盘加载到内存中，发生第二次 IO，在内存中用二分查找确定 29 在 26~30 之间，锁定磁盘块 3 的 P2 指针；
  3. 通过指针加载磁盘块 8 到内存中，发生第三次IO，同时内存中做二分查找找到29，结束查询，总计三次 IO。

在实际生产中，3 层的 b 树可以表示上百万的数据，如果上百万的数据查找只需要三次IO，性能提高将是巨大的，如果没有索引，每个数据项都要发生一次 IO，那么总共需要百万次的IO，显然成本非常非常高。

哪些情况需要创建索引

  1. 主键自动建立唯一索引；
  2. 频繁作为查询条件的字段应该创建索引；
  3. 查询中与其它表关联的字段，外键关系应该建立索引；
  4. 查询中排序的字段。排序字段若通过索引去访问将大大提高排序速度；
  5. 单键/组合索引的选择问题，组合索引性价比更高；
  6. 查询中统计或者分组字段；

哪些情况不需要创建索引

  1. 表记录太少

  2. 频繁更新的字段（经常增删改的表或者字段）不适合创建索引；【因为每次更新不单单要更新数据库表记录还会更新索引表记录】

  3. Where 条件里用不到的字段不创建索引（不用做查询条件的字段）

  4. 过滤性不好的不适合建索引。

    【假如一个表中有 10w 条记录，有一个字段 gender 只有 0 和 1 两种值，且每个值的分布概率大约为 50%，那么对这种字段建立索引一般不会提高数据库的查询速度】

    【索引的选择是指索引列中不同值的数量与表中记录数的比。如果一个表中有 2000 条记录，表索引列有 1980 个不同的值，那么这个索引的选择性就接近于 1，这个索引的效率越高。】

MySQL锁

是什么

定义

锁是计算机协调多个进程或线程并发访问某一资源时的机制。

在数据库中的数据是一种供许多用户共享的资源，所以就可能出现多个用户并发访问的情况。如何保证数据并发访问的一致性、有效性是所有数据库必须解决的一个问题。锁冲突也是影响数据库并发访问性能的一个重要因素。

生活案例

在淘宝上抢手机，库存只剩最后一件了，如果同时有多个用户同时点击了购买，到底谁买到？

  • 首先，这里肯定要使用到事务，我们先从库存表中取出商品数量，然后插入订单，付款后插入付款表信息，最后更新商品数量。在这个过程中，使用锁可以对有限的资源进行保护，解决隔离和并发的矛盾。

注意事项：

  • 数据库锁都是悲观锁！！！

锁的分类

从对数据操作的类型来分可分为：

  • 读锁（共享锁）：针对同一份数据，多个读操作可以同时读而不会互相影响。
  • 写锁（排它锁）：当前写操作没有完成之前，它会阻断其它写锁和读锁。【注意：写和读都被阻断了】

从对数据操作的粒度来分可分为：

  • 行锁（Row_Level_Lock）：开销大，加锁慢；会出现死锁；锁定粒度最小，发生锁冲突的概率最低,并发度也最高。
  • 表锁（Table_Level_Lock）：开销小，加锁快；不会出现死锁；锁定粒度大，发生锁冲突的概率最高,并发度最低。
  • 页锁：开销和加锁时间界于表锁和行锁之间；会出现死锁；锁定粒度界于表锁和行锁之间，并发度一般。

使用场景：

  • 表锁更适合以查询为主的表使用。

表锁 表共享读锁 表独占写锁

案例分析

给表加读锁

  1. 向上面建好的 tb_emp 表中插入数据
  1. 模拟两个线程同时操作 tb_emp 表

线程 T1 线程 T2
lock table tb_emp read; 【OK】
select * from tb_emp; 【OK】
select * from tb_emp; 【OK】
UPDATE tb_emp SET NAME="张飞" WHERE id=1; 【Error】
UPDATE tb_emp SET NAME="张飞" WHERE id=1; 【阻塞】
UNLOCK TABLES;
UPDATE tb_emp SET NAME="张飞" WHERE id=1; 【OK】

分析：

两个线程同时操作数据库中的同一张表，

  1. 首先，线程 T1 先对 emp 表加表共享读锁；
  2. 然后，T1 对加锁的表执行查询操作，OK；【因为 T1 本身就是持有锁的】
  3. 再之，T2 对加锁的表进行查询操作，OK；【因为读锁是一种共享锁，共享锁可以有多个】
  4. T1 在未释放锁之前，想要读取其它表（非加锁的表），发现报错；
  5. T2 读取未加锁的表，OK；
  6. T1 在未释放锁之前，想要修改被加锁的表，报错；
  7. T2 想要修改未加锁的表，发现没有报错 ，但程序被阻塞【一直在转圈，但不能执行结束】
  8. T1 释放读锁；
  9. 这时 T2 立即执行未执行完的更新的操作。

给表加写锁

  • 当前线程对表的 查询+更新+插入 操作都可以执行；
  • 其它线程对表的查询被阻塞，需要等待锁被释放。

总结

  • MyISAM 在执行查询语句（SELECT）前，会自动给涉及的所有表加读锁，在执行增删改操作前，会自动给涉及的表加写锁。
  • 如果想要使用 InnoDB 给表加读/写锁，则需要显式地声明才可以。

结论：

对所有使用 MyISAM 引擎的表进行操作，会有以下情况：

  • 对 MyISAM 表的读操作，不会阻塞其它线程对同一表的读请求，但是会阻塞对同一表的写请求。只有当读锁释放后，才会执行其它线程的写操作。
  • 对 MyISAM 表的写操作，会阻塞其它线程对同一表的读和写操作，只有当写锁释放后，才会执行其它线程的读写操作。

简而言之，就是读的时候可以读，读的时候不能写，写的时候不能读，写的时候不能写（读就是共享读，其它的可以读，不能写；写是独占写，其它的不能读也不能写）

如何分析表锁定

可以通过检查 table_locks_waited 和 table_locks_immediate 状态变量来分析系统上的表锁定。

这里有两个变量记录 MySQL 内部表级锁定的情况，两个变量的说明如下：

  • table_locks_immediate：产生表级锁定的次数，即表锁被获取过的次数，每次获取锁该值+1。
  • table_locks_waited：由于表锁争用而发生等待的次数（不能立即获取锁的次数，每等待一次该值+1）。此值较高则说明存在着较严重的表级锁争用的情况。

MyISAM 不适合作为以写操作为主的表的引擎。因为对表中数据进行的写操作会造成表被自动加写锁，导致其它线程不能做任何操作，大量的更新会使得查询操作很难获取到锁，从而造成永远阻塞。

行锁

特点

  • InnoDB 引擎在执行 SQL 语句时默认加行锁（执行查询语句加默认读锁，执行写操作语句默认加写锁）；
  • 开销大，加锁慢。（需要给每行数据都加锁，自然开销比较大，加锁比较慢）
  • 会出现死锁；
  • 锁的粒度小，
  • 发生锁冲突的概率最低，
  • 并发度最高。

索引失效行锁升级为表锁

行锁必须有索引才能实现，否则会自动锁全表，那么就不是行锁了。

所以在 InnoDB 表中有两种情况会导致行锁升级为表锁：

  • 一是条件字段压根没有建立索引；
  • 二是条件字段的索引在使用中失效了。

  3. Session1 提交后，释放表锁，Session2 得以继续进行，修改成功。

间隙锁案例：

Session1 Session2

  1. 分别关闭 Session1 和 Session2 的事务自动提交；

  2. 更新 id 为（1，6）范围内的所有数据，由于当前表中没有 id 为 2 的数据，所以就是更新 id 为 3，4，5 的数据 age=99。

  3. 但是我们此时使用 Session2 想要插入一条 id=2 的数据，却发现被阻塞了。这时为什么呢？

按理来说，行锁只会作用于当前行，在本例中也就是第 3、4、5 行，而 Session2 操作的明明是第 2 行，为什么也会被阻塞呢？这时因为此时 id=2 的那一行也被加了行锁，这就是传说中的**间隙锁**。
  1. 只有当 Session1 的事务提交后，所有作用于行上的锁被释放了，Session2 的插入操作才能继续完成。

什么是间隙锁？

当我们使用范围条件而不是相等条件检索数据，并请求共享或排它锁时，InnoDB 会给符合条件的已有数据记录的索引项加锁，这是显然的；

对于索引的键值在条件范围内但并不存在的记录，叫做“间隙（GAP）”；

InnoDB 也会对这个“间隙”加锁，这种锁机制就是所谓的间隙锁（Next-Key锁）。

总结：间隙锁 = 使用范围条件检索 + 请求锁 +检索的字段是一个索引

间隙锁的危害

因为 Query 执行过程中通过范围查找的话，它会锁定整个范围内所有的索引键值，即使这个键值在索引表中并不存在。

间隙锁有一个致命的弱点就是当锁定一个范围的键值之后，即使某些不存在的键值也会被无辜的锁定，从而造成在锁定的时候无法插入锁定键值范围内的任何数据。在某些场景下这可能会对性能造成很大的危害。

线索二叉树基本介绍

空指针域计算：

如果一棵二叉树有 n 个节点，那么其必然有（n-1）条连线（即必须使用的指针），按照每个节点有 2 个指针来算，未使用的空指针域共有：2n - (n-1) = (n+1) 个。

  1. n 个结点的完全二叉树中含有 n+1 个空指针域，利用二叉树中的空指针域，存放该结点在某种遍历次序下的前驱和后继结点的指针，这种附加的指针称为”线索“。
  2. 这种加上了线索的二叉数称为线索二叉树(Threaded BinaryTree)，根据线索性质的不同，线索二叉树可分为前序线索二叉树、中序线索二叉树和后序线索二叉树三种。
  3. 一个结点的前一个结点，称为前驱结点。
  4. 一个结点的后一个结点，称为后继结点。

堆排序

堆排序是利用堆这种数据结构而设计的一种排序算法。

  • 堆排序是一种选择排序；
  • 它的最坏，最好，平均时间复杂度均为 **O(nlog^n^)**；
  • 它也是不稳定排序。

基本思想

堆排序的基本思想是：

  1. 将待排序的无序序列构造成一个大顶堆，此时整个序列的最大值就是堆顶节点；
  2. 然后将堆顶元素与末尾元素进行交换，此时末尾元素就是最大值。
  3. 之后忽略掉末尾元素，将剩下的 n-1 个元素重新构造成一个大顶堆，得到其中的最大值（也就是全部 n 个元素中的次小值）。
  4. 如此反复执行，便能得到一个有序序列了。
  5. 如果一个堆有 n 个元素，那么只需要执行 n-1 次堆的重构即可，因为 n-1 次后就只剩下一个元素了，最后一个元素的位置是确定的。
  6. 可以看到在堆排序的过程中，每轮需要比较的元素的个数是在逐渐减少的，因为每次形成一个大顶堆后，就会将一个元素放到其最终位置上。

  • 只要是非叶子节点，都有被比较的意义。因为这个节点都有子节点，所以都可能出现子节点大于根节点而导致需要调整的情况。一般情况下是从最后一个非叶子结点开始递减比较之前的所有节点。当然也可以从下标为 0 的根节点开始递增比较一直到最后一个非叶子节点。

赫夫曼树

是什么？

基本介绍

  1. 给定 n 个权值作为 n 个叶子结点，构造一棵二叉树，若该树的带权路径长度（WPL）达到最小，称这样的二叉树为最优二叉树，也称为**赫夫曼树(Huffman Tree)**。
  2. 赫夫曼树是带权路径长度最短的树，权值较大的结点离根较近。

一些概念

  • 路径
  • 路径长度
  • 结点的权
  • 带权路径长度

如何构建Huffman树？

给你一个数列 {13, 7, 8, 3, 29, 6, 1}，要求转成一颗赫夫曼树。

构成赫夫曼树的步骤：

  1. 将数列按从小到大的顺序进行排序。每个数据都是一个节点，每个节点可以看成是一颗最简单的二叉树。
  2. 取出数列中权值最小的两个节点与一个根节点组成一颗新的二叉树，该新的二叉树的根节点的权值是前面两颗二叉树根节点权值的和。
  3. 再将这棵新二叉树的根节点权值加入数列中再次从小到大排序，不断重复 1-2-3 的步骤，直到数列中，所有的数据都被处理，就得到一颗赫夫曼树。

Spring TX；Spring事务管理

基于 XML 配置文件的事务管理

1 创建事务管理器对象

  • Spring 框架为我们提供了管理事务的对象 DataSourceTransactionManager，这个对象中集成了事务的开启、提交、回滚、关闭等操作。
  • 使用前需要先将 DataSourceTransactionManager 对象添加到 IoC 容器中。
  • DataSourceTransactionManager 对象依赖于 DataSource 对象。

2 配置事务管理器的通知（增强）

事务管理器的通知（简称“事务的通知”）是事务管理器的引用，如果不对其进行动态代理增强的话，就等价于事务管理器。

事务的通知的主要作用就是在通知中动态地添加一些额外的功能。比如：可以在事务的通知中定义事务的属性。

3 配置事务的属性

事务的属性被配置在事务的通知中，被 <tx:attributes> 标签包裹。

<tx:method>

<tx:method> 标签主要用于管理指定的方法。包含如下属性：

  • name：指定要被事务管理的方法的方法名，可以模糊指定或精确指定。【优先级：精准匹配>部分匹配>完全模糊匹配】
  • isolation：用于指定事务的隔离级别。默认值是 Default，表示使用数据库的默认隔离级别。【MySQL 和 Oracle 不同】
  • propagation：用于指定事务的传播行为。默认值是 REQUIRED。
  • read-only：用于指定事务是否只读。只有查询方法才能设置事务为 true。默认值是false，表示读写。
  • timeout：用于指定事务的超时时间。默认值是-1，表示永不超时，如果指定了数值，以秒为单位。
  • rollback-for：用于指定一个异常，当产生该异常时，事务回滚，当产生其它异常时，事务不会滚。没有默认值，表示任何异常都会滚。
  • no-rollback-for：用于指定一个异常，当产生该异常时，事务不会滚。产生其它异常时，事务回滚，没有默认值，表示任何异常都回滚。

  • 即要对哪个（些）方法施加配置好的事务。

5 将事务通知织入到切入点

Tip：

  • 一般情况下，AOP 中的切面使用 <aop:aspect> 标签；
  • 而在事务管理中，AOP 切面则使用语义化更加明确的 <aop:advisor> 标签。

6 总结（重要）

  • 为了保证代码的可读性和易维护性，我们在 Service 层编写的对数据库的操作都是屏蔽了事务操作的、简单易懂的代码。
  • 而把对事务的控制交给了 Spring 的 AOP 去控制。[等价于使用代理对象]
    • 所以 Spring 配置文件中的切入点就是 Service 层中实现类，再通过事务的属性 <tx:method name=""> 定位到实现类中的具体方法，给对应的方法添加配置文件中配置好的事务；

所以只要将 事务的通知 织入 到 切入点 中，就可以完成事务的自动控制。【把事务的通知织入切入点 就是 把事务相关的方法插入到业务方法中执行 的意思。】

@Transactional 注解

作用：该注解的属性与 XML 配置文件中的事务属性的含义一致。

作用范围：该注解可以在接口上、类上和方法上使用。

  • 在接口上使用，表示该接口的所有实现类都支持该事务；
  • 在类上使用，表示该类中的所有方法都支持该事务；
  • 在方法上使用，表示该方法支持该事务。
  • 以上 3 个位置的优先级：方法> 类> 接口

事务的传播行为

是什么？

事务的传播行为：当一个方法运行在另一个事务方法中时，必须指定事务该如何传播。

例如：方法可能继续在现有事务中运行，也可能开启一个新事务，在新事务中运行。

事务的传播行为可以由传播属性指定。Spring 中定义了 7 种事务传播行为。

传播属性 描述
REQUIRED（默认） 如果有事务正在运行，当前方法就在这个事务内运行，否则，就启动一个新的事务，并在新的事务内运行。
REQUIRES_NEW 当前方法必须启动新事务，并在它自己的事务（新事务）内运行。如果有事务正在运行，应该将其挂起。
SUPPORTS 如果有事务正在运行，当前方法就在这个事务内运行；否则它可以不运行在事务中。
NOT_SUPPORTED 当前方法不应该运行在事务中，如果有运行中的事务，就将其挂起。
MANDATORY 当前方法必须运行在事务内部，如果没有正在运行的事务，就抛出异常。
NEVER 当前方法不应该运行在事务中，如果有正在运行的事务，就抛出异常。
NESTED 如果有事务在运行，当前方法就应该在这个事务的嵌套事务内运行。否则，就启动一个新的事务，并在它自己的事务内运行。

怎么用？

  1. 事务的传播属性可以在 xml 配置文件中配置。
  1. 事务的传播属性可以在 @Transactional 注解的 propagation 属性中定义。

Demo

我们来模拟一个场景：

现有数据库中有 100 本书，每本书 60 元，模拟客户买书，每次只能买一本，每次买书都需要付一次钱，可以买多次。客户共有 100 元，一直买书直到买不起为止。

​ 假设买书方法 buyBook() 和支付方法 pay() 都使用默认的事务传播行为 Required。在这种情况下，内部方法 pay() 和外层方法 buyBook() 用的是同一个事务，这种情况下会发生：多次买书的行为要么全部成功，要么全部失败（事务的原子性）。

​ 就比如在本例中，第一次买书后，客户余额 -60，库存 -1，pay() 方法正常结束，但是第二次买书时由于余额不足以支付，所以支付失败，这时候需要回滚，由于内层方法和外层方法使用的是同一个事务，所以会直接回滚到库存 100，用户余额 100 的状态。

​ 如果把事务传播行为改成 REQUIRES_NEW，在这种情况下，用户第一次买书后，就已经完成了一次事务，已经将数据库中的数据修改并提交了，即使第二次买书失败，也只会回滚到第二次买书前的状态（事务的隔离性），即库存 99，用户余额 40 的状态。

事务的隔离级别

前置知识

数据库事务并发导致的问题：

  1. 脏读。事务 A 读到了事务 B 更新但未提交的数据。
  2. 不可重复读。（多次获取）事务两次读到的数据不一致。
  3. 幻读。（多次获取）事务两次读到表数据的数量不一致。

隔离级别

  1. 读未提交。允许事务 A 读取事务 B 未提交的修改。
  2. 读已提交。要求事务 A 只能读取事务 B 已提交的修改。
  3. 可重复读 。确保一个事务可以多次从一个字段中读到相同的值。即事务 A 执行期间禁止其它事务对这个字段进行更新。【字段上锁】
  4. 串行化。确保一个事务可以多次从一张表中读到相同数量的行。即在事务 A 执行期间，禁止其它事务对这张表进行添加、更新、删除操作。可避免任何并发问题，但性能差。【表上锁】

怎么用？

在 Spring 中，通过 isolation 属性来设置事务的隔离级别。

  • 可以在 xml 配置文件中设置；
  • 也可以在注解中设置。

事务的隔离级别与使用的数据库产品相关联，比如 Oracle 就只支持两种隔离级别，而 MySQL 支持 4 种隔离级别。Spring 中默认使用的是数据库产品默认的隔离级别。如：

  • Isolation.REPEATABLE_READ，可重复读。MySQL 默认的隔离级别。
  • Isolation.READ_COMMITTED，读已提交。Oracle 默认的隔离级别。
