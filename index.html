设计模式——代理模式

Posted on 2021-07-19 Edited on 2021-07-20 In 设计模式

是什么

代理模式；Proxy；

代理模式是：通过为目标对象提供一个替身，以控制对这个对象的访问，即通过代理对象访问目标对象。通俗的来讲代理模式就是我们生活中常见的中介。

这样做的好处是：可以在目标对象实现的基础上，增强额外的功能，即扩展目标对象的功能。

被代理的对象可以是：远程对象、创建开销大的对象或需要安全控制的对象。

代理模式有不同的形式，主要有两种：

  1. 静态代理
  2. 动态代理：
    • JDK 代理（接口代理）
    • Cglib 代理：可以在内存动态的创建对象，而不需要实现接口

生活中的代理模式

​ 假如说我现在想买一辆二手车，虽然我可以自己去找车源，做质量检测等一系列的车辆过户流程，但是这确实太浪费我得时间和精力了。我只是想买一辆车而已为什么我还要额外做这么多事呢？于是我就通过中介公司来买车，他们来给我找车源，帮我办理车辆过户流程，我只是负责选择自己喜欢的车，然后付钱就可以了。

为什么要使用代理模式

  • 中介隔离作用：在某些情况下，一个客户类不想或者不能直接引用一个委托对象，而代理类对象可以在客户类和委托对象之间起到中介的作用，其特征是代理类和委托类实现相同的接口。

  • 开闭原则，增加功能：代理类除了是客户类和委托类的中介之外，我们还可以通过给代理类增加额外的功能来扩展委托类的功能，这样做我们只需要修改代理类而不需要再修改委托类，符合代码设计的开闭原则。代理类主要负责为委托类预处理消息、过滤消息、把消息转发给委托类，以及事后对返回结果的处理等。代理类本身并不真正实现服务，而是同过调用委托类的相关方法，来提供特定的服务。真正的业务功能还是由委托类来实现，但是可以在业务功能执行的前后加入一些公共的服务。例如我们想给项目加入缓存、日志这些功能，我们就可以使用代理类来完成，而没必要打开已经封装好的委托类。

LockSupport

是什么？

LockSupport 是用来创建锁和其它同步类的基本线程阻塞原语。

LockSupport 是 JUC 包下的一个类，也可以用于多线程间通信，是 wait/notify（唤醒/等待）机制的改良加强版。

LockSupport 是一个线程阻塞工具类，类中所有的方法都是静态方法，可以让线程在任意位置阻塞，阻塞之后也有对应的唤醒方法。归根结底，LockSupport 调用的是 Unsafe 类中的 native 方法。

LockSupport 提供了 park() 和 unpark() 方法实现阻塞线程和解除线程阻塞的过程。

为什么学 LockSupport

  1. LockSupport 是学习 AQS 的前置知识。
  2. 传统的等待/唤醒机制在某些情况下不适用

Java –>> JVM

JUC –>> AQS –> 前置知识（可重入锁、LockSupport）

JUC 底层原理就是 AQS，就相当于 JVM 之于 Java，学习 AQS 需要先了解 LockSupport 和 可重入锁。

让 notify() 先于 wait() 执行，会导致某个线程进入 Waiting 状态后无法被唤醒，程序无法结束。

执行结果

分析

  1. 在线程 T1 争夺锁对象之前让其先暂停 1s，避免其与 T2 线程争夺锁对象。
  2. T2 线程先得到锁对象，在控制台上输出指定的内容，并随机唤醒当前锁对象维护的等待队列中的一个线程，将其移动到锁对象的同步队列中（当前锁对象的等待队列为空）。最后执行完毕释放锁对象。
  3. T1 线程休眠 1s 后醒来获得锁对象，开始执行同步代码块中的方法，将指定内容打印输出到控制台上。但是在执行到 13 行 obj.wait(); 代码后，由于唤醒动作已经先于阻塞动作执行了，此时线程 T1 进入 Waiting 状态后再没有额外的唤醒动作要执行了，所以 T1 会一直陷入 Waiting 状态，该行代码之后的打印输出动作也就不会被执行到。
  4. 整个进程无法结束。

附加

如果将唤醒操作设置成超时唤醒，则 wait(long timeout) 可以不用与 notify() 方法成对出现，也可以不用关心与 notify() 方法出现的顺序。

总结

  • wait() 和 notify() 方法必须要在 synchronized 同步代码里面且成对使用，且使用时必须先 wait 后 notify（一般情况）。

LockSupport 类中park等待和unpark唤醒

  • 见下文

主要方法

在 LockSupport 中，通过 park() 和 unpark(Thread thread) 方法来实现阻塞和唤醒线程的操作。

原理

  • LockSupport 类使用了一种名为 Permit（许可证）的概念来做到阻塞和唤醒线程的功能，Permit 与线程绑定。
  • Permit 只有两个值 1 和 0，默认是 0。
  • 0 表示线程没有许可证，1 表示线程拥有许可证。每个线程允许携带的许可证的最大上限是 1。

阻塞

park()：阻塞当前线程（会消耗掉一个许可证）

park(Object blocker)：阻塞传入的指定线程（会消耗掉一个许可证）

调用 LockSupport#park() 方法时，底层调用的是 UNSAFE.park() 方法

默认情况 permit == 0，对于一个新的线程直接调用 park() 方法，该线程没有许可证可以被消耗，所以当前线程就会阻塞，直到其它线程给该线程颁发一个许可证，即设置该线程的 permit == 1 时，该线程会从阻塞状态中恢复运行，执行park()方法，消耗掉一个许可证，设置 permit ==0 ，并继续执行 park() 方法之后的代码。

唤醒

unpark(Thread thread)：唤醒某个线程。严格意义上将并不是唤醒线程，而是给线程颁发了一个许可证，这个许可证可以不立即生效，只有在遇到 park() 方法造成的阻塞时，拥有许可证可以保证线程不被阻塞。

调用 unpark(thread) 方法后，设置thread 线程的许可证 permit == 1（注意多次调用 unpark()，permit 的值不会累加，最高只能是 1），自动唤醒 thread 线程，即之前阻塞中的 LockSupport.park() 方法会立即返回。

形象的理解

线程阻塞时需要消耗凭证（permit），这个凭证每个线程最多只有 1 个。

  1. 当调用 park() 方法时，
    • 如果有凭证，则会直接消耗掉这个凭证，然后程序继续执行。
    • 如果无凭证，则必须阻塞等待凭证可用
  2. 而 unpark() 正好相反，它会增加一个凭证，但凭证最多只能有一个，累加无效。

permit 许可证与线程绑定。

permit 是一张万能的许可证，可以提前发给线程，线程每遇到一个关卡（阻塞）就需要消耗许可证。如果有则通过关卡，没有则不让通过，即阻塞。且每个线程最大只能持有一张许可证。

面试题

  1. 为什么可以先唤醒线程后阻塞线程？

unpark() 方法并不是直接唤醒阻塞的线程，而是相当于给线程发放了一个万能的通行证，只要有这个通行证在，线程在遇到阻塞时，直接消费这个通行证就可以通过而不会被阻塞。

  1. 为什么唤醒两次后阻塞两次，但最终结果还会阻塞线程？

因为一个线程持有通行证的最大数量只能为 1，所以调用两次 unpark() 和调用一次 unpark() 效果一样，最多只能获得一个通行证。而调用两次 park() 则会消费两个通行证，证不够就不能放行。

AQS 原理概述（乞丐版）（以 ReentrantLock 为例）

ReentrantLock 和 AQS 的关系

​ ReentrantLock 类有一个内部类 Sync 继承了 AbstractQueuedSynchronizer 抽象类，说白了就是 ReentrantLock 中包含了一个 AQS 对象，AQS 对象是 ReentrantLock 实现加锁和释放锁的关键性核心组件。

ReentrantLock 加锁和释放锁的底层原理

  1. 线程 T1 加锁

如果现在有一个线程尝试调用 ReentrantLock 的 lock() 方法进行加锁，会发生什么事情呢？

很简单。这个 AQS 对象内部有一个核心变量 private volatile int state，代表了加锁的状态。初始状态下，state == 0，另外，这个 AQS 内部还有一个关键变量，用来记录当前加锁的是哪个线程，初始化状态下，这个变量是 null。

接着线程 T1 调用 ReentrantLock#lock() 方法尝试进行加锁，这个加锁的过程，就是用 CAS 操作将 state 值从 0 变成 1。

如果 T1 之前没有其它线程加锁锁，那么 state==0 恒成立。此时线程 T1 就可以加锁成功。

一旦线程 T1 加锁成功后，就可以设置当前加锁线程是自己。如下图所示，就是线程 T1 加锁的过程。

说白了，AQS 其实就是 JUC 包中的一个核心组件，里面有 state 变量、加锁线程变量等核心的东西，维护了加锁的状态。

而ReentrantLock 只是一个外层的 API，内核中的锁机制实现都是依赖于 AQS 组件的。

ReentrantLock 顾名思义是一个可重入锁，即可以使用一个 ReentrantLock 对象执行多次 lock() 和 unlock() 方法。

那么 AQS 是如何实现可重入加锁的呢？线程 T1 每调用一次 lock() 方法，如果 CAS 比较发现 state != 0，接着会再判断当前加锁线程是不是自己，如果是就把 state 的值累加 1，其余不变。同理，调用 unlock() 释放锁的时候就是把 state 值累减 1。

  1. 线程 T2 加锁

如果线程 T1 加锁之后，线程 T2 想要争夺锁会怎么样？（互斥锁）

  1. 线程 T2 尝试获取锁，调用 lock() 方法，首先会去检查 ReentrantLock.AQS.state 变量是否为 0，发现不为 0，说明有线程正在使用锁，CAS 校验失败；
  2. 接着线程 T2 会继续查看当前加锁的线程是不是自己，“加锁线程”这个变量明确记录了是线程 T1 占用了这个锁，所以此时线程 T2 会加锁失败。

  1. 加锁失败

接着，线程 T2 会将自己放入 AQS 的同步等待队列中，因为自己尝试加锁失败了，此时就要将自己放入等待队列中，直到线程 T1 释放锁之后，自己就可以重新尝试加锁了。

所以 AQS 内部还有一个同步等待队列，专门存放那些加锁失败的线程。

  1. 释放锁

接着，线程 T1 在执行完自己的业务逻辑代码之后，就会释放锁。释放锁的过程非常简单，就是将 AQS 内的 state 变量的值递减 1，如果 state 为 0，则彻底释放锁，会将“加锁线程”变量的值也设置为 null。

  1. 释放锁之后

接下来，会从等待队列的队头唤醒线程 T2 重新尝试加锁。

线程 T2 重新尝试加锁，还是使用 CAS 操作将 state 从 0 变成 1，此时就会加锁成功，并将 state 设置为 1，同时把 “加锁线程” 设置为线程 T2 自己，同时线程 T2 从同步等待队列中出队。

是什么

为什么用？（优势）

​ 线程池做的工作主要是控制运行的线程的数量，处理过程中将任务放入队列。然后在线程创建后启动这些任务，如果线程数量超过了最大最大数量超出数量的线程排队等候，等其它线程执行完毕，再从队列中取出任务来执行。

主要特点：线程复用、控制最大并发数、管理线程

  1. 降低资源消耗。通过重复利用已创建的线程降低线程创建和回收造成的消耗。
  2. 提高响应速度。当任务到达时，任务可以不要等待线程创建就能立即执行。
  3. 提高线程的可管理性。线程是稀缺资源，如果无限制的创建，不仅会消耗系统资源，还会降低系统的稳定性，使用线程池可以进行统一的分配、调优和监控

怎么用

架构说明

Java 中的线程池是通过 Executor 框架实现的，该框架用到了 Executor、Executors、ExecutorService、ThreadPoolExecutor 这几个类。

代码实现（3/5种）

  1. Executors.newFixedThreadPool(int nThreads)
  2. Executors.newSingleThreadExecutor()
  3. Executors.newCachedThreadPool()

查看底层源码发现，这三个创建线程池的方法底层都使用了 ThreadPoolExecutor 类。

  1. newFixedThreadPool
  • 创建一个定长的线程池，可控制线程的最大并发数，超出的线程会在队列中等待。
  • newFixedThreadPool 创建的线程池 corePoolSize 和 maximumPoolSize 值是相等的，它使用的是 LinkedBlockingQueue。
  • 执行长期的任务，性能较好

  1. newSingleThreadExecutor
  • 创建一个只包含一个线程的线程池，只会用唯一的线程来执行任务，保证所有任务按照指定顺序执行。
  • newSingleThreadExecutor 将 corePoolSize 和 maximumPoolSize 都设置为 1，它使用的是 LinkedBlockingQueue。
  • 一个任务一个任务执行的场景

  1. newCachedThreadPool
  • 创建一个自适应长度的线程池，即多退少补。
  • newCachedThreadPool 将 corePoolSize 设置为 0，将 maximumPoolSize 设置为 Integer.MAX_VALUE，使用的是 SynchronousQueue。也就是说来了任务就创建线程执行任务，当线程空闲超过 60s，就销毁线程。
  • 适用于：执行很多短期异步的小程序或者负载较轻的服务器

几个重要参数（7大参数）

  1. corePoolSize：线程池中常驻核心线程数

    • 线程池中常驻线程数，即使没有任务时也需要维护的线程数。
    • 当有请求任务进来后，首先会安排常驻线程去执行请求任务；
    • 当线程池中已使用的线程数量达到 corePoolSize 后，就会把新到达的任务放到缓存队列中。

  2. maximumPoolSize：线程池能够容纳同时执行的最大线程数，此值必须大于等于 1。

  3. keepAliveTime：多余的空闲线程的存活时间。[多余就是除了常驻线程之外的线程]

    如果线程池中的线程数量超过 corePoolSize，并且空闲时间达到 keepAliveTime 值时，多余的空闲线程将会被销毁直到只剩下 corePoolSize 个线程。

    默认情况下，只有当前线程池中的线程数大于 corePoolSize 时 keepAliveTime 才会起作用，直到线程池中的线程数不大于 corePoolSize。

  4. unit：keepAliveTime 的时间单位

  5. workQueue：任务队列，被提交但尚未执行的任务

  6. threadFactory：生成线程池中工作线程的工厂，一般使用默认的即可。

  7. handler：拒绝策略。表示当阻塞队列满了并且工作线程大于等于线程池的最大线程数时如何拒绝请求任务的策略。

    （线程池的所有线程都已经在工作了，并且等待执行的任务队列也满了，这时候就需要拒绝新的任务加入）

底层工作原理

工作原理

工作流程图

在创建了线程池后，线程池会等待提交过来的任务请求。每当调用 execute() 方法添加一个任务请求时，线程池会做如下判断：

  1. 如果线程池中正在运行的线程数量小于 corePoolSize，那么马上创建线程运行这个任务；
  2. 如果线程池中正在运行的线程数量大于等于 corePoolSize，那么将这个任务放入阻塞队列；
  3. 如果这时候阻塞队列也满了且正在运行的线程数量小于 maximumPoolSize，就会创建非核心线程来执行这个任务；
  4. 如果队列满了且正在运行的线程数量大于等于 maximumPoolSize，那么线程池会执行饱和拒绝策略。
  5. 当一个线程完成任务时，它会从阻塞队列中选取下一个任务来执行。
  6. 当一个线程无事可做超过一定的时间（keepAliveTime）时，线程池会判断：
    • 如果当前线程池中运行的线程数量大于 corePoolSize ，那么这个线程就会被销毁；
    • 所以线程池的所有任务完成后它最终会收缩到 corePoolSize 的大小。

​ 任务首先放到线程池的核心池中，放不下就放在阻塞队列，再放不下就对线程池扩容，直到扩容到最大。这时候如果继续有很多线程要进入，启动拒绝策略；如果这时候线程开始逐渐减少，就把线程池中超过一定空闲时间的销毁掉，直到线程池中线程数等于核心线程数。

线程池的4种拒绝策略

  1. AbortPolicy（默认）：直接抛出 RuntimeExecutionException 异常，阻止系统正常运行。
  2. CallerRunsPolicy：“调用者“运行一种调节机制，该策略既不会抛弃任务，也不会抛出异常，而是将某些任务回退到调用者，从而降低新任务的流量。

以上内置拒绝策略均实现了 RejectedExecutionHandler 接口。

CPU密集型

CPU 密集的意思是该任务需要大量的运算，而没有阻塞，CPU 一直全速运行。

CPU 密集任务只有在真正的多核 CPU 上才能得到加速（通过多线程）

而在单核CPU上，无论你开几个模拟的多线程该任务都不可能得到加速，因为CPU总的运算能力有限。

CPU 密集型任务的配置尽可能少的线程数量。

一般公式：CPU 核数 + 1个线程的线程池

IO密集型

IO 密集型，即该任务需要大量的 IO，即大量的阻塞。

在单线程上运行 IO 密集型的任务会导致大量的 CPU 运算能力浪费在等待。

所以在 IO 密集型任务中使用多线程可以大大的加速程序运行。即在单核 CPU 上，这种加速主要就是利用了被浪费掉的阻塞时间。

  • 由于 IO 密集型任务并不是一直在执行任务，则应该配置尽可能多的线程，如 CPU核数×2

公平锁 & 非公平锁

是什么

  • 公平锁：

指多个线程按照申请锁的顺序来获取锁，即谁先申请锁谁就优先获得锁。类似于生活中的排队，先到先得。

  • 非公平锁：

只多个线程获得锁的顺序并不是按照申请锁的顺序，有可能后申请锁的线程比先申请锁的线程优先获得锁。在高并发情况下，有可能造成优先级反转或者饥饿现象。

  • 优先级反转现象：就是后申请锁的线程比先申请锁的线程优先获得锁的现象，后申请锁的线程本该是一个低优先级的线程，但是出现了优先级反转，它就提前获得了锁对象。（类似于插队）

  • 饥饿现象：饥饿现象是指如果一直存在插队的线程获得锁，就会导致本该获得锁的线程一直处于等待状态，这就叫做饥饿状态。

二者区别

  • 公平锁

在并发环境中，每个线程在争夺公平锁时会先查看此锁对象维护的等待队列，如果为空，或者当前线程是等待队列中的第一个，才占有锁，否则就会加入到等待队列中，按照先进先出的规则等到自己获取锁。

  • 非公平锁

如果锁对象是一个非公平锁，在并发环境中，每当有新线程进入争夺锁对象时，这个线程会直接去尝试占有锁，如果尝试失败，才会再采用类似公平锁的方式，让线程去锁对象维护的等待队列中排队。

在这种情况下，存在这样一种情况：线程 A 是当前持有锁的对象，线程 A 释放锁之后，按照优先级顺序本该线程 B 得到这个锁对象，但是此时有一个新线程 Z 进来了，这个 Z 不管三七二十一首先就去尝试占有锁了，如果这时候刚好处于 A 释放完锁而 B 有没有获取到锁的时间段，那么 Z 就会直接占有锁，而 B 获取不到锁后，只能继续在等待队列中排队了。这就是 B 线程被插队了。

是什么

可重入锁指的是同一线程的外层方法获得锁之后，如果内层的递归方法仍然需要获得该锁，则可以直接使用，不会因为外层方法的锁对象还没释放而阻塞。（前提是锁对象得是同一个对象）

两（多）个需要锁的递归的同步方法，只使用了一把锁，这把锁在进入了最外层方法后，还可以继续在内存的同步方法中使用，即一锁多用

可重入锁就是说某个线程获得某把锁的使用权，在没有归还锁的之前，这把锁可以一直使用！！！

可重入锁的一个优点是可以一定程度避免死锁。

死锁则是一个外层同步方法需要 A 锁才能进入，进入之后内层方法则需要 B 锁才能进入，使用的不是同一把锁。

1.  如果锁对象 **atomicReference.value** 是第一次被使用，由于锁对象是使用无参构造函数 **new AtomicReference()** 创建的，所以主内存中 `atomicReference.value = null`，代表此时没有其它线程使用该锁对象，当前线程可以获取到锁。

2.  在线程 T1 使用锁的 5s 内，线程 T2 也进来了，通过 myLock 方法尝试获取锁对象，由于此时主内存中 value 值已经不为空了（变成了 T1 线程的 currentThread），所以 compareAndSet 方法返回值为 false，取反为 true，线程 T2 进入循环体，不停地做 CAS 判断。

3.  当线程 T1 执行结束后，会调用 **myUnLock** 方法，由于 T1 执行过程中 T2 始终没有更新主内存中的 value 成功，所以此时主内存中的 value 还是与 T1 本地内存中的 currentThread 快照的值相同，T1 可以成功将主内存中的 value 修改为 null。

4.  由于 value 使用 volatile 修饰，具有可见性。所以当线程 T1 更新 value 后，线程 T2 能马上监听到，这段时间内 T2 一直在执行 myLock 方法，监听到 value 变为 null 后，终于 compareAndSet 方法可以退出循环了，这时候它会把 T2 线程的当前线程对象写回主内存中的 value 位置上，重新由 T2 来占用这把锁。

5.  在 T2 持有锁的时间段内，如果有其它线程进入也只能不断进行自旋直到 T2 执行完毕，修改主内存中的 value 为 null 后才能继续得到锁。

AtomicInteger 为什么要用 CAS 而不是 synchronized ？

CAS 是什么

CAS 是一种思想，可以保证在多线程环境下，不加锁就能保证对象的一致性（同步性）。

  • 在 JDK5 之前，Java 语言是靠 synchronized 关键字来保证同步的，也就是加锁。锁机制在同一时间段只允许一个线程来修改变量，一致性得到了保障，但并发性下降；
  • CAS 在不加锁的情况下保证了变量的一致性，并发性高。

独占锁是一种悲观锁，synchronized 就是一种独占锁，会导致其它所有需要该锁的线程挂起，等待有锁的线程释放锁。

另一种更加有效的锁就是乐观锁，所谓乐观锁就是，每次不加锁而是假设在没有冲突的情况下去完成某项工作，如果最后因发生冲突失败就重试，直到成功为止，乐观锁用到的机制就是 CAS。

CAS；Compare And Swap；比较并交换

​ 在线程从本地内存向主内存写回数据时，如果线程的期望值与主内存的真实值相等，则执行更新（交换）操作；如果线程的期望值与主内存的真实值不相等，不执行更新操作。

是什么

​ CAS 不是真正的加锁，也就是说允许多个线程同时访问一个变量。在 JMM 内存模型中我们已经了解到，线程在工作时不会直接操作主内存中的数据，而是会将主内存中的数据拷贝回自己的本地内存中后再进行操作，操作完成之后再将数据写回主内存中。

​ 原子性不安全的问题就是：根据 JMM 模型，每个线程操作主内存中的变量都需要经历 从主内存中复制到本地内存 -->> 本地内存中对变量进行运算 -->> 将运算结果写回主内存中 三个步骤。原子性就是这三个步骤读、更新、写回是一体的、不可分割的，要么全部执行成功，要么全部执行失败。如果原子性不能得到保证，在多个线程同时操作一份本地变量 X 时，就会出现 T1 线程在经过读、更新后被挂起，然后 T2 线程完整的执行了 读、更新、写回 三个步骤，修改了主内存中的变量 X 为 Y。这时候 T1 线程继续执行，这时候写回主内存的数据是错误的，因为它没有基于主内存中最新的值来进行计算。

​ CAS 思想要求每个线程在拷贝主内存中的数据时在本地内存中保存一份主内存数据的快照，在本地内存中执行完运算任务要写回主内存时，首先会拿着这个快照与当前时刻主内存中数据进行对比，如果快照数据与主内存当前数据一致，则校验成功，再将线程本地内存中的数据写回主内存中；反之，如果校验，则不会将本地内存数据写回主内存。而是让线程重新获取主内存中的数据，重新运算，重新判断是否可以写回主内存中，直到 CAS 校验成功，将本地内存数据写回主内存中为止。

  • 线程的期望值
  • 主内存中的真实值

​ 在 CAS 的具体操作中，我们习惯将快照值称为线程的期望值，因为多线程中每个线程肯定期望自己在本地内存中运算的这段时间内没有其它线程去更新主内存中的值，这样就不会产生线程不一致。

​ 但实际上在多线程环境下，由于多个线程操作同一变量，且每个线程执行的时间长短不一致，时间较短的线程肯定会先于时间较长的线程修改主内存中的共享变量，当长时间的线程执行完线程任务准备将更新后的数据写回主内存中时，这一时刻主内存中的共享变量可能发生了改变，也可能没发生改变，这个值就叫做主内存中的真实值。

​ CAS 就是当主内存中的真实值和线程的期望值相同时，才执行交换（即将线程本地内存中值写回主内存）动作；如果真实值和期望值不相同，则拒绝执行交换操作。

​ 这就类似于 Git 或 Svn 中代码的提交，多个人协同办公就相当于多线程，假设最初的时候每个人都拿到了 1.0 版本的代码，之后在这个版本上进行迭代更新，小A每工作一小时就提交一次代码，而小B则是每天下班之前才提交一次代码。当第一天工作结束后，小B直接提交代码时就会报错，因为他没有兼容最新版的代码，只有在拉取最新的代码、并检查与自己的代码无冲突时，它才可以进行提交。

Usafe 类

​ 用 Java 语言实现的方法无法直接访问底层的操作系统，需要通过本地（Native）方法来访问。

​ sun.misc.Unsafe 类是 CAS 的核心类，该类中含有大量的本地方法，可以可以像 C 的指针一样直接操作内存，直接从操作系统层面操作指定内存位置的数据。

​ Java 中 CAS 操作的执行依赖于 Unsafe 类中的方法。Unsafe 类中大多数的方法都是 native 修饰的，可以直接调用操作系统的底层资源执行相应的任务。

​ Unsafe 类中的本地方法都具有原子性，因为其底层是采用 CAS 并发原语实现的。

CAS 并发原语

​ CAS（Compare-And-Swap）是一条 CPU 并发原语。它的功能是判断内存中某个位置的值是否为预期值，如果是则更改为新的值，这个过程是原子的。

​ CAS 并发原语在 Java 中的体现就是 sun.misc.Unsafe 类中的各个本地方法都具有原子性。调用 Unsafe 类的中的本地方法（CAS 方法），JVM 会帮我们实现出 CAS 汇编指令。CAS 原语能保证原子性则完全依赖于硬件的功能。再次强调，由于 CAS 是一种系统原语，具有原子性，所以不会造成多线程下数据不一致的问题。

原语是由若干条指令组成的、用于完成某个功能的一组指令集，这组指令集的执行顺序是确定的，且在执行过程中不允许被中断，具有原子性。可以直接将原语视为一条大的、执行时间较长的指令。

系统原语就是操作系统把一些常用的原语直接实现在硬件上，以便遇到时方便调用。

    • 接下来要执行 compareAndSwapInt 方法了，该方法执行过程中不能被其它线程打断，
      • 该方法首先根据 var1 + var2 得到主内存中当前时刻 value 的值；
      • var5 是线程的本地内存在创建快照的时刻主内存中 value 的值。
      • 比较这两个值是否相等，若相等，则执行 var5+var4 操作，方法返回值为 true，取反之后是 false，退出循环，将更新后的值 var5 写回主内存。（由于该方法对标的是 i++，所以返回的是 i 而不是加一后的 i）
      • 若不相等，方法返回值为 false，取反之后为 true，则循环继续执行。重新从 var1+var2 指向的地址获取主内存中最新的 value 值，重新记录快照值，重新计算，重新判断，直到方法执行结束。
  • 由于 getIntVolatile 和 compareAndSwapInt 都是 Unsafe 类的本地方法，都使用了 CAS 原语，所以执行时是具有原子性的，但是在这两个方法之间的时刻，可能会 CPU 资源会被其它线程占用。

一句话总结

​ atomicInteger#getAndIncrement 方法就是 20000 个线程执行 num++，但是每次执行完要写会主内存时要先经过 CAS 校验，校验成功了就写回去，校验不成功则重写读取主内存中的值，重新执行 num++，重写写回主内存，反复执行，直到成功为止。

CAS 缺点

  1. 循环时间长开销大

    ​ 如果 CAS 失败（数据从线程本地内存写回主内存时一直校验不成功），就会一直进行尝试（自旋）。如果 CAS 长时间一直不成功，可能会给 CPU 带来很大的开销。

  2. 只能保证一个共享变量的原子操作（源码中就只有一个）

    ​ 当对一个共享变量执行操作时，可以使用 CAS 循环的方式来保证变量的原子性。但是，对于多个共享变量进行操作时，CAS 循环就无法保证操作的原子性，这个时候只能用锁来保证原子性。

  3. CAS 会导致 ABA 问题

CAS 算法只关注头和尾，只要头和尾是正确就判断为真，但是中间数据可能被改过多次。正是因为这样，才会出现由于时间差引起的 ABA 问题。

是什么？

ABA 问题是什么？

​ ABA 问题是 CAS 算法的遗留问题。

​ CAS 算法的一个重要前提是需要取出主内存中某时刻的数据并在当前时刻比较并替换，由于当前时刻与某一时刻存在时间差，这个时间差就会导致数据的变化。

​ 比如说一个线程 T1 从主内存位置 V 取出 A，这时候另一个线程 T2 也从主内存中取出 A，并且线程 T2 进行了一些操作将值变成了 B，然后线程 T2 又将 V 位置的数据变成 A，这时候线程 T1 进行 CAS 操作时发现内存中仍然是 A，然后线程 T1 操作成功。尽管线程 T1 的 CAS 操作成功，但不代表这个过程是没有问题的。

如何解决 ABA 问题？

时间戳的原子引用

​ CAS 算法中，每次线程从本地内存将数据写回主内存中时，只比较线程的期望值与主内存中的当前值是否相同，而不关心主内存中的当前值是否与线程的期望值是不是同一个值，所以会存在 ABA 问题。

​ 但是如果我们给每个值都加一个版本号，在线程从本地内存将数据写回主内存时除了比较数据值还要额外增加版本号的比较，就可以解决 ABA 问题了。

​ 我们来看一个简单的加了时间戳原子引用的例子：

0s 2s~4s 4s~6s 10s
T1 100（1） 100（1） -> 101（2） 101（2） 101（2）->100（3） 100（3）
T2 100（1） 100（1） -> 990（2） 100(1)？100（3）
  1. 线程 T1、T2 并发执行，都得到了主内存中变量 100（1） 的副本拷贝，并且在拷贝数据时同时将版本号也拷贝到本地内存中。
  2. 由于 T2 执行时间较长（假设为 10s），T1 执行时间较短（假设每次为 2s），在 T1 执行的时间段内，T2 已经完成了：
    • 将 100（1）修改为101（2），更新版本号，写回主内存中；
    • 再从主内存中将 101（2）读取到本地内存中；
    • 将 101（2）修改为 100（3），更新版本号，写回主内存中。
  3. 又过了 2s ，T2 线程终于执行完毕了，将100（1）更新为 999（2），准备将 999（2）写回主内存中。由于当前数据是加了时间戳原子引用，所以写回主内存前需要经过 CAS 验证（包含时间戳的 CAS 验证），即比较 100（1）和 100（3），发现这时候虽然这两个变量值相等，但是版本号却不同，也即期望版本号与主内存中当前数据的版本号不同，所以验证不能通过，即写回主内存的操作失败，这个线程只能再重新读取主内存中的最新值重新在本地内存中计算了。这就可以很好的解决 ABA 问题。

现代计算机内存模型

​ 早期计算机中 CPU 和内存的速度是差不多的，CPU 执行指令的速度与内存读写数据的速度基本可以达到一种平衡。但是在现代计算机中，CPU 执行指令的速度要远高于内存的存取速度，所以现代计算机系统一般都会加入一层读写速度尽可能接近 CPU 运算速度的高速缓存（Cache）来作为内存和处理器之间的缓冲。将运算需要使用到的数据复制到缓存中，让运算能快速进行，当运算结束后再从缓存同步回内存之中，这样处理器就无需等待缓慢的内存读写了。

为什么不把数据直接存在 CPU 中呢？这样读写速度岂不是更快了？

  • 现代计算机中，硬件性能的速度从高到低为：CPU > Cache > 内存 > 硬盘。而在同等价值下，内存大小则是刚好反过来的，也就是说，运算速度越快，可存储的数据量就越小。
  • 所以说使用 CPU 来存储程序运行时需要使用到的所有数据是不现实的，在计算机系统中，CPU 主要用于数据计算，其它硬件则用于数据存取

磁盘中的数据什么时候加载到高速缓存（Cache）中呢？

  1. 程序开始运行的时候，需要把运行时需要使用到的数据加载到内存中。在这个阶段就可以把一些高频的、可能会经常用到的数据（如加减乘除等运算符）直接加载到高速缓存中。
  2. 程序在运行过程中，如果发现有一部分数据是被反复使用的（如通过阈值来筛选高频出现的数据），就可以把这一部分数据在运行期再加载到高速缓存中。
  3. 高速缓存中放的只是主内存中的一部分高频使用的数据，并不能把全部数据都复制到高速缓存中，但即使这样，也大大增加了 CPU的利用率。

缓存一致性问题

​ 绝大多数的运算任务都不可能只靠处理器“计算”就能完成，处理器至少要与内存交互，如读取内存数据、存储运算结果等，这个 I/O 是很难消除的。即程序运行时 CPU 需要经历 从内存取数据 --> CPU 运算 --> 将运算结果写回内存，由于 CPU 的运算速度远快于内存的读写速度，就会出现这样一种情况，内存花了 10s 才将要运算的数据传递到 CPU 中，但是 CPU 只用了 1s 就计算完了，那么中间的 9s CPU 就一直没有活干。写出到内存中也存在类似的情况。基于高速缓存的存储很好地解决了处理器与内存的速度矛盾，但是引入了一个新的问题：缓存一致性。

​ 在多处理器系统中，每个处理器都有自己的高速缓存，而它们又共享同一主内存（物理意义上的内存，可以看做是内存条）（比如 4 个 CPU，一根内存条）。当多个处理器的运算任务都涉及同一块主内存区域时，将可能导致各自的缓存数据不一致。举例说明就是某个变量在多个 CPU 之间共享，并且这些 CPU 都对这个变量进行了运算，那么当缓存数据同步回主内存时以谁的缓存数据为主呢？

​ 为了解决缓存一致性的问题，需要各个处理器访问缓存时都要遵循一些协议，在读写的时候要根据协议来进行操作。

JMM 概述

JMM；Java Memory Model；Java内存模型

​ JMM 就是 Java 内存模型，这个模型要符合几个规范，符合这些规范的程序在多线程环境下的安全性能得到较高的保障。但这些规范并不强制所有程序都必须遵循，但是不遵循的话在多线程环境下的隐患就会比较多。

什么是 Java 内存模型？

​ Java 内存模型就是按照功能将内存划分为几个不同的区域，这些区域在内存中不是真实存在的，而是为了方便理解将某个（或某些）真实的内存区域抽象成了概念上不同的区域。

​ 其实 JVM 也是一种内存模型，因为真实的物理内存空间并未划分什么栈空间、堆空间、方法区等，只不过 JVM 在实现时利用一部分物理空间存储具有相似性的数据，所以才把内存抽象成一块一块不同的区域。

​ 从抽象的角度看，JMM 定义了线程和主内存之间的抽象关系：线程之间共享的变量存储在主内存（Main Memory）中，每个线程都有一个私有的本地内存（Local Memory），本地内存中存储了该线程读/写共享变量的副本。

​ 本地内存是 JMM 的一个抽象概念，并不真实存在，它由缓存、写缓冲区、寄存器以及其它的编译期优化共同实现的。它描述的是一组规则或规范，通过这组规范定义了程序中各个变量（包括实例字段、静态字段和构成数组对象的元素）的访问方式。

JMM 工作原理

​ 上面我们讲了：JMM 定义了线程与主内存之间的抽象关系。

​ 在一个运行的 Java 程序中，JVM 是一个进程，运行时包含多个线程，而每个线程创建时 JVM 都会为其创建一个工作内存（栈空间，包括 Java虚拟机栈和本地方法栈），工作内存是每个线程私有的数据区域，而 JMM 中规定所有变量都要存储在主内存（堆或方法区），主内存是共享内存区域，所有线程都可以访问，但 线程对变量的操作（读取、赋值等）必须要在工作内存中进行，首先要将变量从主内存拷贝到自己的工作内存空间，然后对变量进行操作，操作完成后再将变量写回主内存，而不能直接操作主内存中的变量，各个线程中的工作内存中存储着主内存中的变量副本拷贝，因此不同的线程间无法访问对方的工作内存。若多个线程间想要通信，必须通过主内存来完成。其简要访问过程如下图：

JMM 三大特性

  1. 可见性
  2. 原子性
  3. 有序性

​ JMM 三大特性是大多数多线程开发需要遵守的规定。但是 volatile 只支持可见性、有序性（禁止指令重排），而不支持原子性。

可见性

​ 各个线程对主内存中共享变量的计算都是将主内存中的共享变量拷贝到自己的工作内存进行运算然后再写回到主内存中的。

​ 这就可能会造成一个线程 A 修改了共享变量 X 的值但还未写回主内存时，另一个线程 B 又对主内存中同一个共享变量 X 进行操作，但此时 A 线程工作内存中的共享变量 X 对线程 B 来说并不可见。这种工作内存与主内存同步延迟现象造成了可见性问题。

原子性

什么是原子性？

  •      不可分割性，完整性，即某个线程正在做某个具体业务时，中间不可以被加塞或者被分隔，需要整体完整。

  •      要么同时成功，要么同时失败。有点类似于关系型数据库中的事务。

有序性

JMM 关于同步的规定：

  1. 线程加锁前，必须把共享变量的值刷新回主内存；
  2. 线程加锁前，必须读取主内存的最新值到自己的工作内存；
  3. 加锁解锁是同一把锁。

volatile 关键字

  1. volatile 是 Java 提供的轻量级的同步机制（乞丐版的 syschorcize）
  2. volatile 三大特性
    • 保证可见性
    • 不保证原子性
    • 防止指令重排

1 保证可见性

​ volatile 能保证可见性是指当一个线程运行时能感知到另一个线程对共享变量做出的改变。如果两个线程不是同时执行的，而是依次交替执行，因为每次本地内存中对共享变量的改变都会写会到主内存中，另一个线程接着再取出来的是更改过的共享变量。这就相当于是一个单线程了。

​ 简单点就是说，A、B两个线程并发执行，B 线程正在用着 num，并且可能还要继续用一段时间，这时候线程 A 修改了 num 的值，B线程马上就知道了，这叫做线程间变量的可见性。（用着用着 num 突然变了）

​ 而 A、B 两个线程先后被创建，然后开始并发执行，如果B线程在创建之前A线程就已经修改完 num 的值并将其写回主内存了，那么B线程得到的 num 变量的副本自然也是被 A 线程修改后的 num。

​ 由于我们上面这段代码非常简单，在 main 方法中有创建 A、B 两个线程的操作，先创建A，再创建B。但是由于A线程执行的时间太短了，所以在线程B创建之前 A线程已经执行完了 method1 方法，并且把修改后的 flag 重写回了主内存。这时B线程才被创建，所以 B 线程创建时从主内存中复制过去的变量 flag 已经是修改过的值了，即flag==true，所以自然不会进入死循环了。

​ 如果想要验证不加 volatile 就不具备可见性，可以让线程 A 先睡眠 100ms，让线程B先执行 method2 方法进入死循环，再让线程A去修改 flag 的值，发现程序不能结束，这也就验证了此时线程间确实是不具备可见性的。

上面这段代码在运行时，存在两个线程 child 和 main，一个保存在主内存中的变量 flag。

  1. 由于 main 线程在创建子线程后会睡眠100ms，所以 child 线程开始执行，检测到 flag == true，child 线程进入死循环中。
  2. mian 线程在睡醒后继续执行，修改 flag=false并在控制台上输出指定内容，然后主线程结束。
  3. 但这是我们会发现程序并没有结束，因为 child 线程仍处于死循环状态。也就是说main 线程修改全局变量 flag=false ，并不能被 child 线程感知到。

但是如果我们将 volatile 关键字的注释打开，代码的执行流程就变成了：

  1. child 线程开始执行，检测到 flag == true，进入死循环中。
  2. mian 线程在睡醒后继续执行，修改 flag = false并在控制台上输出指定内容，然后主线程结束。
  3. 但是此时 child 线程监听了到主线程对于 flag 变量的修改，即在 child 线程的感知中 flag == false 了，所以 child 线程退出死循环，在控制台上打印指定内容，child 线程结束。
  4. 整个 JVM 进程结束。

总结：

上面的代码可以证明：volatile 关键字可以保证多个线程之间共享变量的可见性。即共享变量在某个线程中的本地内存中发生了变化，其它的线程也可以监听到这种变化！当然这种监听的底层原理还是通过主内存作为传输纽带进行交互的，即单个线程在本地内存中对共享变量的操作最终还是要写回到主内存中，再由主内存将这种变化通知到其它所有使用这个共享变量的线程。

为什么说 volatile 不能保证原子性？

因为在 MyData#autoIncrement 方法中，可能存在多个线程同时执行了 num++ 操作。

num++ 这一行代码的实际操作可分为三个步骤：读 --> 改 --> 写

  1. 读：首先从主内存中拿到 num 的值，复制一份到自己的工作内存中；[getfield #2 <com/juc/day01/Test.n>]
  2. 改：在自己的工作内存中对 num 进行自增操作；[iadd]
  3. 写：将更改后的 num 写回到主内存中去。[putfield #2 <com/juc/day01/Test.n>]

原子性就是对 num 的操作，即 “读 –> 改 –> 写” 这三个步骤是不可分割的，而 volatile 不能保证原子性。

​ 在保证原子性的情况下 num++ 的执行流程应该是：假设线程 A、B、C 并发执行，线程 A 拿到了 num=0，复制副本后在自己的工作空间中将其更新为 num=1，然后再将其写回到主内存中。这时候其它线程再去对 num 进行操作，这是一个完整的具备原子性的操作流程。

​ 而在不能保证原子性的情况下则会出现，线程 A、B、C 同时拿到了主内存中的值 num=0，并在各自的内存空间将 num 更新为 1。接下来三个线程需要将更新后的值写回主内存中，假设 A 先写回主内存，主内存中的共享变量改变后需要通知其它线程，但是在这个通知还未发送到 B、C 线程的时候，B、C 线程就已经把写回主内存的操作都做完了，那么此时主内存中的值仍然会是 1，最后一个写入到主内存的 1 会覆盖前面两个写入到主内存的 1。而我们知道实际的运算结果本应该是 3。所以说在不能保证原子性的前提下，上述代码执行的结果大概率会小于 20000。

3 有序性（指令重排）

计算机在执行程序时，为了提高性能，编译器和处理器往往会对指令做重排。指令重排会造成程序员写的代码的执行的顺序与代码真正在底层执行的顺序是有偏差的。

多线程环境中线程交替执行，由于指令优化重排的存在，多个线程中使用的变量能否保证一致性是无法确定的，程序执行的结果也就无法预测了。

Java 创建对象的过程可分为三个部分：

  1. 在堆中给将要创建的对象分配一块内存空间，并赋初始零值。
  2. 根据对象的构造函数进行初始化（显式赋值）。
  3. 在栈中建立指针与对象的对应关系。

​ Java 创建对象的每一步都需要 CPU 提供相应的指令，且对象数据保存在堆内存中，CPU 需要从内存中去读取这部分数据并拿回来运算，这就有可能会发生指令重排。

​ 步骤2 和 步骤3 不存在数据依赖关系，所以可能会存在指令重排，而且无论重排前还是重排后程序的执行结果在单线程中并没有改变，因此这种重排是允许的。

  1. minCapacity 是新增元素后需要的最小容量

  3. 由于在上一个方法中已经知道原来的数组已经放不下新增的元素了，所以这里首先要进行扩容。oldCapacity >> 1 是位运算的右移操作，右移一位相当于除以 2，所以这句代码就等于 int newCapacity = oldCapacity + oldCapacity/2，即容量扩大为原来的 1.5 倍。

ArrayList 线程不安全

ArrayList 线程不安全案例

根据上面几部分的学习，我们已经知道了 ArrayList#add 方法在写入数据之前，会先确定要写入位置的下标。如果多个线程同时向同一个 list 集合中写入数据，就可能存在多个线程同时取得了同一个下标的写入权，如果不加以限制，就可能会造成如下情况。即一个线程写的值覆盖了另一个线程写的值。

甚至可能会出现并发修改异常：java.util.ConcurrentModificationException

既然 ArrayList#add() 是线程不安全的，为了解决线程安全问题，我们应该怎么做呢？——加锁！！！

  • 查看 Vector#add 的源码，可以看到该方法被 synchronized 修饰，是线程安全的，但是这样一来就会严重影响高并环境下的程序执行速度了。
  • 我们可以看到该方法在 JDK1.2 之后就被声明弃用了。在早期的 JDK 时期，由于硬件设备的落后，单线程或少线程的程序是主流，加锁的 add 方法能在不太影响程序执行速度的情况下加强程序的安全性，性价比还是很高的。但是在当前的硬件环境下，服务器端特别是电商项目大多都需要高并发的支持，这时再使用 Vector#add 方法对性能的影响就很大了。

线程安全的代码

写时复制技术

顾名思义，“写时复制”就是写的时候才复制，也就是说“读的时候不赋值”。我们都知道，多个线程读同一份资源不会有线程安全的问题，只有多个线程对同一份资源进行写操作的时候才可能出现线程安全问题。一般的加锁机制（重锁）不会区分读还是写，只要是某一资源被上锁了，不管是读还是写都需要进行资源的争夺。而如果某份资源在大多数情况下都是多线程读，只有少数情况下才会进行多线程写的话，使用不区分读写的锁显然对性能的浪费很大。而如果将读和写的操作分开，在某些情况下反而能得到更好的性能。写时复制技术就是这样一种技术。

写时复制技术用到了读写分离的思想。

读的时候共享，写的时候复制一份数据的副本，写好以后，覆盖原先的版本。

放在本例中就是，线程 A、B、C 同时执行 list.add() 操作向集合中添加元素。

  1. 如果线程 A、B、C 要读 list 中的数据，可以任意读，无锁。
  2. 如果线程 B 要修改 list 中的数据，如调用 add() 方法添加元素。不好意思，不能直接在原 list 的基础上添加，线程 B 需要将原来的 list 集合的内容复制一份，将要添加的数据加入到 list 副本中。如果添加成功了，再将指向原 list 的指针指向新的 list，这就完成了一次集合中元素的添加。
