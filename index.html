公平锁 & 非公平锁

是什么

  • 公平锁：

指多个线程按照申请锁的顺序来获取锁，即谁先申请锁谁就优先获得锁。类似于生活中的排队，先到先得。

  • 非公平锁：

只多个线程获得锁的顺序并不是按照申请锁的顺序，有可能后申请锁的线程比先申请锁的线程优先获得锁。在高并发情况下，有可能造成优先级反转或者饥饿现象。

  • 优先级反转现象：就是后申请锁的线程比先申请锁的线程优先获得锁的现象，后申请锁的线程本该是一个低优先级的线程，但是出现了优先级反转，它就提前获得了锁对象。（类似于插队）

  • 饥饿现象：饥饿现象是指如果一直存在插队的线程获得锁，就会导致本该获得锁的线程一直处于等待状态，这就叫做饥饿状态。

二者区别

  • 公平锁

在并发环境中，每个线程在争夺公平锁时会先查看此锁对象维护的等待队列，如果为空，或者当前线程是等待队列中的第一个，才占有锁，否则就会加入到等待队列中，按照先进先出的规则等到自己获取锁。

  • 非公平锁

如果锁对象是一个非公平锁，在并发环境中，每当有新线程进入争夺锁对象时，这个线程会直接去尝试占有锁，如果尝试失败，才会再采用类似公平锁的方式，让线程去锁对象维护的等待队列中排队。

在这种情况下，存在这样一种情况：线程 A 是当前持有锁的对象，线程 A 释放锁之后，按照优先级顺序本该线程 B 得到这个锁对象，但是此时有一个新线程 Z 进来了，这个 Z 不管三七二十一首先就去尝试占有锁了，如果这时候刚好处于 A 释放完锁而 B 有没有获取到锁的时间段，那么 Z 就会直接占有锁，而 B 获取不到锁后，只能继续在等待队列中排队了。这就是 B 线程被插队了。

AtomicInteger 为什么要用 CAS 而不是 synchronized ？

CAS 是什么

CAS 是一种思想，可以保证在多线程环境下，不加锁就能保证对象的一致性（同步性）。

  • 在 JDK5 之前，Java 语言是靠 synchronized 关键字来保证同步的，也就是加锁。锁机制在同一时间段只允许一个线程来修改变量，一致性得到了保障，但并发性下降；
  • CAS 在不加锁的情况下保证了变量的一致性，并发性高。

独占锁是一种悲观锁，synchronized 就是一种独占锁，会导致其它所有需要该锁的线程挂起，等待有锁的线程释放锁。

另一种更加有效的锁就是乐观锁，所谓乐观锁就是，每次不加锁而是假设在没有冲突的情况下去完成某项工作，如果最后因发生冲突失败就重试，直到成功为止，乐观锁用到的机制就是 CAS。

CAS；Compare And Swap；比较并交换

​ 在线程从本地内存向主内存写回数据时，如果线程的期望值与主内存的真实值相等，则执行更新（交换）操作；如果线程的期望值与主内存的真实值不相等，不执行更新操作。

是什么

​ CAS 不是真正的加锁，也就是说允许多个线程同时访问一个变量。在 JMM 内存模型中我们已经了解到，线程在工作时不会直接操作主内存中的数据，而是会将主内存中的数据拷贝回自己的本地内存中后再进行操作，操作完成之后再将数据写回主内存中。

​ 原子性不安全的问题就是：根据 JMM 模型，每个线程操作主内存中的变量都需要经历 从主内存中复制到本地内存 -->> 本地内存中对变量进行运算 -->> 将运算结果写回主内存中 三个步骤。原子性就是这三个步骤读、更新、写回是一体的、不可分割的，要么全部执行成功，要么全部执行失败。如果原子性不能得到保证，在多个线程同时操作一份本地变量 X 时，就会出现 T1 线程在经过读、更新后被挂起，然后 T2 线程完整的执行了 读、更新、写回 三个步骤，修改了主内存中的变量 X 为 Y。这时候 T1 线程继续执行，这时候写回主内存的数据是错误的，因为它没有基于主内存中最新的值来进行计算。

​ CAS 思想要求每个线程在拷贝主内存中的数据时在本地内存中保存一份主内存数据的快照，在本地内存中执行完运算任务要写回主内存时，首先会拿着这个快照与当前时刻主内存中数据进行对比，如果快照数据与主内存当前数据一致，则校验成功，再将线程本地内存中的数据写回主内存中；反之，如果校验，则不会将本地内存数据写回主内存。而是让线程重新获取主内存中的数据，重新运算，重新判断是否可以写回主内存中，直到 CAS 校验成功，将本地内存数据写回主内存中为止。

  • 线程的期望值
  • 主内存中的真实值

​ 在 CAS 的具体操作中，我们习惯将快照值称为线程的期望值，因为多线程中每个线程肯定期望自己在本地内存中运算的这段时间内没有其它线程去更新主内存中的值，这样就不会产生线程不一致。

​ 但实际上在多线程环境下，由于多个线程操作同一变量，且每个线程执行的时间长短不一致，时间较短的线程肯定会先于时间较长的线程修改主内存中的共享变量，当长时间的线程执行完线程任务准备将更新后的数据写回主内存中时，这一时刻主内存中的共享变量可能发生了改变，也可能没发生改变，这个值就叫做主内存中的真实值。

​ CAS 就是当主内存中的真实值和线程的期望值相同时，才执行交换（即将线程本地内存中值写回主内存）动作；如果真实值和期望值不相同，则拒绝执行交换操作。

​ 这就类似于 Git 或 Svn 中代码的提交，多个人协同办公就相当于多线程，假设最初的时候每个人都拿到了 1.0 版本的代码，之后在这个版本上进行迭代更新，小A每工作一小时就提交一次代码，而小B则是每天下班之前才提交一次代码。当第一天工作结束后，小B直接提交代码时就会报错，因为他没有兼容最新版的代码，只有在拉取最新的代码、并检查与自己的代码无冲突时，它才可以进行提交。

AtomicInteger 类

  1. AtomicInteger 是一个 Integer 类型的包装类，构造方法中接收 initialValue 值后用 volatile 修饰该值，保证了多线程操作时改变量在线程间的可见性。

  2. valueOffset 表示当前对象中 value 值的内存偏移量，通过 unsafe.objectFieldOffset 方法获取，该方法是一个本地方法，底层是用 C 写的。

Usafe 类

​ 用 Java 语言实现的方法无法直接访问底层的操作系统，需要通过本地（Native）方法来访问。

​ sun.misc.Unsafe 类是 CAS 的核心类，该类中含有大量的本地方法，可以可以像 C 的指针一样直接操作内存，直接从操作系统层面操作指定内存位置的数据。

​ Java 中 CAS 操作的执行依赖于 Unsafe 类中的方法。Unsafe 类中大多数的方法都是 native 修饰的，可以直接调用操作系统的底层资源执行相应的任务。

​ Unsafe 类中的本地方法都具有原子性，因为其底层是采用 CAS 并发原语实现的。

CAS 并发原语

​ CAS（Compare-And-Swap）是一条 CPU 并发原语。它的功能是判断内存中某个位置的值是否为预期值，如果是则更改为新的值，这个过程是原子的。

​ CAS 并发原语在 Java 中的体现就是 sun.misc.Unsafe 类中的各个本地方法都具有原子性。调用 Unsafe 类的中的本地方法（CAS 方法），JVM 会帮我们实现出 CAS 汇编指令。CAS 原语能保证原子性则完全依赖于硬件的功能。再次强调，由于 CAS 是一种系统原语，具有原子性，所以不会造成多线程下数据不一致的问题。

原语是由若干条指令组成的、用于完成某个功能的一组指令集，这组指令集的执行顺序是确定的，且在执行过程中不允许被中断，具有原子性。可以直接将原语视为一条大的、执行时间较长的指令。

系统原语就是操作系统把一些常用的原语直接实现在硬件上，以便遇到时方便调用。

atomicInteger#getAndIncrement

​ 在单线程环境下，getAndIncrement() 与 i++ 的作用是相同的，但是由于 i++ 不能保证原子性，在多线程环境下继续使用 i++ 存在线程安全问题。

​ 而 atomicInteger#getAndIncrement 可以实现多线程环境下，不用加锁就可以保证线程的原子性，这使用到了 CAS 的思想，其具体的底层源码是怎样的呢？

  1. new AtomicInteger(100) 这句代码会在主内存（堆）中创建出一个 volatile 修饰的整型变量 volatile int value = 100

  2. 由于此时还运行着一个 main 线程，所以 mian 线程会将主内存中的数据拷贝一份到自己的工作内存中。

  3. 继续调用 atomicInteger.getAndIncrement() 方法，底层调用的是 unsafe.getAndAddInt(this, valueOffset, 1) 方法。

    • this 代表当前对象的引用，即当前对象在内存中的地址；

    • valueOffset 代表当前（atomicInteger）对象中 value 变量的内存偏移量。该偏移量通过 Unsafe 类中的一个本地方法获得。

      通过 this + valueOffSet（对象地址 + 对象中 value 变量的偏移地址）的组合可以在内存中定位到唯一一个变量的地址。

  4. 继续向下调用 Unsafe#getAndAddInt 方法，该方法接收三个参数，当前对象的内存地址（var1）、当前对象中 value 变量的内存偏移量（var2），value 将要增加的值（var4）

    • var5 = this.getIntVolatile(var1, var2); 是 Unsafe 类中的一个本地方法（具有原子性），作用是通过当前对象的内存地址和对象中value的偏移量唯一定位到一个值，所以此时 var5 的值就是线程从主内存中拷贝过来的快照值。
    • this.compareAndSwapInt(var1, var2, var5, var5 + var4) 是 Unsafe 类中的另一个本地方法（具有原子性）。
    • 上图中黄色字体前后都是 Unsafe 类中的本地方法，编译成的 CAS 原语具有原子性，所以在这两个方法中不会发生线程切换的情况。而在这两个方法执行的间隙则是可能会发生线程切换的，假设此时发生了线程切换，由另一个线程执行了 3s 再切回来。
    • 接下来要执行 compareAndSwapInt 方法了，该方法执行过程中不能被其它线程打断，
      • 该方法首先根据 var1 + var2 得到主内存中当前时刻 value 的值；
      • var5 是线程的本地内存在创建快照的时刻主内存中 value 的值。
      • 比较这两个值是否相等，若相等，则执行 var5+var4 操作，方法返回值为 true，取反之后是 false，退出循环，将更新后的值 var5 写回主内存。（由于该方法对标的是 i++，所以返回的是 i 而不是加一后的 i）
      • 若不相等，方法返回值为 false，取反之后为 true，则循环继续执行。重新从 var1+var2 指向的地址获取主内存中最新的 value 值，重新记录快照值，重新计算，重新判断，直到方法执行结束。
  • 由于 getIntVolatile 和 compareAndSwapInt 都是 Unsafe 类的本地方法，都使用了 CAS 原语，所以执行时是具有原子性的，但是在这两个方法之间的时刻，可能会 CPU 资源会被其它线程占用。

一句话总结

​ atomicInteger#getAndIncrement 方法就是 20000 个线程执行 num++，但是每次执行完要写会主内存时要先经过 CAS 校验，校验成功了就写回去，校验不成功则重写读取主内存中的值，重新执行 num++，重写写回主内存，反复执行，直到成功为止。

CAS 缺点

  1. 循环时间长开销大

    ​ 如果 CAS 失败（数据从线程本地内存写回主内存时一直校验不成功），就会一直进行尝试（自旋）。如果 CAS 长时间一直不成功，可能会给 CPU 带来很大的开销。

  2. 只能保证一个共享变量的原子操作（源码中就只有一个）

    ​ 当对一个共享变量执行操作时，可以使用 CAS 循环的方式来保证变量的原子性。但是，对于多个共享变量进行操作时，CAS 循环就无法保证操作的原子性，这个时候只能用锁来保证原子性。

  3. CAS 会导致 ABA 问题

CAS 算法只关注头和尾，只要头和尾是正确就判断为真，但是中间数据可能被改过多次。正是因为这样，才会出现由于时间差引起的 ABA 问题。

ABA 问题是什么？

​ ABA 问题是 CAS 算法的遗留问题。

​ CAS 算法的一个重要前提是需要取出主内存中某时刻的数据并在当前时刻比较并替换，由于当前时刻与某一时刻存在时间差，这个时间差就会导致数据的变化。

​ 比如说一个线程 T1 从主内存位置 V 取出 A，这时候另一个线程 T2 也从主内存中取出 A，并且线程 T2 进行了一些操作将值变成了 B，然后线程 T2 又将 V 位置的数据变成 A，这时候线程 T1 进行 CAS 操作时发现内存中仍然是 A，然后线程 T1 操作成功。尽管线程 T1 的 CAS 操作成功，但不代表这个过程是没有问题的。

如何解决 ABA 问题？

时间戳的原子引用

​ CAS 算法中，每次线程从本地内存将数据写回主内存中时，只比较线程的期望值与主内存中的当前值是否相同，而不关心主内存中的当前值是否与线程的期望值是不是同一个值，所以会存在 ABA 问题。

​ 但是如果我们给每个值都加一个版本号，在线程从本地内存将数据写回主内存时除了比较数据值还要额外增加版本号的比较，就可以解决 ABA 问题了。

​ 我们来看一个简单的加了时间戳原子引用的例子：

0s 2s~4s 4s~6s 10s
T1 100（1） 100（1） -> 101（2） 101（2） 101（2）->100（3） 100（3）
T2 100（1） 100（1） -> 990（2） 100(1)？100（3）
  1. 线程 T1、T2 并发执行，都得到了主内存中变量 100（1） 的副本拷贝，并且在拷贝数据时同时将版本号也拷贝到本地内存中。
  2. 由于 T2 执行时间较长（假设为 10s），T1 执行时间较短（假设每次为 2s），在 T1 执行的时间段内，T2 已经完成了：
    • 将 100（1）修改为101（2），更新版本号，写回主内存中；
    • 再从主内存中将 101（2）读取到本地内存中；
    • 将 101（2）修改为 100（3），更新版本号，写回主内存中。
  3. 又过了 2s ，T2 线程终于执行完毕了，将100（1）更新为 999（2），准备将 999（2）写回主内存中。由于当前数据是加了时间戳原子引用，所以写回主内存前需要经过 CAS 验证（包含时间戳的 CAS 验证），即比较 100（1）和 100（3），发现这时候虽然这两个变量值相等，但是版本号却不同，也即期望版本号与主内存中当前数据的版本号不同，所以验证不能通过，即写回主内存的操作失败，这个线程只能再重新读取主内存中的最新值重新在本地内存中计算了。这就可以很好的解决 ABA 问题。

现代计算机内存模型

​ 早期计算机中 CPU 和内存的速度是差不多的，CPU 执行指令的速度与内存读写数据的速度基本可以达到一种平衡。但是在现代计算机中，CPU 执行指令的速度要远高于内存的存取速度，所以现代计算机系统一般都会加入一层读写速度尽可能接近 CPU 运算速度的高速缓存（Cache）来作为内存和处理器之间的缓冲。将运算需要使用到的数据复制到缓存中，让运算能快速进行，当运算结束后再从缓存同步回内存之中，这样处理器就无需等待缓慢的内存读写了。

为什么不把数据直接存在 CPU 中呢？这样读写速度岂不是更快了？

  • 现代计算机中，硬件性能的速度从高到低为：CPU > Cache > 内存 > 硬盘。而在同等价值下，内存大小则是刚好反过来的，也就是说，运算速度越快，可存储的数据量就越小。
  • 所以说使用 CPU 来存储程序运行时需要使用到的所有数据是不现实的，在计算机系统中，CPU 主要用于数据计算，其它硬件则用于数据存取

磁盘中的数据什么时候加载到高速缓存（Cache）中呢？

  1. 程序开始运行的时候，需要把运行时需要使用到的数据加载到内存中。在这个阶段就可以把一些高频的、可能会经常用到的数据（如加减乘除等运算符）直接加载到高速缓存中。
  2. 程序在运行过程中，如果发现有一部分数据是被反复使用的（如通过阈值来筛选高频出现的数据），就可以把这一部分数据在运行期再加载到高速缓存中。
  3. 高速缓存中放的只是主内存中的一部分高频使用的数据，并不能把全部数据都复制到高速缓存中，但即使这样，也大大增加了 CPU的利用率。

缓存一致性问题

​ 绝大多数的运算任务都不可能只靠处理器“计算”就能完成，处理器至少要与内存交互，如读取内存数据、存储运算结果等，这个 I/O 是很难消除的。即程序运行时 CPU 需要经历 从内存取数据 --> CPU 运算 --> 将运算结果写回内存，由于 CPU 的运算速度远快于内存的读写速度，就会出现这样一种情况，内存花了 10s 才将要运算的数据传递到 CPU 中，但是 CPU 只用了 1s 就计算完了，那么中间的 9s CPU 就一直没有活干。写出到内存中也存在类似的情况。基于高速缓存的存储很好地解决了处理器与内存的速度矛盾，但是引入了一个新的问题：缓存一致性。

​ 在多处理器系统中，每个处理器都有自己的高速缓存，而它们又共享同一主内存（物理意义上的内存，可以看做是内存条）（比如 4 个 CPU，一根内存条）。当多个处理器的运算任务都涉及同一块主内存区域时，将可能导致各自的缓存数据不一致。举例说明就是某个变量在多个 CPU 之间共享，并且这些 CPU 都对这个变量进行了运算，那么当缓存数据同步回主内存时以谁的缓存数据为主呢？

​ 为了解决缓存一致性的问题，需要各个处理器访问缓存时都要遵循一些协议，在读写的时候要根据协议来进行操作。

JMM 概述

JMM；Java Memory Model；Java内存模型

​ JMM 就是 Java 内存模型，这个模型要符合几个规范，符合这些规范的程序在多线程环境下的安全性能得到较高的保障。但这些规范并不强制所有程序都必须遵循，但是不遵循的话在多线程环境下的隐患就会比较多。

什么是 Java 内存模型？

​ Java 内存模型就是按照功能将内存划分为几个不同的区域，这些区域在内存中不是真实存在的，而是为了方便理解将某个（或某些）真实的内存区域抽象成了概念上不同的区域。

​ 其实 JVM 也是一种内存模型，因为真实的物理内存空间并未划分什么栈空间、堆空间、方法区等，只不过 JVM 在实现时利用一部分物理空间存储具有相似性的数据，所以才把内存抽象成一块一块不同的区域。

​ 从抽象的角度看，JMM 定义了线程和主内存之间的抽象关系：线程之间共享的变量存储在主内存（Main Memory）中，每个线程都有一个私有的本地内存（Local Memory），本地内存中存储了该线程读/写共享变量的副本。

​ 本地内存是 JMM 的一个抽象概念，并不真实存在，它由缓存、写缓冲区、寄存器以及其它的编译期优化共同实现的。它描述的是一组规则或规范，通过这组规范定义了程序中各个变量（包括实例字段、静态字段和构成数组对象的元素）的访问方式。

JMM 工作原理

​ 上面我们讲了：JMM 定义了线程与主内存之间的抽象关系。

​ 在一个运行的 Java 程序中，JVM 是一个进程，运行时包含多个线程，而每个线程创建时 JVM 都会为其创建一个工作内存（栈空间，包括 Java虚拟机栈和本地方法栈），工作内存是每个线程私有的数据区域，而 JMM 中规定所有变量都要存储在主内存（堆或方法区），主内存是共享内存区域，所有线程都可以访问，但 线程对变量的操作（读取、赋值等）必须要在工作内存中进行，首先要将变量从主内存拷贝到自己的工作内存空间，然后对变量进行操作，操作完成后再将变量写回主内存，而不能直接操作主内存中的变量，各个线程中的工作内存中存储着主内存中的变量副本拷贝，因此不同的线程间无法访问对方的工作内存。若多个线程间想要通信，必须通过主内存来完成。其简要访问过程如下图：

JMM 三大特性

  1. 可见性
  2. 原子性
  3. 有序性

​ JMM 三大特性是大多数多线程开发需要遵守的规定。但是 volatile 只支持可见性、有序性（禁止指令重排），而不支持原子性。

可见性

​ 各个线程对主内存中共享变量的计算都是将主内存中的共享变量拷贝到自己的工作内存进行运算然后再写回到主内存中的。

​ 这就可能会造成一个线程 A 修改了共享变量 X 的值但还未写回主内存时，另一个线程 B 又对主内存中同一个共享变量 X 进行操作，但此时 A 线程工作内存中的共享变量 X 对线程 B 来说并不可见。这种工作内存与主内存同步延迟现象造成了可见性问题。

原子性

什么是原子性？

  •      不可分割性，完整性，即某个线程正在做某个具体业务时，中间不可以被加塞或者被分隔，需要整体完整。

  •      要么同时成功，要么同时失败。有点类似于关系型数据库中的事务。

有序性

  2. 线程加锁前，必须读取主内存的最新值到自己的工作内存；
  3. 加锁解锁是同一把锁。

为什么说 volatile 不能保证原子性？

因为在 MyData#autoIncrement 方法中，可能存在多个线程同时执行了 num++ 操作。

num++ 这一行代码的实际操作可分为三个步骤：读 --> 改 --> 写

  1. 读：首先从主内存中拿到 num 的值，复制一份到自己的工作内存中；[getfield #2 <com/juc/day01/Test.n>]

​ 在保证原子性的情况下 num++ 的执行流程应该是：假设线程 A、B、C 并发执行，线程 A 拿到了 num=0，复制副本后在自己的工作空间中将其更新为 num=1，然后再将其写回到主内存中。这时候其它线程再去对 num 进行操作，这是一个完整的具备原子性的操作流程。

​ 而在不能保证原子性的情况下则会出现，线程 A、B、C 同时拿到了主内存中的值 num=0，并在各自的内存空间将 num 更新为 1。接下来三个线程需要将更新后的值写回主内存中，假设 A 先写回主内存，主内存中的共享变量改变后需要通知其它线程，但是在这个通知还未发送到 B、C 线程的时候，B、C 线程就已经把写回主内存的操作都做完了，那么此时主内存中的值仍然会是 1，最后一个写入到主内存的 1 会覆盖前面两个写入到主内存的 1。而我们知道实际的运算结果本应该是 3。所以说在不能保证原子性的前提下，上述代码执行的结果大概率会小于 20000。

线程安全的代码

写时复制技术

顾名思义，“写时复制”就是写的时候才复制，也就是说“读的时候不赋值”。我们都知道，多个线程读同一份资源不会有线程安全的问题，只有多个线程对同一份资源进行写操作的时候才可能出现线程安全问题。一般的加锁机制（重锁）不会区分读还是写，只要是某一资源被上锁了，不管是读还是写都需要进行资源的争夺。而如果某份资源在大多数情况下都是多线程读，只有少数情况下才会进行多线程写的话，使用不区分读写的锁显然对性能的浪费很大。而如果将读和写的操作分开，在某些情况下反而能得到更好的性能。写时复制技术就是这样一种技术。

写时复制技术用到了读写分离的思想。

读的时候共享，写的时候复制一份数据的副本，写好以后，覆盖原先的版本。

放在本例中就是，线程 A、B、C 同时执行 list.add() 操作向集合中添加元素。

  1. 如果线程 A、B、C 要读 list 中的数据，可以任意读，无锁。
  2. 如果线程 B 要修改 list 中的数据，如调用 add() 方法添加元素。不好意思，不能直接在原 list 的基础上添加，线程 B 需要将原来的 list 集合的内容复制一份，将要添加的数据加入到 list 副本中。如果添加成功了，再将指向原 list 的指针指向新的 list，这就完成了一次集合中元素的添加。

对象锁；全局锁（静态锁/类锁）

synchronized 实现同步的基础：Java 中每一个对象都可以作为锁。具体表现为以下 3 种形式：

  1. 对于同步方法，锁对象是当前实例对象（this）

  2. 对于同步代码块，锁对象是 synchronized 括号里配置的对象

  3. 对于静态同步方法，锁对象是当前类的 Class 对象

锁对象是当前对象this

锁对象是当前对象 this 的情况有 2 种：

  1. 如果对象中存在被 synchronized 修饰的普通成员方法，那么该方法默认的锁对象就是当前对象 this。
  2. 如果对象中存在 synchronized 代码块，也可以让当前对象 this 作为锁对象，但是这不是确定的。因为锁对象可以是任意对象。

锁对象是当前类的 Class 对象

锁对象是当前类的 Class 对象的情况也有 2 种：

  1. 如果类中存在被 static synchronized 修饰的静态成员方法，那么该方法默认的锁对象就是当前类的 Class 对象。
  2. 如果对象中存在 synchronized 代码块，同样也可以在同步代码块中指定锁对象为当前类的 Class 对象，这同样也不是确定的。

synchronized 机制

如果一个方法被 synchronized 修饰，它锁的范围不仅仅是当前方法，而是包含这个方法的整个锁对象。

  • 即如果 synchronized 作用在一个普通成员方法/变量上，synchronized 锁的就是这个 this 对象。
  • 即如果 synchronized 作用在一个静态成员方法/变量上。synchronized 锁的就是这个类的 Class 对象。

在一个对象里面，不管这个对象中有多少个同步方法，只要一个线程先访问了该对象中的任一同步方法，那么它锁的不仅仅是它访问的这一个同步方法，它锁的是该方法所在的整个对象。

一个对象里如果有多个 synchronized 方法，某一时刻，只要有任意一个线程去调用其中任一个 synchronized 方法，其它的线程就只能等待。换句话说就是，某一时刻，只能有唯一一个线程去访问这个对象中的 synchronized 方法。而对于对象中的非加锁方法则无限制。

如果锁对象是当前对象 this，被锁定后，其它线程都不能进入到当前对象的其它 synchronized 方法，但可以进入当前对象的非加锁方法。

如果锁对象是 Class 对象，被锁定后，其它线程就不能进入到该 Class 对象中执行其它的被 synchronized 修饰的方法，同样也可以执行非加锁方法。

同第 7 种情况类似，两个线程使用两个不同的锁对象，不存在竞争关系。发短信的速度快于发邮件，所以发短信的操作先执行。

小结（重要）

加锁机制其实就是一个锁对象争夺的问题。

首先我们要明确锁对象是一个单例，如果一个对象中只含有一个锁对象，并且该对象中所有被 synchronized 修饰的方法用的都是同一把锁，那么显然同一时刻只能有一个线程持有这把锁，其它线程没有锁就只能等待锁被释放后再去争夺了。

在上面举的例子中，每个对象中就只设置了一个锁对象。那么自然该对象中所有被 synchronized 修饰的方法用的就是这把锁，就会出现线程 A 未释放锁前线程 B 不能执行其它被 synchronized 修饰的方法这种情况（因为其它 synchronized 方法执行所需要的锁还是线程 A 正在持有的锁）。

那么如果我们在一个对象中定义多把锁，每个加锁方法（被 synchronized 修饰的方法）用不同的锁。假设为：

  • 定义两把锁 A、B
  • 定义两个加锁的方法 methodA()、methodB()。methodA() 方法用 A 锁，methodB() 方法用 B 锁，

那么会出现一种情况就是：让线程1先被创建，线程2再被创建，执行同一个对象中的两个加锁方法methodA() 和 methodB()，但是线程1 执行方法 A 的时候，线程 B 也能执行方法 B，而不用等到线程 A 归还锁对象后线程 B 再执行方法 B，因为这两个加锁方法虽然在同一个对象中，但是使用的不是一个锁对象。

也就是判断一个对象中的多个方法能否被多个线程同时执行的最根本的依据就是这些方法使用的是不是同一把锁。

多线程间通信

​ 多线程并发执行时，默认情况下 CPU 是随机切换线程的，当我们需要多个线程能有规律的执行任务时，就需要通过线程来协调任务进度。

宗旨：

  1. 高内聚低耦合的情况下，线程 操作 资源类。
  2. 线程执行体内的结构：判断/干活/通知
  3. 防止虚假唤醒

​

​ 在实际的多线程应用中，多个线程不可能仅仅只是卖票（卖票是多个线程操作同一份资源，但执行的动作是相同的，也就是说无论哪个线程先执行对结果都不会影响，即使有一个线程特别牛逼，它一个人把票都卖完了也可以）。

​ 通常情况会是：多个线程会操作同一资源，且操作的方式不相同。典型的就是生产消费线程，一个线程生产，另一个线程消费。这时候如果还是让 CPU 随机调度线程的话，就可能会出现供不应求（一直消费很少生产）或供过于求（一直生产很少消费）的情况，这都是不合适的。为了能让线程能按我们设计的顺序执行（比如生产线程执行一次，消费线程再执行一次，交替循环执行），达到供给平衡，我们就必须通过线程间的通信，通过频繁的挂起/唤醒线程来让线程按指定的顺序执行。

​ 在一个线程执行体内，以生产/消费为例，最核心的代码当然是干活的代码，在生产线程中就是生产产品，在消费线程中就是消费产品。但是我们不能让线程直接去执行干活的代码，如果产品数量为0了，消费线程还去执行的话，显然是不合理的；同理，产品已经多到卖不完了，生产线程还一直生产产品也是不合适的。

​ 在一个标准的线程执行体内，

  1. 第一步代码代码应该是判断。主要是根据实际需求判断当前线程应该进入被挂起还是干活。在生产消费线程中，如果生产线程抢到了锁对象和 CPU 资源，在执行方法的时候首先如果发现当前产品是足够的，它也就不会去干活，而是把自己挂起，让其它线程去干活。消费线程也类似。
  2. 只有判断之后才能开始干活。对于生产线程而言干活就是生产，对于消费线程而言干活就是消费。
  3. 在干完活之后需要通知其它线程。这里的通知其实就是唤醒其它线程，因为在该线程干活之前可能存在为了让出锁和 CPU 资源而让自己进入 Waiting 状态的其它线程，这种线程只有通过 notify 方法才能将其唤醒。这里的通知并不是把修改后的数据通知给其它线程，把修改后的数据通知给其它线程是通过 synchronized 的原子性来保证的。

    • 每次当 add() 方法发现 num 为 1 时，不会干活，直接就把锁对象让出去了，只有 num==0 时，才会干活，进行 +1 操作。
    • 同理，每次当 sub() 方法发现 num==0 时，不会干活，直接就把锁对象让出去了，只有 num==1 时，才会干活，进行 +1 操作。

    这样程序的执行结果永远只会是 0 和 1，且 0、1 交替出现，那么为什么会出现 2 呢？这种情况就叫做虚假唤醒，那么造成虚假唤醒的原因是什么呢？

虚假唤醒原理

  1. 只有两个线程的情况：

在只有两个线程的情况下，线程 A 负责加操作，线程 B 负责减操作。

  1. 当线程 A 获取到锁和 CPU 资源后，开始执行 add 方法。首先要判断当前 num 是否为 0，

    • 如果 num == 0 才进行 +1 操作；
    • 否则，直接释放锁对象，自己进入 Waiting 状态。

  2. 同理，当线程 B 获取到锁和 CPU 资源后，开始执行 sub 方法。首先判断当前 num 是否为 0，、

    • 如果 num == 0 直接释放锁对象，自己进入 Waiting 状态；
    • 否则才进行 -1 操作。

    ​ 但是请注意（以线程 A 进入 Waiting 状态为例）：假设线程 A 进入了 Waiting 状态，那么此时的 num == 1 恒成立，然后线程 B 必然会进入 Runnable 状态，并在随后得到 CPU 资源执行 sub 方法：1、进行 -1 操作 2、唤醒 A 线程。

    ​ 重点来了！！！线程 B 唤醒 A 线程后，B 与 A 此时都处于 Runnable 状态，然后开始竞争锁对象，假设这时候 A 获得了锁对象（当然如果 B 获得了锁对象也没有关系，因为此时 num==0，即使 B 获得了锁对象最终还是会执行 wait 方法归还锁对象），那么线程 A 并不会从头开始重新执行一遍 add 方法，而是直接续着上次执行的位置继续执行。 也就是判读操作在线程睡眠之前已经执行过一次了，线程醒来后就不再判断了，而是直接从上次断开的位置往下走。

​ 在只有 2 个线程的环境中这是 OK 的，因为只有两种状态，不会对结果产生影响。那么如果有 4 个线程呢？我们来分析一下。

  1. 四个线程并发执行的情况：

  1. 初始情况，num == 0，ABCD 四个线程开始抢夺锁对象；

  2. 假设 A+ 线程抢到了锁对象，执行 add 方法，由于num==0，开始干活，更新 num==1，唤醒其它线程，最后释放锁对象。

  3. 接着 C+ 线程抢到了锁对象，执行 add 方法，由于num==1，不干活，进入 Waiting 状态，直接释放锁对象。

  4. 此时 A+ 又抢到了锁对象，执行 add 方法，由于num==1，也不干活，进入 Waiting 状态，直接释放锁对象。

  5. 接下来线程 B- 终于抢到锁对象了。执行 sub 方法，由于num==1，开始干活，更新 num==0，唤醒其它线程，最后释放锁对象。

    请注意：这时候 A+ 和 C+ 线程又重新被唤醒了，进入了 Runnable 状态，ABCD 四个线程又处于同一起跑线上准备抢夺锁对象。

  6. 这时因为 C+ 线程已经等了很久了，所以它又得到了锁对象，执行 add 方法，注意，它是从上一次断开的位置继续执行的，也就是 wait() 方法的后一行，也就是说这次 if 判断不再执行了，它直接就得干活，更新 num==1，然后唤醒其它线程，释放锁对象。

  7. 接着 A+ 线程得到锁对象，执行 add 方法，同样它也会从上一次断开的位置继续执行，没有 if 判断，直接干活。诶，这时候你就会发现，num++ 执行后 num==2，if 判断失效了，这是不对的。我们想要的代码应该是发现 num==1 后就不再 +1 了，而是让 sub 方法去执行 -1 操作。这就是传说中的虚假唤醒原理。

题目：

要求：多线程之间按顺序调用，3个线程按照 A->B->C 的顺序执行。A线程打印5次“AA”，B 线程打印10次“BB”，C线程打印15次“CC”；重复10轮

分解：

  1. A线程打印5次“AA”，A线程进入 Waiting 状态，并通知B线程；
  2. B线程被唤醒，打印10次“BB”，B线程进入 Waiting 状态，并通知C线程；
  3. C线程被唤醒，打印10次“CC”，C线程进入 Waiting 状态，并通知A线程；
  4. 这就是一轮，重复10轮。

分析1：

​ 要想实现三个线程ABC依次执行，可以为每一个线程设置唯一的标志位，假设为：线程A：1；线程B：2；线程C：3。

  1. 假设线程 A 先得到了锁对象，线程A执行完之后，先修改标志位，再将锁抛出；
  2. 这时候按照需求本应该线程B来执行了，但是由于线程A将锁抛出之前唤醒了所有线程，所以此时线程A、B、C会一起争夺锁对象，谁得到锁谁就执行；
  3. 假如 C 得到了锁对象，由于标志位不匹配，还是会将锁抛出，让线程 ABC 重新争夺锁对象；
  4. 我们可以看到，如果线程B优先级贼低的话，就可能会出现AC两人一块玩儿，不带B。当然这种情况一般很少会发生，一般线程等待很长时间后优先级都会提高一点儿。
  5. 反复循环，直到 B 得到锁对象，B 执行完方法后，修改标志位，唤醒其它所有线程，抛出锁对象后。完了，又开始新一轮的ABC抢占锁对象和CPU资源。
  6. 所以如果我们这么设计代码的话，在电脑上正常是可以按需求顺利执行的。因为 ABC 线程的优先级是同一级别的，基本上这3个线程得到锁的概率都差不多。但是每次都是抢占式调度的话，每次让正确的线程得到锁对象的时间都会花费较多的时间。
  7. 但是理论上存在：本该某一个线程干活，但是这个线程就是死活得不到锁对象，锁对象只在其它的线程间流转的情况，这就会导致程序一直停不下来。

    • 首先在局部变量中分配一块空间用于保存变量 j 的值；
    • 再进行等号右边的运算。
      1. 将当前局部变量表中 i 的值 1 压入操作数栈
      2. 局部变量表中的 i 进行自增运算，从 1 变为 2
    • 最后进行等号左边的运算：
      • 将操作数栈中的栈顶元素 1 赋值给局部变量表中的 j ，此时 j = 1。
    • 栈顶元素出栈。
  4. int k = i + ++i * i++
    • 首先在局部变量中分配一块空间用于保存变量 k 的值；
    • 再进行等号右边的运算。
      1. i：将局部变量表中当前 i 的值 2 压入操作数栈；
      2. +：将 + 压入操作数栈
      3. ++i：先将局部变量表中的 i 自增变为 3 ，再压入操作数栈；
      4. *：将 * 压入操作数栈
      5. i++：先将局部变量表中 i 的值 3 压入操作数栈，再让 i 在局部变量表中自增变为 4；
      6. 依次取出栈顶的元素（出栈）：3、*、3 进行乘法运算：3*3=9；
      7. 将临时变量 9 再压入操作数栈；
      8. 依次取出栈顶的元素：9 、 + 、2，进行加法运算：9+2=11
      9. 将临时变量 11 再压入操作数栈中
    • 再进行等到左边的运算：
      • 将栈顶元素的值 11 赋值给局部变量表中的 k ，得到 k=11。

所以最终的输出结果为：i=4 j=1 k=11

2.1 小结

  • 赋值操作=，最后再计算。即先进行 = 右边的运算，再进行 = 左边的运算。
  • = 右边的运算按从左到右的顺序依次压入操作数栈。
  • 实际先算哪个，是按照运算符的优先级来的。
  • 自增、自减都是直接修改局部变量表中变量的值，不经过操作数栈。
  • 在进行最后的赋值操作之前，产生的临时结果也是存储在操作数栈中。
  • i++ 就是先把 i 的值压入操作数栈，再自增；++i 就是先自增，再把自增后的结果压入操作数栈。

2 要点

  1. 怎样保证某个类只能有一个实例？
  1. 将构造方法私有化。
  2. 将实例对象的创建提前到类加载阶段。

分析：

  • 将构造方法私有化，即不对外提供创建该类的实例对象的入口。那么创建实例对象的时机只能提前到类加载阶段。
  • 一个类只能被加载一次，每个类 <clinit>() 方法只会被执行一次，只要在 <clinit>() 方法中执行创建对象的行为，该对象就一定会被创建出来一个，且我们已经关闭了外部创建该对象的入口，那么该对象就有且仅有一个了。
  • <clinit>() 方法只会执行类中的静态变量赋值和静态代码块，所以要创建的对象必须通过一个静态变量来保存。
  1. 该类必须向整个系统提供这个实例。
  1. 直接对外暴露该对象在内存中的地址。
  2. 通过一个静态的 get() 方法获取该对象实例。

3 几种单例模式的常见形式

饿汉式

  • 介绍：不管该单例对象在程序运行中是否会被使用，都会创建该对象。
  • 优点：不存在线程问题。因为创建对象是在类加载阶段进行的，这个阶段不会存在线程安全问题。
  • 几种实现方式：
    1. 直接实例化饿汉式（简单直观）
    2. 枚举式（最简单）
    3. 静态代码块饿汉式（适合较为复杂实例化场景）

懒汉式

  • 介绍：程序运行时不会立即创建对象，知道该单例对象要被使用时才创建对象
  • 几种实现方式：
    1. 线程不安全（适用于单线程）
    2. 线程安全（适用于多线程）
    3. 静态内部类形式（适用于多线程）

小结

  • 如果是饿汉式，枚举形式最简单
  • 如果是懒汉式，静态内部类形式最简单

2.1 单一职责原则

2.1.1 基本介绍

​ 对类来说的，一个类应该只负责一项职责。

依赖倒置原则

基本介绍

依赖倒转原则（Dependence Inversion Principle）是指：

  1. 高层模块不应该依赖低层模块，二者都应该依赖其抽象
  2. 抽象不应该依赖细节，细节应该依赖抽象
  3. 依赖倒置的中心思想是面向接口编程
  4. 依赖倒置原则是基于这样的设计理念：相对于细节的多变性，抽象的东西要稳定得多。以抽象为基础搭建的架构比以细节为基础的架构要稳定的多。在 java 中，抽象指的是接口或抽象类，细节就是具体的实现类。
  5. 使用接口或抽象类的目的是制定好规范，而不涉及任何具体的操作，把展现细节的任务交给他们的实现类去完成

面向对象中的继承性的思考和说明

  1. 继承包含这样一层含义：父类中凡是已经实现好的方法，实际上是在设定规范和契约，虽然它不强制要求所有的子类必须遵循这些契约，但是如果子类对这些 已经实现的方法任意修改，就会对整个继承体系造成破坏。
  2. 继承在给程序设计带来便利的同时，也带来了弊端。比如使用继承会给程序带来侵入性，程序的可移植性降低，增加对象间的耦合性。如果一个类被其他的类所继承，则当这个类需要修改时，必须考虑到所有的子类，并且父类修改后，所有涉及到子类的功能都有可能产生故障。

基本介绍

  1. 里氏替换原则（Liskov Substitution Principle）
  2. 如果对每个类型为 T1 的对象 o1，都有类型为 T2 的对象 o2，使得以 T1 定义的所有程序 P 在所有的对象 o1 都代换成 o2 时，程序 P 的行为没有发生变化，那么类型 T2 是类型 T1 的子类型。换句话说，所有引用基类的地方必须能透明地使用其子类的对象 。
    • 所有引用基类的地方必须能透明地使用其子类的对象：基类就是父类，这句话的意思就是说：在所有使用父类的地方，如果用其子类代替的话不会产生歧义。
  3. 在使用继承时，遵循里氏替换原则，在子类中尽量不要重写父类的方法。
  4. 里氏替换原则告诉我们，继承实际上让两个类耦合性增强了，在适当的情况下，可以通过聚合，组合，依赖来解决问题。

1 什么是通信协议？

通信协议就是通信的双方共同规定的都要实现的标准。

2 TCP 和 UDP 有什么区别 ？TCP 为什么是三次握手，而不是两次？

2.1 TCP 和 UDP 的区别：

  • TCP（Transfer Control Protocol）是一种 面向连接的、可靠的 传输层通信协议。
    • 类似于 打电话，点对点的通信，高可靠的，效率比较低，占用的系统资源比较多
  • UDP（User Datagram Protocol）是一种 无连接的、不可靠的 传输层通信协议
    • 类似于 广播，不需要建立连接，发送方不管接收方有没有准备好，直接发送消息；可以进行广播发送。传输不可靠，有可能丢失消息。效率比较高；协议比较简单，占用的系统资源比较少。

2.2 TCP 为什么是三次握手，而不是两次

答：如果是两次握手，可能会造成连接资源浪费的情况。

网络是一种非常不稳定的资源，TCP 建立连接三次握手，断开连接四次挥手。

假设只有两次握手，

  • 第一次握手：客户端向服务器端发送消息，请求建立连接；
  • 第二次握手：服务端同意/拒绝建立连接，并将同意/拒绝消息发送给客户端；

​ 此时若服务器同意与客户端建立连接，就会将“同意建立连接”的消息发送给客户端并建立服务器与客户端之间的网络通道。由于网络是一种非常不稳定的资源，假设这个在过程中由于网络波动服务器发出的“同意建立连接”消息丢失了，客户端就无法收到服务器“同意建立连接”的消息，也就不会知道其实它与服务器间通信的信道已经建立好了。客户端在这次请求超时后，会做出两种选择：

  1. 重新发送请求与服务器建立连接；
  2. 不再发送请求放弃与服务器建立连接。

​ 假设请求超时后客户端放弃与服务器建立连接：我们可以看到，此时服务器与客户端之间的网络通道其实是存在的，只不过客户端没并不知道这个网络通道的存在，就不会去用这个通道与服务器交互。在无其它操作的情况下，这个网络通道就会一直存在并浪费客户端和服务器的资源。

3 Java 有哪几种 IO 模型 ？有什么区别？

3.1 什么是同步？什么是异步？什么是阻塞？什么是非阻塞？

同步

  • 指的是用户进程触发 IO 操作等待并轮询的去查看 IO 操作是否就绪。

异步

  • 异步指的是用户进程触发IO操作后便开始做自己的事情，而当IO操作已经完成的时候会得到IO完成的通知（异步的特点就是通知）。

阻塞

  • 所谓阻塞方式的意思是指, 当试图对该文件描述符进行读写时, 如果当时没有东西可读,或者暂时不可写, 程序就进入等待状态, 直到有东西可读或者可写为止。

非阻塞

  • 非阻塞状态下, 如果没有东西可读, 或者不可写, 读写函数马上返回, 而不会等待。

同步 和 异步 最大的区别是是否有 通知 。

而 阻塞 和 非阻塞 最大的区别就是是否 等待 。

  1. 同步阻塞IO（BIO）
    • 同步并阻塞，服务器实现模式为一个连接一个线程，即客户端有连接请求时服务器端就需要启动一个线程进行处理，如果这个连接不做任何事情会造成不必要的线程开销，当然可以通过线程池机制改善。
  2. 同步非阻塞IO（NIO）
    • 同步非阻塞，服务器实现模式为一个请求一个线程，即客户端发送的连接请求都会注册到多路复用器上，多路复用器轮询到连接有I/O请求时才启动一个线程进行处理。用户进程也需要时不时的询问IO操作是否就绪，这就要求用户进程不停的去询问。
  3. 异步阻塞IO（NIO）
    • 此种方式下是指应用发起一个IO操作以后，不等待内核IO操作的完成，等内核完成IO操作以后会通知应用程序，这其实就是同步和异步最关键的区别，同步必须等待或者主动的去询问IO是否完成，那么为什么说是阻塞的呢？因为此时是通过select系统调用来完成的，而select函数本身的实现方式是阻塞的，而采用select函数有个好处就是它可以同时监听多个文件句柄（如果从UNP的角度看，select属于同步操作。因为select之后，进程还需要读写数据），从而提高系统的并发性！
  4. 异步非阻塞IO（AIO）
    • 在此种模式下，用户进程只需要发起一个IO操作然后立即返回，等IO操作真正的完成以后，应用程序会得到IO操作完成的通知，此时用户进程只需要对数据进行处理就好了，不需要进行实际的IO读写操作，因为真正的IO读取或者写入操作已经由内核完成了。

BIO 同步阻塞IO

  1. 当有一个客户端向服务器发起请求时，服务器就调用一个线程去处理这个请求。

  2. 存在的问题：

    当有多个客户端同时访问服务器时，每个客户端请求都会占用一个进程，服务器就需要启动多个线程处理这些请求。当一次进入服务器的请求非常多时，对服务器的压力就会非常大。

缺点：

  1. 可靠性差，当线程丢失时，请求就会丢失。
  2. 吞吐量低。线程不能建立太多，线程太多服务器容易崩溃。
  3. 适用于连接比较少的场景。
  4. 在 JDK 1.4 之前，BIO 是唯一的IO模型。

NIO 同步非阻塞IO

为了解决服务器一次开启多个线程压力过大的问题。JDK 1.4 中新增了 Selector，用于接收多个请求，并决定处理这些请求的优先级。这样就可以实现服务端线程与客户端请求的一对多关系，在一定程度上解决了BIO模式下服务器压力过大的问题。

可靠性比较好

吞吐量比较高

适用于连接比较多并且连接比较短的场景（轻操作）

AIO 异步非阻塞IO

AIO 又解决了 NIO 模型服务端需要一直守着线程的这个问题。可靠性最好，吞吐量也是非常高的，适用于连接比较多，并且连接比较长的场景（重操作）。例如：相册服务器。从JDK 7版本开始支持。

客户端在发送完请求后，可以执行任何操作，不需要一直等待或每隔一段时间来询问服务器请求是否处理完毕。相应的服务器在处理完请求后，会将处理结果保存到一个队列中，并逐一发送通知给请求的客户端，客户端在接收到通知后获取返回消息即可。

可靠性是最好的

吞吐量也非常高

适用于连接比较多，并且连接比较长的的操作（重操作）

同步 & 异步：针对请求

阻塞 & 非阻塞：针对客户端

在一个网络请求中，客户端会发送一个请求到服务端：

  1. 客户端发送请求后，就一直等待服务端的响应。
    • 客户端：阻塞！ 请求：同步
  2. 客户端在发送请求后，就去干别的事情了。但是还需要时不时的过来检查服务器端是否给出了响应。
    • 客户端：非阻塞！ 请求：同步
  3. 客户端在发送请求后，就歇下了，也不会去干别的事，但是也不会一直监听服务器的响应。服务器在处理完数据后，会发送给客户端一个通知，告诉客户端该接收响应了。
    • 客户端：阻塞 请求：异步

5、描述 HTTP 和 HTTPS 的区别

HTTP：是互联网上应用最为广泛的一种网络通信协议，基于 TCP，可以使浏览器工作更为高效，减少网络传输。

HTTPS：是 HTTP 的加强版，可以认为是 HTTP + SSL（Secure Socket Layer）。在 HTTP 的基础上增加了一系列的安全机制。一方面保证了数据在网络中传输的安全性，另一方面对访问者增加了验证机制。是目前现行架构下，最为安全的解决方案。

主要区别：

  1. HTTP 的连接是简单无状态的，HTTPS 的数据传输是经过证书加密的，安全性更更高。
  2. HTTP 是免费的，而 HTTPS 需要申请证书，而证书通常是需要收费的。
  3. 它们的传输协议不同，所以它们使用的端口也是不一样的，HTTP 默认使用 80 端口，而 HTTPS 默认使用 443 端口

HTTPS 的缺点：

  • HTTPS 的握手协议比较费时，所以会影响服务的响应速度及吞吐量。

    （HTTP 协议是基于 TCP 的“三次握手，四次挥手”，而 HTTPS 则是再 HTTP 的基础上，又增加了一系列证书校验的操作，所以耗时更长）

  • HTTPS 也并不是完全安全的。它的证书体系其实并不是完全安全的，并且 HTTPS 在面对 DDOS 这样的攻击时，几乎起不到任何作用。

  • 证书需要收费，并且功能越强大的证书费用越高。
