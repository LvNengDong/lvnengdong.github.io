NOSQL

NoSQL（Not Only SQL）：意思是”不仅仅是SQL”，是一种全新的数据库理念，泛指非关系型数据库。

NoSQL vs. SQL

随着互联网Web2.0网站的兴起，传统的关系型数据库在应付超大规模和高并发的SNS类型的WEB2.0纯动态网站的开发已经显得力不从心，暴露了很多难以克服的问题，而非关系型数据库则由于其本身的特点得到了非常迅速的发展。NoSQL数据库的产生就是为了解决大型规模数据集合多重数据种类带来的挑战，尤其是大数据应用难题。

NOSQL和关系型数据库比较

优点

  1. 成本低。
    • NOSQL数据库简单易部署，基本都是开源软件，无需购买即可使用。
  2. 查询速度快。
    • NOSQL数据库将数据存储在缓存中，关系型数据库将数据存储在硬盘中。自然NOSQL数据库查询速度要更快。
  3. 存储数据的格式多样化。
    • NOSQL的存储格式是{key,value}形式、文档形式、图片形式等等，所以可以存储基础类型以及对象或者是集合等各种格式，而数据库则只支持基础类型。
  4. 扩展性好。
    • 关系型数据库有类似于join这样的多表查询机制的限制导致扩展很难。

缺点

  1. 因为NOSQL数据库的历史较短，所以维护的工具和资料有限。
  2. 没有标准的SQL（结构化查询语句）的支持。 没有一个统一的标准去限定NOSQL语句的书写，也就是说，不同的NOSQL数据库的查询、增删改等语句可能迥然不同，将增加用户的学习和使用成本。
  3. 不提供关系型数据库对事务的处理

非关系型数据库的优势

  1. 性能：
    • 因为NoSQL是基于键值对的，而且不需要经过SQL层的解析，所以性能非常高。
  2. 可扩展性：
    • 同样也是因为基于键值对，数据之间没有耦合，所以非常容易水平扩展。

关系型数据库的优势

  1. 复杂查询：
    • 可以用SQL语句方便的在一个表以及多个表之间做非常复杂的数据查询
  2. 事务支持
    • 事务支持使得对于安全性能要求很高的数据访问需求得以实现。

总结

  1. 关系型数据库与NOSQL数据库并非对立而是互补的关系。即通常情况下使用关系型数据库，在适合使用NOSQL的时候使用NOSQL数据库。
  2. 一般情况下，会将数据存储在关系型数据库中，在NoSQL数据库中存储关系型数据库中数据的备份。

主流的NoSQL产品

键值对（key-value）存储数据库
相关产品 Tokyo Cabinet、Tyrant、Redis、Voldemort、Berkeley DB
典型应用 内容缓存，主要用于处理大量数据的高访问负载
数据模型 键值对
优势 快速查询
劣势 存储的数据缺少结构化

列存储数据库
相关产品 Cassandra、Hbase、Riak
典型应用 分布式的文件系统
数据模型 以列簇式存储，将同一列数据存在一起
优势 查找速度快，可扩展性强，更容易进行分布式扩展
劣势 功能相对局限

文档型数据库
相关产品 CouchDB、MongoDB
典型应用 Web应用（与key-value类似）
数据模型 键值对
优势 数据结构要求不严格
劣势 查询性能不高，而且缺乏统一的查询语法

图形数据库
相关产品 Neo4J、InfoGrid、Infinite Graph
典型应用 社交网络
数据模型 图结构
优势 利用图结构的相关算法
劣势 需要对整个图做计算才能得出结果，不容易做分布式的集群方案

Redis是一款高性能的NOSQL系列的非关系型数据库。

下载得到一个压缩文件，解压直接可以使用。文件中包含：

  • redis.windows.conf：配置文件
  • redis-cli.exe：redis的客户端
  • redis-server.exe：redis服务器端
  • ……

Redis服务器端的默认端口号是：6379

Redis中的数据类型

Redis是用C语言开发的一个开源的高性能键值对（key-value）数据库，官方提供测试数据，50个并发执行100000个请求，读的速度是110000次/s，写的速度是81000次/s，且Redis通过提供多种键值数据类型来适应不同场景下的存储需求，目前为止Redis支持的键值数据类型如下：

Redis在内存中以键值对的形式存储数据。其中所有键（key）都是String类型，而值（value）有5种不同的数据结构，我们常说的Redis的数据类型其实是指Redis键值对中值（value）的数据类型。

  • 字符串类型 String
  • 哈希类型 hash ： Map格式
  • 列表类型 list ： LinkedList格式，支持重复元素
  • 集合类型 set ： LinkedList格式，不允许重复元素
  • 有序集合类型 sortedset ： LinkedList格式，不允许重复元素，且元素有序。

Redis命令操作

字符串类型–String

  1. 存储：set key value

  2. 获取：get key

  3. 删除：del key

哈希类型–Hash

  1. 存储：hset key field value

  2. 获取：

    • 获取指定field对应的值：hget key field
    • 获取所有的filed和value：hgetall key

  3. 删除：hdel key field

列表类型–List

  1. 添加：可以添加一个元素到列表的头部（左边）或尾部（右边）

    • lpush key value：将元素加入列表左边
    • rpush key value：将元素加入列表右边

    List和String的区别是：当key相同时存储元素，String类型是覆盖，而List是追加

  2. 获取：lrange key start end，根据范围获取。0 -1表示获取全部元素。

  3. 删除：

    • lpop key：删除列表最左边的元素，并将元素返回
    • rpop key：删除列表最右边的元素，并将元素返回

集合类型–Set

不允许重复元素

  1. 存储：sadd key value

  2. 获取集合中的所有元素：smembers key

  3. 删除集合中的某个元素：srem key value

有序集合类型–SortedSet

不允许重复元素，且元素有序，每个元素都会关联一个double类型的分数（score），Redis通过分数（score）来为集合中的成员从小到大排序。

  1. 存储：zadd key score value

  2. 获取：

    • zrange key start：按分数（score）从小到大的顺序获取value
    • zrange key start end withscores：按分数（score）从小到大的顺序获取value和score

  3. 删除：zrem key value

通用命令

  1. 查询所有的键：keys *
  2. 获取键对应的value的数据类型：type key
  3. 删除指定的键值对：del key

持久化

Redis是一个内存数据库，当Redis服务器重启，或者服务器电脑重启后数据就会丢失。为了解决这个问题，我们可以将Redis内存中的数据持久化到硬盘的文件中。

AOF

AOF：通过日志记录的方式持久化Redis缓存中的数据。

启动步骤：

  1. 编辑redis.windows.conf文件，设置

    appendonly no（关闭AOF）

    -->>

    appendonly yes （开启AOF）

    详细参数：

    • appendfsync always ： 每一次操作都进行持久化
    • appendfsync everysec ： 每隔一秒进行一次持久化
    • appendfsync no ： 不进行持久化

  2. 修改配置文件后，需重启Redis服务器。此时，启动时需要指定配置文件名称。

Redis的应用场景

  • 缓存（数据查询、短连接、新闻内容、商品内容等等）
  • 聊天室的在线好友列表
  • 任务队列。（秒杀、抢购、12306等等）
  • 应用排行榜
  • 网站访问统计
  • 数据过期处理（可以精确到毫秒）
  • 分布式集群架构中的session分离
