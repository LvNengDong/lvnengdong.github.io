ArrayList 线程不安全

带着问题学习：

在 Java8 中，

  1. new ArrayList<>() 的底层数据结构是什么？
  2. new ArrayList<>() 的初始长度是多少？
  3. 数组长度一旦初始化就不能再修改了，Java 底层对数组进行扩容使用的方法是什么？
  4. 数组扩容的标准是什么？每次扩容多大？
  5. ArrayList 是线程安全的吗？
  6. 写一个 ArrayList 不安全的例子？
  7. 如何解决 ArrayList 线程不安全？有几种方法？

答案：

  1. new ArrayList<>() 底层默认的数据结构是 Object 数组，

  2. 如果使用无参构造方法，初始的数组长度为 10，即 new ArrayList<>() 初始会在内存空间中创建一个 Object[10] 的数组对象。

    但这句并不正确，应该说：无参构造器默认不会在内存中创建新数组，而是指向一个在类加载阶段就已经创建好了的容量为 0 的空数组，只有在调用了 add() 方法时，才会在堆空间中创建一个数组对象，并将设置数组对象的初始容量为 10。

  3. Java 底层对数组进行扩容使用的是 Arrays.copyOf 方法，再向下一层是 System.arraycopy 方法（System.arraycopy 是一个本地方法）。

    • ArrayList 是一个集合对象，底层是使用数组实现的。对于数组而言，其容量在第一次分配内存空间时就已经确定好了，不能再发生改变了。数组扩容就是新创建一个容量更大的数组，将原先数组中的数据全部拷贝到新数组中去。

  4. 正常情况下，数组扩容的标准是每次扩张一半，即每次扩张到原来的 1.5 倍。

    • 比如，新建一个 ArrayList 数组，数组容量最初为 10，第一次扩容后从 10 扩展到 15，第二次则是从 15 扩展到 22。

    但是，如果数组扩容 1.5 倍后仍然不够用的话，那么有多少个元素，就把容量到多少，即数组的容量刚好能盛下所有元素；

    还有一种情况，数组的容量是有上限的，这个上限值被规定为 Integer.MAX_VALUE（因为数组下标是 int 类型的所以最大值不能超出 Integer 的取值范围），达到最大值后再怎么扩容数组容量都不会再变了。

    • 扩展：HashMap 的初始长度是 16，每次扩容容量增加一倍。

  5. ArrayList 是线程不安全的。

  6. 例子见下文

  7. 解决 ArrayList 的线程不安全有多种方法，根据实际需要自行选择。本文罗列出来了 X 种。

    • new Vector()
    • Collections.synchronizedList(new ArrayList<>())
    • new CopyOnWriteArrayList<>()

ArrayList 概述

ArrayList 实现了 List 接口，继承了 AbstractList 类，底层是数组实现的，一般我们把它认为是可自动扩容的数组。

它是非线程安全的，一般多用于单线程环境下（与 Vector 最大的区别就是，Vector 是线程安全的，但是 ArrayList 性能更好）；

它实现了 Serializable 接口，支持序列化，能够通过序列化传输；

它实现了 RandomAccess 接口，支持快速随机访问，这里表现为可以通过下标直接访问（底层是数组实现的，所以可以直接用数组下标来索引）；

实现了 Cloneable 接口，能被克隆。

  • 数组是不可以自动扩容的！

  • **minCapacity ** 是新增元素后需要的最小容量。

    • 如果结果为真，则说明 elementData 是使用无参构造函数创建的，初始 size 为 0，size+1 即 minCapacity 将会是 1，使用 Math.max(DEFAULT_CAPACITY, minCapacity) 取较大值后返回的是 10。所以说无参构造器默认初始化一个容量为 10 的数组。但其实并不严格，我认为这里应该说：无参构造器默认不会在内存中创建新数组，而是指向一个在类加载阶段就已经创建好了的容量为 0 的空数组，只有在调用了 add() 方法时，才会在堆空间中创建一个数组对象，并将设置数组对象的初始容量为 10。
    • 注意：add() 方法只能一个一个地添加元素，不能一次性添加多个。
  2. 得到了 minCapacity 数组容量这个值后，当前方法出栈继续调用上一个方法 ensureExplicitCapacity。
  1. ensureExplicitCapacity 方法

  1. modCount++ 是用于在快速失败机制中判断使用的，与本文相关性不大，暂时略过。

  2. 接下来判断，新增元素后需要的最小容量是否超过了当前数组的容量，如果超过了，则证明需要扩容了。

    • 如果是使用无参构造器创建的 ArrayList 对象，在这一步 minCapacity == 10；而 elementData.length 则是 0，所以必然满足扩容的条件。

  3. 调用 grow 方法进行扩容。

  1. grow 方法

  1. minCapacity 是新增元素后需要的最小容量
  2. oldCapacity 是原来（未新增元素前）数组的容量
  3. 由于在上一个方法中已经知道原来的数组已经放不下新增的元素了，所以这里首先要进行扩容。oldCapacity >> 1 是位运算的右移操作，右移一位相当于除以 2，所以这句代码就等于 int newCapacity = oldCapacity + oldCapacity/2，即容量扩大为原来的 1.5 倍。
  4. 比较扩容后新数组的容量 newCapacity 与 新增元素后需要的最小容量 minCapacity ，如果扩容后仍然不够，则直接让 newCapacity == minCapacity，而不再是计算 1.5 倍的扩容。（即如果扩容 1.5 倍后仍然容量不够，那就直接让数组的容量等于元素的个数）
    • 例如：使用无参构造函数的 ArrayList 对象第一次调用 add() 方法扩容，由于扩容前数组的容量为 0，即使扩容 1.5 倍也仍然是 0，所以这时候直接让新的数组长度等于 10 就行了，这种特殊情况可以无视扩容 1.5 倍的规则。
    • 如果使用有参构造函数创建了一个长度为 1 的 ArrayList 对象，即底层数组长度为 1，第一次扩容时扩大到 1.5 显然也是不合适的，所以可以直接扩容到 2。
  5. 如果扩容后（第3或4步）发现新数组的容量超过了规定的最大容量 MAX_ARRAY_SIZE，则还需调用 hugeCapacity方法进行处理。（在数组长度已经很大的基础上，直接增加 0.5 倍的数组长度也是很大的，所以这时候我们需要额外加一些保护机制）
    • MAX_ARRAY_SIZE = Integer.MAX_VALUE - 8，因为数组下标是整型的数据，所以其取值范围不能超过这个范围。

所有的处理结束后，调用 Arrays.copyOf 方法将原数组中的所有元素拷贝到新数组中。

  1. hugeCapacity 方法（非必须执行的方法）

在数组长度已经很大的基础上，直接扩容 1.5 倍增长是很大的，所以这时候我们需要额外加一些保护机制。不再是在原数组长度的基础上增加 0.5 倍了，而是根据新增元素后需要的最小容量来改变新数组的容量（只有 2 种情况）。凡是能进入到该方法的都是大容量的数组（即扩大 1.5 就会超出上限）。

  1. 如果新增元素后需要的最小容量大于 MAX_ARRAY_SIZE ，则将新数组的容量设置为 Integer.MAX_VALUE；
  2. 否则，将新数组的容量设置为 MAX_ARRAY_SIZE；

方法执行完后，当前栈帧出栈，执行上一个方法。

  1. Arrays.copyof 底层真正执行的是 System.arraycopy 方法。

  1. 而 System.arraycopy 方法则是一个本地方法。

ArrayList 线程不安全

ArrayList 线程不安全案例

根据上面几部分的学习，我们已经知道了 ArrayList#add 方法在写入数据之前，会先确定要写入位置的下标。如果多个线程同时向同一个 list 集合中写入数据，就可能存在多个线程同时取得了同一个下标的写入权，如果不加以限制，就可能会造成如下情况。即一个线程写的值覆盖了另一个线程写的值。

甚至可能会出现并发修改异常：java.util.ConcurrentModificationException

既然 ArrayList#add() 是线程不安全的，为了解决线程安全问题，我们应该怎么做呢？——加锁！！！

解决方法1：Vector

加锁是最简单的方法，Java 的开发者们已经为我们提供好了 Vector#add() 方法用于执行线程安全的 add 方法。

Vector 类

  • Vector 同样也实现了 List<E> 接口，继承了 AbstractList<E> 类。和 ArrayList 是同一层级的类。

线程安全的代码

  • 只需将 new ArrayList 改为 new Vector 即可。

Vector 的缺点

  • 查看 Vector#add 的源码，可以看到该方法被 synchronized 修饰，是线程安全的，但是这样一来就会严重影响高并环境下的程序执行速度了。
  • 我们可以看到该方法在 JDK1.2 之后就被声明弃用了。在早期的 JDK 时期，由于硬件设备的落后，单线程或少线程的程序是主流，加锁的 add 方法能在不太影响程序执行速度的情况下加强程序的安全性，性价比还是很高的。但是在当前的硬件环境下，服务器端特别是电商项目大多都需要高并发的支持，这时再使用 Vector#add 方法对性能的影响就很大了。

线程安全的代码

写时复制技术

顾名思义，“写时复制”就是写的时候才复制，也就是说“读的时候不赋值”。我们都知道，多个线程读同一份资源不会有线程安全的问题，只有多个线程对同一份资源进行写操作的时候才可能出现线程安全问题。一般的加锁机制（重锁）不会区分读还是写，只要是某一资源被上锁了，不管是读还是写都需要进行资源的争夺。而如果某份资源在大多数情况下都是多线程读，只有少数情况下才会进行多线程写的话，使用不区分读写的锁显然对性能的浪费很大。而如果将读和写的操作分开，在某些情况下反而能得到更好的性能。写时复制技术就是这样一种技术。

写时复制技术用到了读写分离的思想。

读的时候共享，写的时候复制一份数据的副本，写好以后，覆盖原先的版本。

放在本例中就是，线程 A、B、C 同时执行 list.add() 操作向集合中添加元素。

  1. 如果线程 A、B、C 要读 list 中的数据，可以任意读，无锁。
  2. 如果线程 B 要修改 list 中的数据，如调用 add() 方法添加元素。不好意思，不能直接在原 list 的基础上添加，线程 B 需要将原来的 list 集合的内容复制一份，将要添加的数据加入到 list 副本中。如果添加成功了，再将指向原 list 的指针指向新的 list，这就完成了一次集合中元素的添加。

带着问题学习

  1. HashSet 底层的数据结构是什么？
  2. HashMap 是键值对的结构，而 HashSet#add 方法只用传递一个参数进去，为什么？

答案：

  1. HashSet 底层的数据结构是 HashMap。
  2. HashSet 底层确实调用了 HashMap#put(K, V) 方法，但是我们在调用 HashSet#add 方法时只需添加一个参数即可。这时因为 Java 开发人员在源码级别帮我们做了自动填充。HashSet 只是利用了 HashMap 键（Key）的不可重复性，至于值（Value）则是用一个常量填充。也就是说，我们每次调用 HashSet#add(X) 时，实际上调用的是 HashMap#put(X, 常量) 。至于这个常量是什么，在下文的源码分析中可以看到，其实只是一个类加载阶段就创建好的对象的地址。
