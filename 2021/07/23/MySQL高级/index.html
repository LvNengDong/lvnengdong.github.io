MySQL 架构

1.1 MySQL逻辑架构

​ MySQL 可以在多种不同场景中应用并发挥良好作用。主要体现在存储引擎的架构上，插件式的存储引擎架构将查询处理和其它的系统任务以及数据的存储提取相分离。这种架构可以根据业务的需求和实际需要选择合适的存储引擎。其主要逻辑架构可分为四层，从上到下分别是：

  1. 连接层
  2. 服务层
  3. 引擎层
  4. 存储层

第三层的存储引擎主要负责数据存储和提取，存储引擎不会去解析 sql，不同存储引擎之间不会通讯，只会简单地响应上层服务器的请求。

1.1.1 连接层

连接层主要处理客户端与 MySQL 服务端的连接、授权认证、安全等问题。包含本地 socket 通信和大多数基于客户端/服务端工具实现的类似于 TCP/IP 的网络通信。主要完成一些类似于连接处理、授权认证、及相关的安全方案。在该层上引入了线程池的概念，为通过认证安全接入的客户端提供线程。同样在该层上可以实现基于 SSL 的安全链接。服务器也会为安全接入的每个客户端验证它所具有的操作权限。

​

1.1.2 服务层

名字 介绍
Management Serveices & Utilities 系统管理和控制工具
SQL Interface SQL 接口。接受用户的SQL 命令，并且返回用户需要查询的结果。
Parser 解析器。SQL 命令传递到解析器的时候会被解析器验证和解析
Optimizer 查询优化器。SQL 语句在查询之前会使用查询优化器对查询进行优化。
Cache 和Buffer 查询缓存。

Management Serveices & Utilities：系统管理控制工具

​ 可以看做是类似于 Hadoop 中的 Zookeeper，或者是 SpringCloud 中的配置中心，主要作用是协调整管理个系统的各个模块。

SQL Interface：SQL 接口

​ 类似于 MVC 三层架构中的 Controller 层，只是负责接收请求和返回最终的结果，具体的业务逻辑交给更底层的服务去做。比如 select ... from ... 就会调用 SQL Interface。

Parser：解析器

SQL Interface 中的 SQL 命令传递到解析器的时候会被解析器验证和解析。主要功能有：

  • 验证：对接收的 SQL 语句进行语义和语法的分析，如果在分解构成中遇到错误，那么就说明这个 SQL 语句是不合理的，验证失败，不继续网下执行，直接抛出错误。
  • 解析：如果验证成功，则根据语义和语法，将 SQL 语句划分为不同的操作类型，然后进行针对性的转发到后续步骤。比如查询语句就转发到查询对应的服务，更新操作就转发到更新对应的服务。

Optimizer：查询优化器

SQL 语句在查询之前会使用查询优化器对查询进行优化。就是优化客户端请求的 SQL 语句，根据客户端请求的 SQL 语句，和数据库中的一些统计信息，在一系列算法的基础上进行分析，得出一个最优的策略，告诉后面的程序如何执行这个 SQL 语句。

查询优化器就类似于指令重排，对于开发人员来说，写一条 SQL 可能自我感觉良好，性能天下第一，但是对于底层机器来说，它有自己的优化标准，经过查询优化器优化后的 SQL 语句的执行顺序并不一定与开发人员设计的 SQL 执行顺序相同。

Cache和Buffer：查询缓存

如果查询缓存有命中的查询结果，查询语句就可以直接去查询缓存中取数据。这个缓存机制是由一系列小缓存组成的。比如表缓存，记录缓存，key 缓存，权限缓存等。

1.1.3 引擎层

​ 存储引擎真正的负责了 MySQL 中数据的存储和提取，服务器通过 API 与存储引擎进行通信。

​ 存储引擎接口模块可以说是 MySQL 数据库中最有特色的一点了。目前各种数据库产品中，基本上只有 MySQL 可以实现其底层数据存储引擎的插件式管理。这个模块实际上只是 一个抽象类，但正是因为它成功地将各种数据处理高度抽象化，才成就了今天 MySQL 可插拔存储引擎的特色。

​ 可插拔的存储引擎的实现，让 MySQL 在处理不同的业务逻辑时，可以根据实际应用场景选择效率尽可能高的存储引擎。这就是 MySQL 相较于其它同类数据库产品的最大优势。

  • 注意：存储引擎是基于表的，而不是数据库。

1.1.4 存储层

数据存储层，主要是将数据存储在硬件层面的文件系统上，并与存储引擎交互。

  • 行锁：操作时只锁某一行，不会对其它行产生影响。适合高并发的操作。
  • 表锁：即使只操作一条记录也会锁住整张表，不适合高并发的操作。【这是 MyISAM 被逐渐淘汰的主要原因之一】
  • 既缓存索引又缓存真实数据堆内存要求高，而内存大小对性能有着决定性的影响，已使用的内存越大，性能就越差。

1.3 大致查询流程

MySQL 查询的流程大致是：

  1. MySQL 客户端通过协议与 MySQL 服务器建连接；
  2. 发送查询语句，先查询缓存，如果命中，直接返回结果；
  3. 否则进行语句解析，也就是说，在解析查询之前，服务器会先访问查询缓存（Query Cache）——它存储了 SQL 语句以及
    相应的查询结果集。
  4. 如果某个查询结果已经位于缓存中，服务器就不会再对 SQL 语句进行解析、优化、以及执行。它仅仅将缓存中的结果返回给用户即可，这将大大提高系统的性能。

索引

是什么

官方定义：

索引（Index）是帮助 MySQL 高效获取数据的数据结构。

索引的目的在于提高查询效率。

​ 可以类比为字典。如果想要查询 “mysql” 这个单词，首先肯定要定位到 m 字母，然后在 m 字母的范围内继续找第 2 个 y 字母，以此类型，找到剩下的三个字母。

​ 如果没有索引，那么每找一个字母都需要全部查找一遍。

索引可以理解为“排好序的快速查找的数据结构”！！！

索引会影响到两大功能的执行效率：

  1. 排序：ORDER BY
  2. 查找：WHERE；HAVING

​ 在数据之外，数据库系统还维护着满足特定查找算法的数据结构，这些数据结构以某种方式指向数据（记录了真实真实数据的地址），在这些数据结构上我们可以实现高效的查找算法。这种数据结构，就是索引。

下图就是一种可能的索引方式示例：

  • 左边是索引表，一共有两列七条记录，最左边记录的是真实数据的物理地址，Col1 是主键字段，Col2 是索引字段。
  • 为了加快 Col2 的查找，可以维护一个右边所示的二叉查找树，每个节点分别包含索引键值和一个指向对应数据记录物理地址的指针，这样就可以运用二叉查找在一定的复杂度内获取到相应数据，从而快速的检索出符合条件的记录。
  • 一般来说索引本身也很大，不可能全部存储在内存中，因此索引往往以索引文件的形式存储的磁盘上。

优缺点

优势：

  • 索引能提高数据检索速率，对于一些经常被用于查询的字段，如果将其设置为索引，能大大地提高查询的速率。将频繁使用的字段就将其设置为索引，比不设索引时查询速度要快。

  • 通过索引列对数据进行排序，降低数据排序的成本。

劣势：

  • 实际上索引也是一张表，该表保存了主键与索引字段，并指向实体表的记录。所以索引也是要占用空间的。
  • 虽然索引大大提高了查询速度，但同时却会降低更新表的速度。比如在对表进行 INSERT、UPDATE 和 DELETE 操作时，因为更新表时 MySQL 不仅要更新数据，还要更新索引表。
    • 对于 UPDATE 操作，记录变更后索引字段在索引表中的位置就可能发生变化，需要更新。
    • 对于 INSERT 操作，添加记录后需要将新记录的索引字段加到索引表中。同理 DELETE 需要删除索引表中的记录。

索引分类

  • 单值索引：一个索引只包含单个列，一个表可以有多个单列索引。
  • 唯一索引：索引列的值必须唯一（即用于建立索引的列必须符合 UNIQUE），但允许有空值。
  • 复合索引：即一个索引包含多个列。

基本语法

  1. 创建
  2. 删除
  3. 查看表的所有索引
  • 主键自动就是一种唯一索引。

检索原理

【初始化介绍】

一颗 B 树，浅蓝色的块我们称之为一个磁盘块，可以看到每个磁盘块包含了数据项（深蓝色所示）和指针（黄色所示）。如磁盘块1 包含数据项 17 和35，包含指针 P1、P2、P3。

  • P1 指向数据项小于 17 的磁盘块，P2 指向数据项在17~35 之间的磁盘块，P3 指向数据项大于 35 的磁盘块。
  • 真实的数据存储在于叶子节点中，即 3、5、9、10、13、15、28、29、36、60、75、79、90、99。
  • 非叶子节点只不存储真实的数据，只存储指引搜索方向的数据项。如 17、35 并不真实存在于数据表中。

【查找过程】

  1. 如果要查找数据项 29，那么首先会把磁盘块 1 由磁盘加载到内存中，此时发生一次 IO，在内存中用二分查找确定 29 在17~35 之间，锁定磁盘块 1 的 P2 指针；
  2. 通过磁盘块 1 的 P2 指针的磁盘地址把磁盘块 3 由磁盘加载到内存中，发生第二次 IO，在内存中用二分查找确定 29 在 26~30 之间，锁定磁盘块 3 的 P2 指针；
  3. 通过指针加载磁盘块 8 到内存中，发生第三次IO，同时内存中做二分查找找到29，结束查询，总计三次 IO。

在实际生产中，3 层的 b 树可以表示上百万的数据，如果上百万的数据查找只需要三次IO，性能提高将是巨大的，如果没有索引，每个数据项都要发生一次 IO，那么总共需要百万次的IO，显然成本非常非常高。

哪些情况需要创建索引

  1. 主键自动建立唯一索引；
  2. 频繁作为查询条件的字段应该创建索引；
  3. 查询中与其它表关联的字段，外键关系应该建立索引；
  4. 查询中排序的字段。排序字段若通过索引去访问将大大提高排序速度；
  5. 单键/组合索引的选择问题，组合索引性价比更高；
  6. 查询中统计或者分组字段；

哪些情况不需要创建索引

  1. 表记录太少

  2. 频繁更新的字段（经常增删改的表或者字段）不适合创建索引；【因为每次更新不单单要更新数据库表记录还会更新索引表记录】

  3. Where 条件里用不到的字段不创建索引（不用做查询条件的字段）

  4. 过滤性不好的不适合建索引。

    【假如一个表中有 10w 条记录，有一个字段 gender 只有 0 和 1 两种值，且每个值的分布概率大约为 50%，那么对这种字段建立索引一般不会提高数据库的查询速度】

    【索引的选择是指索引列中不同值的数量与表中记录数的比。如果一个表中有 2000 条记录，表索引列有 1980 个不同的值，那么这个索引的选择性就接近于 1，这个索引的效率越高。】

性能分析

MySQL Query Qptimizer（MySQL查询优化器）

  • MySQL 中有专门负责优化 SELECT 语句的优化器模块。主要功能是：通过计算分析系统收集到的统计信息，为客户端请求的 SQL 语句提供它认为最优的执行计划。（机器认为最优的数据检索方式，但 BDA 不一定认为是最优的）
  • 当客户端向 MySQL 服务器请求一条 SQL ，命令解析器模块完成请求分类，区别出是 SELECT 并转发给 MySQL_Query_Qptimizer 时， MySQL_Query_Qptimizer 首先会对整条 SQL 进行优化，处理掉一些常量表达式的预算，直接转换成常量值。并对 SQL 中的查询条件进行简化和转换，如去掉一些无用或显而易见的条件、结构调整等。……

MySQL 的常见瓶颈

  1. CPU：CPU 在饱和的时候一般发生在数据装入内存或从磁盘上读取数据的时候。
  2. IO：磁盘 IO 瓶颈发生在装入数据远大于内存容量的时候。
  3. 服务器硬件的性能瓶颈：top，free，iostat 和 vmstat 来查看系统的性能状态。

MySQL锁

是什么

定义

锁是计算机协调多个进程或线程并发访问某一资源时的机制。

在数据库中的数据是一种供许多用户共享的资源，所以就可能出现多个用户并发访问的情况。如何保证数据并发访问的一致性、有效性是所有数据库必须解决的一个问题。锁冲突也是影响数据库并发访问性能的一个重要因素。

生活案例

在淘宝上抢手机，库存只剩最后一件了，如果同时有多个用户同时点击了购买，到底谁买到？

  • 首先，这里肯定要使用到事务，我们先从库存表中取出商品数量，然后插入订单，付款后插入付款表信息，最后更新商品数量。在这个过程中，使用锁可以对有限的资源进行保护，解决隔离和并发的矛盾。

注意事项：

  • 数据库锁都是悲观锁！！！

锁的分类

从对数据操作的类型来分可分为：

  • 读锁（共享锁）：针对同一份数据，多个读操作可以同时读而不会互相影响。
  • 写锁（排它锁）：当前写操作没有完成之前，它会阻断其它写锁和读锁。【注意：写和读都被阻断了】

从对数据操作的粒度来分可分为：

  • 行锁（Row_Level_Lock）：开销大，加锁慢；会出现死锁；锁定粒度最小，发生锁冲突的概率最低,并发度也最高。
  • 表锁（Table_Level_Lock）：开销小，加锁快；不会出现死锁；锁定粒度大，发生锁冲突的概率最高,并发度最低。
  • 页锁：开销和加锁时间界于表锁和行锁之间；会出现死锁；锁定粒度界于表锁和行锁之间，并发度一般。

使用场景：

  • 表锁更适合以查询为主的表使用。
读锁（共享锁） 写锁（独占锁）
表锁 表共享读锁 表独占写锁
行锁 行共享读锁 行共享写锁

MySQL 锁机制：

  • MySQL 的锁机制比较简单，就是不同的存储引擎支持不同的锁机制。
  • 比如：MyISAM 和 Memory 存储引擎采用的是表级锁。
  • InnoDB 存储引擎既支持行锁（默认），也支持表锁。
  • MyISAM 表在进行读操作的时候会自动对表加共享读锁，在进行写操作的时候会自动对表加共享写锁。
  • InnoDB 表在进行读操作的时候会自动对行加共享读锁，在进行写操作的时候会自动对行加共享写锁。

案例分析

给表加读锁

  1. 向上面建好的 tb_emp 表中插入数据
  1. 模拟两个线程同时操作 tb_emp 表

线程 T1 线程 T2
lock table tb_emp read; 【OK】
select * from tb_emp; 【OK】
select * from tb_emp; 【OK】
UPDATE tb_emp SET NAME="张飞" WHERE id=1; 【Error】
UPDATE tb_emp SET NAME="张飞" WHERE id=1; 【阻塞】
UNLOCK TABLES;
UPDATE tb_emp SET NAME="张飞" WHERE id=1; 【OK】

分析：

两个线程同时操作数据库中的同一张表，

  1. 首先，线程 T1 先对 emp 表加表共享读锁；
  2. 然后，T1 对加锁的表执行查询操作，OK；【因为 T1 本身就是持有锁的】
  3. 再之，T2 对加锁的表进行查询操作，OK；【因为读锁是一种共享锁，共享锁可以有多个】
  4. T1 在未释放锁之前，想要读取其它表（非加锁的表），发现报错；
  5. T2 读取未加锁的表，OK；
  6. T1 在未释放锁之前，想要修改被加锁的表，报错；
  7. T2 想要修改未加锁的表，发现没有报错 ，但程序被阻塞【一直在转圈，但不能执行结束】
  8. T1 释放读锁；
  9. 这时 T2 立即执行未执行完的更新的操作。

给表加写锁

  • 当前线程对表的 查询+更新+插入 操作都可以执行；
  • 其它线程对表的查询被阻塞，需要等待锁被释放。

总结

  • MyISAM 在执行查询语句（SELECT）前，会自动给涉及的所有表加读锁，在执行增删改操作前，会自动给涉及的表加写锁。
  • 如果想要使用 InnoDB 给表加读/写锁，则需要显式地声明才可以。

结论：

对所有使用 MyISAM 引擎的表进行操作，会有以下情况：

  • 对 MyISAM 表的读操作，不会阻塞其它线程对同一表的读请求，但是会阻塞对同一表的写请求。只有当读锁释放后，才会执行其它线程的写操作。
  • 对 MyISAM 表的写操作，会阻塞其它线程对同一表的读和写操作，只有当写锁释放后，才会执行其它线程的读写操作。

简而言之，就是读的时候可以读，读的时候不能写，写的时候不能读，写的时候不能写（读就是共享读，其它的可以读，不能写；写是独占写，其它的不能读也不能写）

表锁分析

查看哪些表被加锁了

如何分析表锁定

可以通过检查 table_locks_waited 和 table_locks_immediate 状态变量来分析系统上的表锁定。

这里有两个变量记录 MySQL 内部表级锁定的情况，两个变量的说明如下：

  • table_locks_immediate：产生表级锁定的次数，即表锁被获取过的次数，每次获取锁该值+1。
  • table_locks_waited：由于表锁争用而发生等待的次数（不能立即获取锁的次数，每等待一次该值+1）。此值较高则说明存在着较严重的表级锁争用的情况。

MyISAM 不适合作为以写操作为主的表的引擎。因为对表中数据进行的写操作会造成表被自动加写锁，导致其它线程不能做任何操作，大量的更新会使得查询操作很难获取到锁，从而造成永远阻塞。

行锁

特点

  • InnoDB 引擎在执行 SQL 语句时默认加行锁（执行查询语句加默认读锁，执行写操作语句默认加写锁）；
  • 开销大，加锁慢。（需要给每行数据都加锁，自然开销比较大，加锁比较慢）
  • 会出现死锁；
  • 锁的粒度小，
  • 发生锁冲突的概率最低，
  • 并发度最高。

案例分析

建表

行锁的基本演示

  • 模拟两个线程同时操作 stu 表数据。建立两个会话分别为 Session1 和 Session2。
Session1 Session2
SET autocommit = 0 SET autocommit = 0
UPDATE tb_stu SET age=11 WHERE id=1;【OK】
SELECT * FROM tb_stu;【查看后发现更新成功】
SELECT * FROM tb_stu;【未读到Session1未提交的事务】
UPDATE tb_stu SET age=19 WHERE id=1;【写操作被阻塞】
COMMIT;【Session1 提交事务】
【阻塞解除，更新操作继续执行成功。此时，id=1对应的age=19】
  1. 分别关掉 Session1 和 Session2 事务的自动提交行为；
  2. 更新 id=1 的这一条记录，在 Session1 中更新成功，但在 Session2 中并未读取到 Session1 的更新。这是由于 MySQL 默认的隔离级别（可重复读）导致的。
  3. 此时，如果在 Session2 中想要更新同一行数据，发现 Session 的操作被阻塞，这是因为 InnoDB 默认给每行数据加了行锁。
  4. 使用 COMMIT 提交事务后，行锁也会被释放，这时 Session2 的更新操作得以继续执行。

索引失效行锁升级为表锁

行锁必须有索引才能实现，否则会自动锁全表，那么就不是行锁了。

所以在 InnoDB 表中有两种情况会导致行锁升级为表锁：

  • 一是条件字段压根没有建立索引；
  • 二是条件字段的索引在使用中失效了。

举例

在 tb_stu 表中，id 为 int 类型，age 为 varchar 类型。并且为这两个字段分别创建了索引 a 和 b。如果我们在更新数据的过程中，故意把 varchar 类型的数据写成 int 类型的数据，由于 MySQL 存在自动转型，int 类型也能被识别，但是此时就破坏了索引 b 的结构，就会导致索引 b 失效。从而导致行锁会升级到表锁。

  1. 破坏索引结构

  2. 经过上面的操作后，age 对应的索引 b 的结构已经失效了，如果接下来还有以 age 为条件字段的操作，则行锁会升级为表锁

    Session1 Session2
    BEGIN;
    UPDATE tb_stu SET id=66 WHERE age='60';【OK】
    UPDATE tb_stu SET id=77 WHERE age=70【被阻塞】
    COMMIT;
    【更新操作继续执行】

Session1 设置为手动事务提交，Session2 设置为自动事务提交。

  1. Session1 开启事务，更新 age='60' 的行记录，OK；
  2. Session2 更新 age='70' 的行记录，却发现被阻塞。但是明明 tb_stu 表使用的是行锁呀？这就是索引失效后行锁自动升级为表锁了。
  3. Session1 提交后，释放表锁，Session2 得以继续进行，修改成功。

间隙锁危害

间隙锁案例：

Session1 Session2
SET autocommit = 0 SET autocommit = 0
UPDATE tb_stu SET age=99 WHERE id>1 AND id<6;
【更新 1<id<6的所有数据】
SELECT * FROM tb_stu;【查看后发现更新成功】
INSERT INTO tb_stu VALUES(2,"20");
【产生阻塞，插入操作无法执行完毕】
COMMIT;【Session1 提交事务】
【阻塞解除，插入操作继续执行成功。id=2的数据被成功插入】

  1. 分别关闭 Session1 和 Session2 的事务自动提交；

  2. 更新 id 为（1，6）范围内的所有数据，由于当前表中没有 id 为 2 的数据，所以就是更新 id 为 3，4，5 的数据 age=99。

  3. 但是我们此时使用 Session2 想要插入一条 id=2 的数据，却发现被阻塞了。这时为什么呢？

按理来说，行锁只会作用于当前行，在本例中也就是第 3、4、5 行，而 Session2 操作的明明是第 2 行，为什么也会被阻塞呢？这时因为此时 id=2 的那一行也被加了行锁，这就是传说中的**间隙锁**。
  1. 只有当 Session1 的事务提交后，所有作用于行上的锁被释放了，Session2 的插入操作才能继续完成。

什么是间隙锁？

当我们使用范围条件而不是相等条件检索数据，并请求共享或排它锁时，InnoDB 会给符合条件的已有数据记录的索引项加锁，这是显然的；

对于索引的键值在条件范围内但并不存在的记录，叫做“间隙（GAP）”；

InnoDB 也会对这个“间隙”加锁，这种锁机制就是所谓的间隙锁（Next-Key锁）。

总结：间隙锁 = 使用范围条件检索 + 请求锁 +检索的字段是一个索引

间隙锁的危害

因为 Query 执行过程中通过范围查找的话，它会锁定整个范围内所有的索引键值，即使这个键值在索引表中并不存在。

间隙锁有一个致命的弱点就是当锁定一个范围的键值之后，即使某些不存在的键值也会被无辜的锁定，从而造成在锁定的时候无法插入锁定键值范围内的任何数据。在某些场景下这可能会对性能造成很大的危害。

案例结论

优势：

InnoDB存储引擎由于实现了行锁，虽然在锁定机制的实现方面所带来的性能损耗肯能比表锁要高一些，但是在整体并发处理的能力是要远远优于 MyISAM 的表锁的。当系统并发量较高时，InnoDB 的整体性能和 MyISAM 相比就会有比较明显的优势了。

缺点：

但是，InnoDB 也有其脆弱的一面，当我们使用不当时（如索引失效导致行锁升级为表锁 ），可能会让 InnoDB 的整体性能表现不仅不能比 MyISAM 高，甚至可能会更差。

行锁分析

如何分析行锁？

  • 通过检查 InnoDB_row_lock 状态变量来分析行锁的争夺情况：

对各个状态量的说明如下：

  • InnoDB_row_lock_current_waits：当前正在等待锁的数量；
  • InnoDB_row_lock_time：从系统启动到当前时间该行数据被锁定的总时间；【★】
  • InnoDB_row_lock_time_avg：每次等待所花的平均时间；【★】
  • InnoDB_row_lock_time_max：从系统启动到当前时间等待最长的一次所花的时间；
  • InnoDB_row_lock_waits：系统启动到当前时间总共等待的次数；【★】

对于这五个状态量，比较重要的有三个。尤其是当等待次数很高，而且等待平均时长也不小的时候，我们就需要分析系统中为什么会有 这么多等待，然后根据分析结果制定优化计划。

优化建议

  • 尽可能让所有数据检索都通过索引来完成，避免无索引行锁升级为表锁；
  • 合理设计索引，尽量缩小锁的范围；
  • 尽可能缩小检索条件，避免频繁出现间隙锁 ；
  • 尽量控制事务大小，减少锁定资源量和时间长度；
  • 尽可能低级别事务隔离。

页锁

  • 页锁的开销和加锁时间介于表锁和行锁之间；
  • 会出现死锁；
  • 锁的粒度介于表锁和行锁之间；
  • 并发度一般。
