String

Posted on 2021-06-13 In JavaSE

本篇文章默认使用 HotSpot 虚拟机在 JDK8 环境下运行，如有其它情况会具体指出。

1 String 的基本特性

String；字符串，顾名思义，字符串就是把一个个的字符串在一起，所以说字符串的底层实现就是一个字符数组。

  • String 被声明为 final 的，表示不可变，不可被继承，不可被重写。
  • String 实现了 Serializable 接口，表示字符串是支持序列化的。
  • String 实现了 Comparable 接口，表示 String 可以比较大小。
  • 在 JDK8 中，String 使用 final char[] value 数组来存储字符串数据。

1.1 String存储结构变更

  • 在 JDK9 之前，String（字符串）的底层是通过字符数组 char[] 来保存字符串的。
  • 从 JDK9 开始，String 底层变成了通过字节数组 byte[] 来保存字符串数据。

String 底层将 char[] 变成 byte[] 来存储字符串数据，可以节约一些空间。

同时，基于 String 的一些对象，如 StringBuffer、StringBuilder 等也作出了同样的改变，将存储结构变更为了 byte[] 。

1.2 String 的不可变性

  • String：代表不可变的字符序列

    • 当对字符串重新赋值时，需要重写指定内存区域，重新赋值，不能在原先 value 的基础上进行赋值。
    • 当对现有的字符串进行拼接操作时，也需要重新指定内存区域，重新赋值，不能在原先 value 的基础上进行赋值。
    • 当调用 String.replace() 方法修改指定的字符串或字符时，也需要重新指定内存区域并重新赋值，不能在原先 value 的基础上进行赋值。

  • 通过字面量的方式（区别于 new）给一个字符串赋值，此时字符串的值保存在堆空间的字符串常量池中。

  • 字符串常量池中是不会存储相同内容的字符串的 。

    • String 的 StringPool 是一个固定大小的 HashTable，默认值大小长度在不同的JDK版本中均有所不同。如果放进 String Pool 的 String 非常多，就会造成 Hash 冲突严重，从而导致链表会很长，而链表长了后直接会造成的影响就是当调用 String.intern() 时性能会大幅下降。
    • 使用 -XX:StringTableSize 可以设置 StringPool 的长度。
    • JDK6中，StringPool 的默认长度为 1009 ，可以任意设置大小；
    • 从 JDK7 开始，StringPool 的默认长度增加到了 60013 ，
    • 从 JDK8 开始，可设置的最小长度为 1009 。

2 String的内存分配

​ Java语言中有 8 种基本数据类型和一种比较特殊的类型 String 。这些类型为了使它们在运行过程中速度更快、更节省内存，都提供了一种常量池的概念。 常量池类似于一个 Java 系统级别提供的缓存。8种基本数据类型的常量池都是系统协调的， String类型的常量池比较特殊，想要在常量池创建字面量对象的主要方法有两种 ：

  1. 直接使用双引号声明出来的 String 对象会直接存储在常量池中。
    • 比如： String info = "HelloWorld";
  2. 如果不是双引号声明的 String 对象，可以使用 String 提供的 intern() 方法。

JDK8 中 String 对象在 JVM 内存中的几种情况

  1. String s1 = "Hello World";
  2. String s1 = new String("Hello World");
  3. 只存在于字符串常量池中；
  4. 只存在于堆中的非字符串常量池空间；
  5. 二者

2.1 不同JDK版本中 StringPool 在内存中的调整

  • 在 Java7 以前，字符串常量池存放在方法区中（在 HotSpot 中就是永久代）。
  • 从 Java7 开始，HotSpot 将字符串常量池的位置调整到了 Java 堆中。

问题：String Pool 为什么要从方法区调整到堆空间中？

  • 保存在方法区的缺点：
    1. 方法区（永久代/元空间）默认占用的内存空间较小，如果在方法区中保存大量的 String 字面量对象，极易导致方法区空间出现 OOM 异常。
    2. 即使将方法区初始的内存空间设置为一个较高的值，但是由于方法区进行 GC 的频率极低，这也就意味着在方法区中，String对象在使用完后，无法做到及时的 GC，也会导致方法区频繁出现 OOM 异常。
  • 保存在堆空间的优点：
    1. 所有的 String 字面量对象都保存在堆（Heap）中，它就和其它普通对象一样，这样的好处就是在进行JVM调优时只需调整堆的大小就可以了。

​ Java语言规范里要求完全相同的字符串字面量对象，应该包含同样的Unicode字符序列（包含同一份码点序列的常量），并且必须指向同一个String类实例。

4 字符串拼接操作

  • 常量 + 常量
  • 常量 + 变量
  • 变量 + 变量

常量 + 常量

​ 常量与常量的拼接结果还保存在常量池中，原理是编译期优化。

问题：如何理解“编译期优化”？

答：在 .java 源代码中使用多个字符串常量进行拼接，这些字符串常量在被编译为 .class 字节码文件时就会将多个字符串合为一个字符串常量，也就是说被加载到 JVM 内存中的只是 .class 字节码文件中的拼接后的字符串常量。

常量/变量 +变量

  • 常量池中不会保存相同内容的字面量对象。

  • 在字符串拼接操作中，多个字符串只要其中有一个是变量，结果就只会保存在堆中的非常量池区域。

  • 含有变量的字符串拼接原理是：只要拼接的多个字符串中存在变量，那么 JVM 在进行字符串拼接的时候，

    1. 首先会 new StringBuilder() 对象，
    2. 然后调用该对象中的 append() 方法拼接字符串，
    3. 最后再调用 StringBuilder#toString() 方法，将拼接后的字符串对象转换成 String 对象，并将这个 String 对象保存在堆中的非常量池区域。

    Tip：此时拼接后的 String 对象只会保存在堆中的非常量池区域，而不会同步保存到字符串常量池中！！！

5 intern() 的使用

源码分析

​ intern() 方法是 String 类中的一个方法，查看源码可知，String#intern() 方法是一个 native 修饰的本地方法。也就是说，其具体实现是由非 Java 的 C/C++ 来实现的。

主要作用：

​ 假设存在一个字符串对象 String s1 ，当调用 s1.intern() 方法时，

  1. 首先要去判断字符串常量池中是否已经存在一个字面量对象 s2 与 s1 对象的值相等，即 s1.equals(s2) == true 是否恒成立。
  2. 如果成立，那么就返回字符串常量池中字面量对象的内存地址。
  3. 若不成立，就将 s1 对象的值复制一份作为一个字面量对象添加到字符串常量池中，并返回该字面量对象在字符串常量池中的内存地址。

intern() 的使用

  • 如果不是通过双引号创建的String对象，可以使用 String 提供的 intern() 方法，将 String 对象放入常量池中。
  • intern() 方法的作用：intern() 方法会从字符串常量池中查询当前字符串是否存在，若不存在就会将当前字符串放入常量池中。也就是说，在任意字符串上调用 String#intern() 方法,，其返回结果所指向的地址值，必须和直接以常量形式出现的字符串实例完全相同。通俗点讲，Interned String 就是确保字符串在内存中只有一份数据，这样可以节约内存空间，加快字符串操作任务的执行速度。
  • Tip：这个值只会被存放在字符串内部池（String Intern Pool）。

问题：如何保证变量 s 指向的字符串是字符串常量池中的数据而不是堆中非常量池区域的数据呢？

答：有2种方式

  1. 直接通过双引号而不是 new 关键字来创建 String 对象。如：String s = "HelloWorld";
  2. 对得到的字符串对象，调用 intern() 方法。（这种方法并不能完全保证）

问题：new String("HelloWorld") 会创建几个对象？

答：如果在此之前，字符串常量池中已有字面量对象 “HelloWorld”，则新创建1个对象；否则是新创建2个对象，一个保存在堆中非字符串常量池区域（和普通对象一样），另一个是字符串常量池中的字面量对象 “HelloWorld”。

  1. 首先，在堆中非常量池区域中创建一个 String 对象。
  2. 其次，在字符串常量池中创建字面量为 “HelloWorld” 的字面量对象。
  3. 再次，调用 String 类的构造方法，对字符串对象 s 进行显式初始化；
  4. 最后，将结果地址保存到局部变量表下标为1的位置上。

扩展：String str = new String("a") + new String("b"); 会创建多少个对象？

分析：

  • 对象1：new StringBuilder()， 首先，两个字符串变量拼接时，需要一个辅助对象 StringBuilder 。

  • 对象2：new String(“a”)，

  • 对象3：常量池中的字面量对象 “a”

  • 对象4：new String(“b”)，

  • 对象5：常量池中的字面量对象 “b”

  • 对象6：在调用 StringBuilder.toString() 方法时，根据 toString() 方法的源码，还要创建一个 String 对象。

    • Tip：toString() 方法虽然方法的调用，在字符串常量池中不会创建对应的字面量对象。

面试题

不同版本JDK中不同方式创建字符串对象的先后顺序的影响

  1. 如果只在堆的非字符串常量池区域创建对象： String s1 = new String("a") + new String("b"); 此时常量池中没有该字符串值对应的字面量对象。
  2. 如果这时继续在常量池中创建 “ab” 字符串对应的字面量对象。如调用 s1.intern(); 方法。
    • 从 JDK7 开始，为了节省内存，在常量池中创建字面量对象这一步，如果在堆的非常量池区域存在相同字面量值的对象，就不再创建一个“真正”的字面量对象了，而是在字面量对象的地址中保存堆中非常量池区域那个对象的地址值。
    • 而在 JDK7 之前，在常量池中创建的对象时真正意义上的字面量对象，保存的是真正的数据值，而不是对象的引用。
  1. 如果使用 String s1 = new String("1"); 创建对象，这时会创建两个对象，即字符串常量池中的字面量对象和Java堆的非字符串常量池区域的对象，这两个对象都是真正意义上的对象。上述1，2的规则就不再使用了。

总结：String中intern()的使用：

  • 在 JDK7 之前，将字符串对象尝试放入字符串常量池中时：

    • 如果字符串常量池中已经含有，则并不会放入，并返回已有对象的内存地址；
    • 如果字符串常量池中没有，则会将当前对象的值复制一份，放入字符串常量池，并返回字符串常量池中该对象的内存地址；

  • 从 JDK7 开始，将字符串对象尝试放入字符串常量池中时：

    • 如果字符串常量池中已经含有该字面量对象，则直接返回该字面量对象的内存地址即可；
    • 如果没有，则会把对象的引用地址复制一份，放入字符串常量池，并返回字符串常量池中的引用地址。

背景说明

背景

通过对大量 Java 应用的研究测试表明：

  • 堆中存活的数据集合中 String 对象占据了约 25%；
  • 其中 String 对象中重复的对象又占了约 13.5%；重复的意思就是： string1.equals(string2) == true
  • 堆上存在重复的String对象必然是一种内存资源的浪费。
  • String 对象的平均长度是 45

许多大规模Java应用的瓶颈在于内存，而这些项目将在G1垃圾收集器中实现自动持续对重复的String对象进行去重，这样能避免浪费内存。
